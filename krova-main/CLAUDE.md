# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Krova is a self-service cloud infrastructure platform for managing lightweight Cubes (Firecracker microVMs) on dedicated bare-metal servers. Users create spaces (personal or team), provision Cubes, and manage them through a dashboard. Billing is credit-based with **Plans & Tiers**: every space is on a plan from the DB-driven `plans` catalog (a free default plus operator-defined paid + custom plans, all editable from Orbit) that sets per-feature limits, and credit is consumed per-hour. Per-space limit overrides on the `spaces` row let an operator grant individual customers higher caps without duplicating the plan. Spaces subscribe to, upgrade, downgrade, or cancel a paid plan through **Polar** recurring subscriptions; each active period grants the plan's included credit. Spaces also top up their credit balance with prepaid one-time purchases via Polar. The Orbit admin grant path still exists alongside both.

**This is a live production system with real customers, real data, and active billing.** Every database change, migration, and data update must be treated as if it runs against production immediately — because it does. There are no staging environments with identical data. See Rule 40 for the full production data safety policy.

## Architecture

This is a critical architectural invariant** (Rule 1).

## Conventions

- **Path alias — ALWAYS use `@/` for in-repo imports, NEVER relative paths**: `@/*` maps to project root (`tsconfig.json` `paths`), e.g. `@/lib/db`, `@/components/ui/button`. This applies to EVERY internal import including same-directory siblings — write `import { execCommand } from "@/lib/ssh/exec"`, never `from "./exec"` or `from "../foo"`. The whole codebase is `@/`-only (audited 2026-06-01); a relative `./`/`../` import is a regression. Node-builtin (`node:fs`) and package imports are unaffected.
- **shadcn/ui**: radix-lyra style, Phosphor icons, RSC-compatible. Add components via `pnpm shadcn add <component>`
- **Server actions** live in `app/actions/` and handle mutations with form validation
- **API auth enforcement order**: `requireSession` → `requireSpaceMember` → `requirePermission` → `requireCubeAccess`
- **Lifecycle logging**: All mutations write to `lifecycleLogs` with `entityType` (cube | space)
- **Audit logging**: All mutations also write to `auditLogs` via `audit()` from `lib/audit.ts`
- **Email templates**: React Email components in `lib/email/components/`, rendered by `lib/email/renderer.ts`
- **Server SSH**: Port 2822, root access, keys encrypted with AES-256-GCM at rest using `APP_SECRET`
- **ORM-only queries**: Use Drizzle ORM helpers (`eq`, `and`, `sum`, `count`, `inArray`, `ilike`, `isNotNull`, `lt`, `gte`, `lte`, `desc`, `asc`, `notInArray`, `countDistinct`, `ne`) — never raw `sql` template literals except for DDL or operations with no ORM equivalent

### Form Validation (react-hook-form + Zod)

All forms use `react-hook-form` with `zodResolver` for validation:

- **Schema**: Define a Zod schema per form. Use `z.string().min(1, ...)` for required fields, `z.string().email(...)` for emails, `z.number().min(...)` for numbers.
- **Form setup**: `useForm<T>({ resolver: zodResolver(schema), defaultValues, mode: "onChange" })`
- **Form UI**: Wrap in `<Form {...form}><form onSubmit={form.handleSubmit(onSubmit)}>` using components from `@/components/ui/form`.
- **Inline errors**: Display below inputs via `<FormMessage />` — never use `toast.error()` for validation.
- **Server errors**: Use `form.setError("root", { message: result.error })` and display via `<Alert variant="destructive">` inside the form.
- **Submit button disabled**: New entity forms: `!isValid || isPending`. Edit forms: `!isValid || !isDirty || isPending`.
- **Success feedback**: Use `toast.success()` from `sonner`.
- **Number inputs**: Use `z.number()` (not `z.coerce.number()`) with manual `valueAsNumber` conversion in `onChange`.

### UI Component Patterns

- **Sheets (side panels)** for any UI that collects user input: forms, configuration, date pickers, resource selection. Always slide in from the right using `Sheet`/`SheetContent` from `@/components/ui/sheet`.
- **AlertDialog** ONLY for confirmations — including delete confirmations that require typing a name. Never use `Dialog` for input forms — always use Sheet.
- **React key props**: Always use `<Fragment key={...}>` (not `<>`) when mapping over lists. Never leave keys on children inside fragments.
- **Page headers**: The page title (`h1`) and primary action buttons must be on the **same line** using `flex items-center justify-between`.
- **Adjacent buttons must have the same height** — never mix `size="sm"` and default-size buttons in the same row.
- **Two-column layouts**: Use `grid gap-6 lg:grid-cols-2` for related card pairs. Stacks on mobile.

## Development Practices

Behavioral guidelines that bias toward caution over speed (for trivial tasks, use judgment). **Full rationale, examples, and the incident histories behind each are in [docs/development-practices.md](docs/development-practices.md) — read it when a point below needs context.** The principles:

- **Quality over speed.** Slow + correct beats fast + re-corrected. After fixing one instance of a pattern, grep the whole codebase for the same pattern unprompted. After any dependency bump (even a patch), read the changelog and audit against it. Verify (typecheck, lint, tests) before claiming anything works — "should work" is not acceptable.
- **Research → Brainstorm → Think → Ask → Implement.** Every non-trivial change follows this order. Don't skip steps; don't implement then ask.
- **Think before coding.** State assumptions explicitly; surface trade-offs; if multiple interpretations exist, present them — don't pick silently. When unclear, stop and ask one tight question.
- **Simplicity first.** Minimum code that solves the problem. No speculative features, abstractions, or "flexibility" that wasn't requested. If 200 lines could be 50, rewrite it.
- **Surgical changes.** Touch only what the request requires; match existing style; don't refactor what isn't broken; remove only imports/vars your change made unused; mention unrelated dead code rather than deleting it.
- **Default to warn, not auto-fix.** When unsure whether to take a destructive or mutating action vs. surface a warning, pick the warning — auto-fixes hide the user's intent from them.
- **Goal-driven execution.** Define success criteria up front (often a failing test), then loop until verified.
- **Third-party integrations — research BEFORE writing or claiming.** Fetch the vendor's CURRENT docs (never training-data memory), pin + confirm the installed version, re-verify on every touch — writing, modifying, reviewing, AND auditing. Call out breaking changes you find.
- **Follow external specs** exactly — implement THAT spec, cite the URL/version, say which parts are out rather than fudging.
- **No fake features / scaffolding theater.** Every route/page/table ships with a real handler / render / reader + writer. If you can't implement something real this turn, don't ship the surface.
- **Surgical purges.** When told to remove something, remove it completely (code, generated files, DB, routes, UI, nav, docs, references); grep for residual references; leave typecheck + lint green.
- **Reuse existing primitives** — search for similar implementations first, study 2–3 references, match established tokens/patterns; never invent a parallel one. When no pattern exists, say so and get confirmation.
- **User-facing naming is a product decision** — never adopt a worse label to dodge an internal migration; propose the rename.
- **Phase handoffs — always ask, never auto-advance.** End each phase with a status + a concrete go/no-go question; wait for explicit approval before the next phase.
- **Don't promise timelines/outcomes** ("production-ready", "fully tested", "24/7") unless literally true — describe what you did, what works, what doesn't, what's next.
- **Reply style — short, conversational.** A few lines by default; one question at a time; no bullet-heavy walls unless needed; `path:line` references; a one- to two-sentence end-of-turn summary.

## Rules

1. Never perform SSH or infrastructure operations directly in Next.js routes — always enqueue a pg-boss job and handle it in the worker.
2. Always validate user permissions — API routes use `lib/api/auth-helpers.ts`, server actions use `lib/actions/auth-helpers.ts`. Never reimplement permission checking inline.
3. When creating or updating Cubes, validate resource values against `config/platform.ts` using `isValidRangeValue()` from `lib/cube-options.ts`.
4. **No raw SQL** — use Drizzle ORM helpers for all database interactions. Only use `sql` template literals for DDL scripts or operations with no ORM equivalent.
5. All environment variables must be defined in `lib/env.ts` with Zod validation — never access `process.env` directly.
6. **Schema changes go through `pnpm db:generate` — never hand-write migration files or `_journal.json` entries.** Workflow: edit `db/schema/*.ts` → run `pnpm db:generate` (drizzle-kit creates the SQL file, the `meta/<idx>_snapshot.json`, and the matching `_journal.json` entry with a correct `Date.now()` `when` value) → commit all three → `pnpm db:migrate`. The worker container runs `pnpm db:migrate` automatically on startup (its `CMD` is `worker:deploy` = `db:migrate && worker:start`), so each deploy applies pending migrations before the pg-boss worker boots; run `pnpm db:migrate` by hand only for local dev. `pnpm db:push` is dev-only. **Why this rule exists**: drizzle-kit's migrator compares `entry.when > lastApplied.created_at` and silently skips any migration whose `when` is older than what's already in `__drizzle_migrations`. Hand-written entries with the wrong epoch (e.g. typed `2025` instead of `2026`) make `pnpm db:migrate` look successful while applying nothing — the production-down incident on 2026-05-04 with the `user_data` column was caused exactly this way. Hand-edited migration files also skip the snapshot, which corrupts every subsequent `pnpm db:generate` diff. If you ever need to fix an existing migration (e.g. add `IF NOT EXISTS` guards for re-run safety), edit the SQL only and leave the journal/snapshot alone.
7. All background job handlers must be idempotent and handle retries gracefully.
8. Always write lifecycle logs for any significant entity changes to maintain an audit trail.
9. Always write audit logs for every mutation using `audit()` from `lib/audit.ts` — in server actions, API routes, worker handlers, and auth hooks.
10. Use React Email components in `lib/email/components/` for all outgoing emails. Never use inline HTML strings for emails.
11. Follow the established file and directory conventions for route groups, service layer organization, and database schema.
12. Use Sheet (side panel) for any UI collecting user input. Use AlertDialog only for confirmations. Never use Dialog for input forms.
13. In `.map()` calls that return multiple elements, use `<Fragment key={...}>` — never use keyless `<>` shorthand.
14. **No code duplication** — shared business logic must live in `lib/` modules, never be copy-pasted across files (validation, cost calc, credit checks, billing summaries, server allocation, member management, auth checks, pagination, cube state transitions, storage backend deletion).
15. Every server must be assigned to a region. Region is required when creating a server.
16. When decrementing server resources in worker handlers, always read the current value first, then set with `Math.max(0, ...)` — never use raw SQL arithmetic on columns.
17. **Port allocation is dynamic** — `allocated_ports` only stores ports currently in use. Use `allocatePort()` / `freePort()` / `freePortsByCube()` from `lib/server/ports.ts`.
18. **No bulk pre-seeding** — never insert thousands of placeholder rows. If something has a known range, track only what's in use and derive availability at runtime.
19. **All forms must use react-hook-form** with `zodResolver`. Errors display inline via `<FormMessage />` — never use `toast.error()` for validation. Submit buttons disabled until valid (and dirty for edit forms). Server errors via `form.setError("root", ...)`.
20. **Adjacent buttons must match height** — never mix button `size` variants in the same row.
21. **Non-submit buttons inside forms** must have `type="button"` to prevent accidental form submission.
22. **Keep CLAUDE.md, README.md, and the extracted detail docs up to date** — when adding features, changing architecture, adding new rules/conventions, modifying setup, or altering the database schema, update them. CLAUDE.md is a lean index; the detailed subsystem references live in [docs/architecture/](docs/architecture/) (one file per subsystem, mapped from the Architecture section), the full command catalog in [docs/commands.md](docs/commands.md), and the behavioral-guideline rationale in [docs/development-practices.md](docs/development-practices.md). When you change a subsystem, update its `docs/architecture/*` file — not just the CLAUDE.md summary — and add a pointer in the Architecture map (and `docs/architecture/README.md`) if you introduce a new subsystem doc.
23. **Never store file data in the database** — snapshots/backups go to S3-compatible storage backends; built images live on the Dokploy host filesystem. The DB stores only references (object keys, metadata, hashes).
24. **Snapshot/backup transfers are server-side** — the worker orchestrates via SSH commands on the host. The worker process never downloads or uploads snapshot file data through its own memory. **Snapshots** go through `lib/storage/restic/` (host-side `restic backup`/`restore`/`forget --prune` — per-cube content-addressed dedup repo). **Backups** go through `lib/storage/s3-transfer.ts` (host-side `rclone copy` multipart — full `.ext4.zst` blob). Lightweight worker→S3 operations (bulk deletion, listing, capacity probe) use `lib/storage/s3-direct.ts` via the AWS SDK. Never re-route snapshot data through the worker process and never use the rclone path for snapshots.
25. **All timestamps must be UTC, and every UI render of a timestamp goes through `<LocalDate />`.** Servers are configured with `timedatectl set-timezone UTC`; all DB timestamps use `withTimezone: true`. Customer- and operator-facing display is formatted in the BROWSER's local timezone via [components/local-date.tsx](components/local-date.tsx) — never call `format()` / `formatDistanceToNow()` / `toLocaleDateString()` / `toLocaleString()` / `Intl.DateTimeFormat` inline. The reason is the Next.js SSR gotcha: code that lives in a server component (no `"use client"` directive) — or in a component that is imported by a client component and therefore SSR-rendered server-side first — runs in the server's Node process, which is UTC. An inline `format()` there ships UTC in the HTML and either renders UTC verbatim (true server pages, no hydration) or causes a hydration mismatch (client trees, where the browser then re-renders in local time). `<LocalDate iso={...} />` solves both: it's `"use client"`, so the formatter runs in the browser, and it uses `suppressHydrationWarning` so the intentional server↔client mismatch doesn't pollute the console. Modes: default (absolute `"MMM d, yyyy HH:mm"`), `mode="relative"` (relative time with absolute tooltip), `mode="date"` (date only). For **email templates**, never use `<LocalDate />` (no JS in inboxes) — use `formatEmailDateUtc(date)` from `lib/email/helpers.ts` instead, which renders an explicit `"<Month> <day>, <year> at <HH>:<MM> UTC"` string so the recipient can interpret it from any timezone.
26. **Use `useWatch` instead of `form.watch()`** — `form.watch()` is incompatible with the React Compiler. Use `useWatch({ control: form.control, name: "field" })`.
27. **Client data fetching uses SWR, never `fetch` inside `useEffect`** — use `useSWR` from `swr` with `fetcher` from `@/lib/fetcher`. The `useState` + `useEffect` + `fetch` pattern trips `react-hooks/set-state-in-effect`.
28. **External store subscriptions use `useSyncExternalStore`** — for state outside React (media queries, embla, event-emitting libs), use `React.useSyncExternalStore` instead of `useState` + `useEffect` + `addEventListener`.
29. **Reset derived state with "adjust state during render", not `useEffect`** — when state needs to reset on a dependency change, compare a key via `useState` during render and call the setter inline.
30. **All non-sensitive platform config lives in `config/platform.ts`** — credit rates, cube resource ranges, OS images, snapshot settings, error notify emails, product name, logo path, default credit grant. Sensitive credentials (SMTP, Pusher, OAuth) are env-only.
31. **Multi-backend architecture for snapshots/backups** — use `selectBackend()` from `lib/storage/backends.ts` for uploads (picks by free capacity) and `getBackendConnection(backendId)` for downloads/deletes. Always store `storageBackendId` alongside `storagePath`.
32. **Admin API input validation** — all admin API routes (`/api/orbit/*`) must validate input types and ranges. Numeric fields must be checked as positive numbers, FK references verified to exist before use.
33. **Admin self-modification guard** — admins cannot modify their own admin status via the PATCH user endpoint.
34. **Multi-step background handlers use `JobLogger`** — instantiate from `lib/worker/job-log.ts` at the top of `runHandler` with `(job.id, jobName, entityType, entityId)`, then wrap each unit of work in `await log.step(label, async () => { ... })`. Persists rows to `job_logs` and emits `job.log` Pusher events on `private-{entity}-{id}` so the UI can stream live. **Any handler that keeps a cube in `pending|booting|stopping` longer than 10 minutes MUST wrap the slow span in `withCubeHeartbeat(cubeId, async () => { … })` from `lib/worker/cube-heartbeat.ts`.** `cube.stale-check` flags cubes in those statuses whose `updatedAt` is older than 10 minutes and routes the cleanup through `createPreDeletionBackup()` (from `lib/cubes/create-pre-deletion-backup.ts`) when the cube has actually run before (`cubes.lastStartedAt` is set) — that creates a salvage `cube_backups` row so the customer can redeploy if the auto-cleanup turns out to be a false positive; the `backup.create` job then enqueues `cube.delete` in its `finally` block. If the cube never ran (no rootfs to compress) or the backup-row creation fails, stale-check falls back to enqueueing `cube.delete` directly. Without a heartbeat, the delete races the handler and `rm -rf`s the in-flight rootfs (the 2026-05-21 "Decompressed rootfs is too small or missing (NaN bytes)" failure mode). Currently used in `backup-create` (compress + upload) and `backup-redeploy` (download + decompress); add it to any future slow-SFTP / slow-compress handler that holds those statuses.
35. **Server hardware totals are auto-detected, not entered** — `totalCpus`, `totalRamMb`, `totalDiskGb` on `servers` are written by the bootstrap handler from `nproc`/`/proc/meminfo`/`df -B1G /`. The create-server form does not collect them. Capacity stays at `0` until bootstrap completes; allocation only considers `status='active'` servers, so the gap is safe.
36. **No host snapshots — Timeshift was removed (2026-05-29).** The platform no longer takes Timeshift (or any) snapshots of bare-metal hosts. Timeshift was fragile on real server hardware (it segfaulted with exit 139 during the pre-bootstrap rsync on a bare-metal box) and the per-phase rollback it backed was deemed unnecessary — setup phases are idempotent, so a failed phase is recovered by retrying it, and admin actions on `ready` servers (`update-images`, `refresh-caddy`, `update-caddy`) run without a pre-action snapshot. The `server_snapshots` table, `server.restore-snapshot` job, `lib/server/timeshift.ts`, and the per-phase Restore UI were all deleted (migration `0063` drops the table + `server_snapshot_phase` enum). **Do NOT reintroduce Timeshift or a `takeSnapshot` call.** If host rollback is ever genuinely needed, build it on a tool proven to work on server RAID/LVM/ZFS layouts (e.g. a plain `rsync` copy or filesystem-native snapshots), not Timeshift. Bootstrap keeps its own sshd_config rollback marker (a backgrounded-sleep auto-restore) — that is unrelated to Timeshift and stays.
37. **Rootfs uses the distro-default `iptables-nft` backend, paired with `CONFIG_NF_TABLES`-enabled kernel.** Both layers must agree:
    - **Kernel side** (`setup/images/build-all-images.sh` `build_kernel`): `CONFIG_NF_TABLES=y` plus the full `NFT_*` family (NAT, MASQ, REDIR, COMPAT, FIB, etc. — see the kernel options block in the build script). Verified by the REQUIRED list at end of `build_kernel`. Without these, ANY tool calling `iptables-nft` (which is the distro default) fails with `iptables: Failed to initialize nft: Protocol not supported` and Docker / firewalld / kube-proxy refuse to start. This is the bug we hit early in Phase F before adding nftables to the kernel.
    - **Rootfs side**: leave the distro-shipped default in place. Ubuntu 24.04 ships `iptables-nft` as the `iptables` binary. **Do NOT add `update-alternatives --set iptables /usr/sbin/iptables-legacy`** (or the RHEL `alternatives` equivalent) — RHEL 9 / AlmaLinux 9 don't even ship an `iptables-legacy` package, so the dnf install would fail with `No match for argument: iptables-legacy`. Aligning with the distro default is also forward-compatible with future Linux distros where iptables-legacy may be removed entirely.
    - **If `iptables: Failed to initialize nft` reappears in cube logs**: it means a kernel rebuild silently dropped a `CONFIG_NFT_*` option. Check the REQUIRED-config verification at end of `build_kernel` and the post-`olddefconfig` warnings.
38. **Cube billing fairness — never charge unexpected shutdowns** — `billing-hourly`'s **running-compute** pass only charges cubes with `status="running"`, so error / sleeping / booting / pending cubes are never hourly-charged for compute. Prorated compute charges for the partial hour since `lastBilledAt` only fire from CUSTOMER-INITIATED stop handlers: `cube-sleep.ts`, `cube-delete.ts`, `snapshot-restore.ts`. The `cube-state-sync.ts` periodic sync DOES NOT call `chargeProratedUsage` when it detects an unexpected shutdown (DB="running" + hypervisor="shut off"/"paused") — those transitions are by definition not customer-initiated (kernel panic, OOM, host reboot, or `shutdown -h` from inside the guest bypassing our flow), and charging for the partial hour of compute leading up to the crash is unfair. Past full-hour compute charges from the cron stand because they covered actual running time. When adding any new code path that transitions a cube to `error` or `sleeping`, follow the same rule: charge prorated only when the customer explicitly asked for the stop. Never charge compute for cubes that are entering or already in `error` state. **Sleep-storage billing is independent and applies to every `status="sleeping"` cube** — including cubes that entered sleep via an unexpected hypervisor shutdown. Disk occupancy on the host is the same regardless of intent, so a sleeping cube always pays `DISK_RATE × diskLimitGb × tier multiplier` per hourly tick — same per-GB rate AND full-disk basis as running disk (RAM and disk are sold 1:1 with the host, no overselling, no free-disk allowance). Single source of truth: `config/platform.ts`, via `calculateSleepHourlyCost()` in `lib/cost-shared.ts`. Set `DISK_RATE = 0` in `config/platform.ts` to disable (also disables running-disk billing in lockstep); the fairness rule is about **compute**, not storage.
39. **Quoting traps inside `bash -c '...'` blocks** — `setup/images/build-all-images.sh` and any other script that ships a payload to a remote shell via `bash -c '<long script>'` has the entire body wrapped in single quotes. Two distinct mistakes break it; both have happened in this codebase:
    1. **Apostrophes anywhere in the block** (comments, error strings, heredoc bodies — heredocs do NOT escape the outer quoting) terminate the outer single quote and produce `unexpected EOF while looking for matching '`. Write `arent` / `cant` / `isnt` / `Firecrackers DSDT` / `device vda`, never `aren't` / `can't` / `isn't` / `Firecracker's DSDT` / `device 'vda'`.
    2. **Backticks or `$var` / `$(cmd)` inside an UNQUOTED heredoc body** (`<<EOF`) get expanded by the inner shell when `cat << EOF` runs, even when the heredoc is wrapping plain config text. A backticked phrase like `` `virtio_mmio.device=4K@0xc0001000:5` `` will try to execute as a command, fail, and silently corrupt the heredoc output. Always use a quoted delimiter — `<<"EOF"` (NOT `<<'EOF'`, since apostrophes break the outer wrapping per rule 1). The double-quoted delimiter is exactly equivalent to single-quoted for expansion-suppression and survives the outer quoting.
       After every edit to a script using this pattern, run `bash -n <script>` to confirm syntax is still valid before claiming it ships.
40. **Production data safety — live customers, live data, live billing.** This system runs in production with real users and real money. Every database change must be treated with the same care you would give a surgical procedure on a patient who is awake.
    - **All DDL must be additive and non-locking.** Adding nullable columns (`ALTER TABLE … ADD COLUMN … TEXT`) is safe in PostgreSQL 11+ (metadata-only, no table rewrite). Adding a column with a `DEFAULT` constant is also safe in PG 11+. Never `DROP COLUMN`, `ALTER COLUMN TYPE`, or `RENAME COLUMN` on a live table without a multi-step shadow-column migration — those operations rewrite or lock the table.
    - **Never DROP a table, index, or constraint while code that references it is deployed.** Remove the code reference first, deploy, then drop the object in a follow-up migration.
    - **All migrations must be idempotent.** Use `ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, `DO $$ BEGIN … EXCEPTION WHEN duplicate_column THEN NULL; END $$` patterns. This lets migrations be re-run safely if a deploy is interrupted or retried.
    - **Data backfill UPDATEs must be bounded and idempotent.** Never run an unbounded `UPDATE` on a large table in a single transaction during a live deploy — it locks rows and blocks reads/writes. For large tables (>10k rows), batch the UPDATE in chunks of 1000–5000 rows with `WHERE id > $last_id LIMIT 5000`. Always use a `WHERE` condition that makes re-runs safe (e.g. `WHERE role IS NULL` not `WHERE 1=1`).
    - **Migration deploy order: migrate first, then deploy code.** New columns must exist in the DB before the code that reads them goes live. Never deploy code that references a column that does not yet exist. If a migration removes something the old code still uses, deploy the new code first, then migrate.
    - **Never run `pnpm db:push` against production.** `db:push` bypasses the migration journal and can silently drop columns or constraints. It is dev-only.
    - **Before any schema change, answer these questions:** (1) Which live queries touch this table? (2) Does this change lock rows? (3) Is the migration reversible? (4) If it fails halfway, what state is the DB in, and is the running app still functional?
    - **Data migrations that set values on existing rows** (like backfilling `role` from `isAdmin`) must be written so they can be re-run without corrupting already-correct rows. Use `WHERE role IS NULL` not `WHERE 1=1`.
41. **virtio-mem boot floor is invariant.** Every cube must boot with `mem_size_mib=VIRTIO_MEM_BOOT_FLOOR_MIB` (currently 1024 MiB) regardless of the customer's configured `ramMb`. The remainder is hot-plugged via `PATCH /hotplug/memory { requested_size_mib }` after `InstanceStart`. Never raise the boot floor for an existing cube — `mem_size_mib` is set at boot and cannot shrink. If the platform RAM minimum ever changes, `VIRTIO_MEM_BOOT_FLOOR_MIB` and `VIRTIO_MEM_TOTAL_SIZE_MIB` in `config/platform.ts` must be updated together (the total = max RAM − floor, and must remain a multiple of `VIRTIO_MEM_SLOT_SIZE_MIB`).
42. **Polar customer is per-EMAIL, not per-SPACE — never address a Polar customer by `external_id = spaceId`.** Polar enforces email uniqueness per organization AND `external_id` immutability. When a user subscribes their second (or third, …) space, Polar reuses the existing customer record whose `external_id` is pinned to their FIRST space's id — every sibling space shares the same Polar customer. Any API call that takes `externalId: spaceId` (`customers.getStateExternal`, `customers.updateExternal`) silently 404s for a sibling, and any event sent with `externalCustomerId: spaceId` either fails or auto-creates a phantom customer. **Always resolve `spaces.polar_customer_id` first** and use the canonical id (`subscriptions.list({metadata.spaceId})`, `customers.update({id})`, `events.ingest({customerId})`). For lookup-by-space, use `subscriptions.list({metadata: {spaceId}, active: true})` — the metadata-filter route works for every space regardless of email collision; never query the customer state by external id. The reconcile cron MUST also defend in depth: if `getActiveSubscriptionForSpace` returns empty for a space that still has a recorded `providerSubscriptionId`, verify by `getSubscription(id)` before synthesizing a `canceled` event — Polar returning empty from the list query does NOT prove the subscription is gone (it may be a metadata-tagging gap on the Polar side). The 2026-05-21 incident downgraded a live customer's space to Trial every hour because the reconcile cron skipped this check.
43. **`useMutation` + `revalidate: false` is a defensive pattern, not stale code — do not remove without understanding why.** `hooks/use-mutation.ts` calls `router.refresh()` after a successful mutation when `revalidate !== false`. Per Next.js 16 docs, `router.refresh()` always refreshes the **current** route. So on a mutation that destroys the current entity (the three `triggerDelete` handlers on `components/orbit/space-detail.tsx`, `user-detail.tsx`, `server-detail.tsx`), letting the default refresh fire causes the current-page server component to re-fetch a now-deleted entity, hit `notFound()`, and flash a 404 page before the navigation lands. The correct pattern for these is `{ revalidate: false, onSuccess: () => { router.push(targetList); router.refresh() } }` — skip the current-page refresh, navigate, then refresh the destination. For mutations that stay on the same page (any list-page delete / status-change), the default `revalidate: true` is correct and no override is needed.
44. **Single source of truth for status displays — extend `lib/status-display.ts`, not the consumer.** Every status enum (`cube` / `snapshot` / `credit-purchase` / `domain` / `subscription` / `server` / `server-setup-phase` / `billing-event-type` / Cloudflare hostname / generic resource status) routes its filter-dropdown options, Badge variant, and color-class through `lib/status-display.ts`. 46. **Every host-side binary the worker shells out to MUST be installed by `server-install.ts` AND verified by the `verify host tools` step AND retrofitted by `pnpm install:host-tools`.** The bug class this rule exists to catch: adding a `nc -z ...` or `file -b ...` shell invocation somewhere in `lib/worker/handlers/` without making sure the binary is actually on the host. The 2026-05-24 SSH-reachability outage was exactly this — `cube-reachability.ts` shelled out to `nc`, but `nc` was never in the base-packages install list, so the L2 SSH probe silently failed on every host for the entire lifetime of the feature. The mandatory sequence when introducing a new host-side dependency: (1) add the distro package(s) to the `base packages` step in [server-install.ts](lib/worker/handlers/server-install.ts) (Debian/Ubuntu + RHEL branches both); (2) add the binary to the REQUIRED list in the `verify host tools` step at the end of the same file; (3) add the same package(s) to the install command in [scripts/install-host-tools.ts](scripts/install-host-tools.ts); (4) if the tool has meaningful version differences (multipart performance, repo format, CLI flag drift), add a `<TOOL>_VERSION` constant to `config/platform.ts` and follow the rclone/restic install-step pattern with curl + verify-then-install + ground-truth `grep -qF EXPECTED` gate. Coreutils, util-linux, iproute2, procps, and systemd binaries are exempt — they're framework-level guarantees on every supported distro. Everything else is in scope. Tools we already install upstream-from-distro: `caddy` (`CADDY_VERSION`), `restic` (`RESTIC_VERSION`), `rclone` (`RCLONE_VERSION`), `firecracker` (`FIRECRACKER_VERSION`).
47. **Auto-snapshot retention scopes `restic forget --keep-last/daily/weekly` to EXACTLY the auto snapshots via repeated `--tag`, because restic has NO `--keep-id` flag.** `--keep-id` is a `rustic`-only flag — verified against restic 0.18.1 source (`cmd/restic/cmd_forget.go` registers only `--keep-last/-hourly/-daily/-weekly/-monthly/-yearly`, `--keep-within*`, `--keep-tag`) and the 0.18.1 forget docs. The earlier implementation emitted `--keep-id`, so `restic forget` exited non-zero and the daily cron **silently no-op'd for every cube with a pinned/manual snapshot** — those per-cube repos + `storage_backends.usedBytes` grew unbounded (caught by the 2026-05-31 snapshots/backups audit; cubes with zero manual snapshots pruned fine, masking it). The correct mechanism: every snapshot is tagged in restic with its own unique `cube_snapshots.id` (attached at backup time by `resticBackup` — NOT a shared `auto` tag), so the auto-prune handler [snapshot-auto-prune.ts](lib/worker/handlers/snapshot-auto-prune.ts) collects every `kind='auto' status='complete'` snapshot's id and `buildResticForgetArgs` ([lib/snapshots/forget-args.ts](lib/snapshots/forget-args.ts)) emits one `--tag <id>` per auto snapshot ahead of the `--keep-*` policy. restic ORs repeated `--tag` flags and **leaves any snapshot whose tag is absent from the list completely untouched** (verified against the 0.18.1 forget docs), so manual/pinned snapshots — a `kind='manual'` DB row, deliberately excluded from the list — are never forget candidates, with no dependency on a shared tag and no `restic rewrite` at pin time. `buildResticForgetArgs` returns `null` (handler SKIPS the prune) when EITHER every bucket is 0 OR `autoTagIds` is empty; the second is a **HARD SAFETY guard** — a `forget --keep-* --prune` with NO `--tag` filter applies the policy to the WHOLE repo and could forget a pinned snapshot. NEVER emit a tag-less retention forget; NEVER reintroduce `--keep-id`. Pure-unit-tested in [lib/snapshots/forget-args.test.ts](lib/snapshots/forget-args.test.ts). The 2026-05-25 snapshot overhaul added this rule; the 2026-05-31 audit corrected the mechanism.

48. **The cube's SSH port is NEVER hardcoded as `22` — it lives in `tcp_port_mappings.cubePort` on the `isSsh=true` row, and it has its own endpoint separate from the generic TCP-mapping CRUD.** Every code path that touches sshd's port (iptables DNAT install/removal, the reachability cron's L2 probe, the lifecycle log message, the `tcp_port_mappings` insert) reads or writes this column. The boot-time default comes from `DEFAULT_CUBE_SSH_PORT` in [config/platform.ts](config/platform.ts) (currently 22 — what the rootfs ships sshd on); never literal-write `22` anywhere else. **The customer changes the port via `PUT /api/spaces/[spaceId]/cubes/[cubeId]/ssh-port` with body `{ cubePort: <int> }`** ([app/api/spaces/[spaceId]/cubes/[cubeId]/ssh-port/route.ts](app/api/spaces/[spaceId]/cubes/[cubeId]/ssh-port/route.ts)) — a one-per-cube resource, NOT a row in the TCP-mapping collection. Generic TCP-mapping CRUD (`POST /tcp-mappings`, `DELETE /tcp-mappings/[id]`, whitelist edit, exposure toggle, isSsh toggle) stays under [app/api/spaces/[spaceId]/cubes/[cubeId]/tcp-mappings/](app/api/spaces/[spaceId]/cubes/[cubeId]/tcp-mappings/); the SSH port lives outside that tree because there's only one SSH mapping per cube, it's created automatically at boot, and it can't be deleted. 50. **`cube-sleep` / `cube-power-off` error recovery MUST forward-flip to `sleeping` once the hypervisor is actually paused/killed — never revert to `running`.** Both handlers track a `vmPaused` / `vmKilled` boolean that flips true the instant `sleepCube` / `powerOffCube` returns. 52. **The `lastBilledAt ⇔ status='running'` invariant.** A cube in `status='sleeping'` MUST have `lastBilledAt = null`; a cube in `status='running'` MUST have `lastBilledAt != null` (set to `new Date()` at every start: provision, wake, cold-restart, snapshot-restore, transfer, auto-relaunch, reboot-recovery, resize). The invariant exists because the hourly cron charges on `status='running'` and prorates from `lastBilledAt`. Breaking it on the sleeping side (the previous bug in `cube-resize` / `cube-transfer`) doesn't immediately mis-charge because the cron's status filter saves us — but it's a time bomb for any future code path that ever uses `lastBilledAt IS NOT NULL` as a "billable" check without joining on status. Both `cube-resize.ts:430` and `cube-transfer.ts:844` now write `lastBilledAt: wasRunning ? new Date() : null` to preserve the invariant when the cube ends sleeping.

53. **Sleep storage uses CUBE DISK PRICING ON THE FULL DISK SIZE — ALWAYS ON, single source of truth in `config/platform.ts`.** Sleeping cubes are charged hourly at `DISK_RATE × diskLimitGb × multiplier` (DISK_RATE is per HOUR, not per month). Same per-GB rate AND full-disk basis as the running-disk component — running and sleeping cubes both occupy every allocated GB on the host. **Krova sells RAM and disk 1:1 with the host and does not oversell either resource; there is no free-disk allowance.** Single source of truth: `calculateSleepHourlyCost()` in [lib/cost-shared.ts](lib/cost-shared.ts), used by both the worker billing pass ([billing-hourly.ts](lib/worker/handlers/billing-hourly.ts)) and the burn-rate query ([lib/billing.ts](lib/billing.ts) `getSpaceBurnRate`) so the customer-facing runway projection always matches what the worker actually charges. **There is no operator toggle, no Orbit UI knob, no per-installation override** — setting `DISK_RATE = 0` in `config/platform.ts` is the ONLY way to disable sleep-storage billing, and it disables running-disk billing in lockstep. The classifier derives from the `billingEventType` pgEnum's `enumValues` so a missing branch fails TypeScript at compile time when a new enum value is introduced. Adding a new charge type = add to the pgEnum + add to the Set in `billing-events.ts`; nothing else.

55. **The hourly cron and the prorated-charge path MUST apply the same per-event hour ceiling.** [billing-hourly.ts](lib/worker/handlers/billing-hourly.ts) clamps each tick's billable window with `Math.min(elapsedHours, 1)`; `chargeProratedUsage` in [lib/cost.ts](lib/cost.ts) MUST clamp identically (`const billedHours = Math.min(elapsedHours, 1)`) and use `billedHours` for the charge AND for every customer-facing description, lifecycle log, and audit row — never the raw `elapsedHours`. Both paths bill the same elapsed window since `lastBilledAt`; if their ceilings differed, a customer-initiated stop fired during a STALLED-cron window would bill the entire backlog on the prorated path while the cron caps each tick at 1h — the same time charged under two different ceilings. The cap deliberately favors the customer when our own worker is delayed. The 2026-05-29 audit found the prorated path missing the clamp the cron has always had; any new elapsed-time billing path inherits this rule.

56. **Every `JOB_NAMES` queue MUST have an explicit entry in `QUEUE_OPTIONS` ([lib/worker/ensure-queues.ts](lib/worker/ensure-queues.ts)).** That object is a FULL `Record<JobName, …>` (NOT `Partial`) precisely so `pnpm typecheck` / `pnpm build` FAILS the moment a job is added to `JOB_NAMES` without a queue config — a compile-time guard against a job silently inheriting pg-boss defaults. The bug this catches (2026-05-29 audit): `CUBE_TRANSFER_CANCEL` ran on the default `standard` policy with no explicit retry/expire because it was never added to `QUEUE_OPTIONS`, and since a cube stranded in `cancelling` has no reaper, a terminal handler failure could have stuck it forever. If a job genuinely wants pg-boss defaults, give it an explicit `{}` entry with a one-line comment saying why — never leave the choice implicit. This is IN ADDITION TO the recurring-queue rule (every `boss.schedule()` queue MUST be `policy: "exclusive"`).

57. **In cancel / rollback / error-cleanup handlers, a customer-facing or externally-visible restore step that has no dependency on a particular guard MUST run at top level — never nested inside an unrelated conditional that can skip it.** The 2026-05-29 audit found `cube.transfer-cancel`'s Cloudflare origin-restore (re-points a customer's custom domain back to the live source cube — a pure Cloudflare API call with NO host dependency) nested inside the `if (destId) { … }` destination-cleanup block; when `destId` resolved null the restore was skipped entirely and the customer's domain stayed pointed at a torn-down destination — the exact failure the feature existed to prevent. Pattern: gate each cleanup step ONLY on its own precondition (origin-restore → `shouldRevertDomainRouting && activeDomains.length`; destination-dir wipe → `destId`; source-wake → `shouldWakeSource`). When a step is critical or customer-visible, hoist it OUT of the broadest guard and cross-check the same handler's sibling paths ([cube-transfer.ts](lib/worker/handlers/cube-transfer.ts)'s own rollback already runs the equivalent restore at top level).

58. **PREFLIGHT ALL CHECKS — every precondition that CAN be evaluated before the first side effect MUST run at the TOP of the process, and a failing check MUST abort before anything is mutated.** This applies to EVERY process: worker job handlers (`cube.provision`, `cube.transfer`, `cube.transfer-cancel`, `snapshot.*`, `backup.*`, `cube.import-rootfs`, `cube.resize`, the `server.*` setup phases), server actions, and API routes. The bug class this exists to stop (it caused multiple 2026-05 production incidents): a check — "does the destination server still exist / have capacity?", "is the source/host reachable?", "is the cube in the expected state?", "is the plan within limits?" — placed AFTER the handler already claimed an intermediate-state row, started an SSH op, wrote to a host, or called a third-party API. When a late check then fails, the process aborts with partial state already applied (a row stranded in `creating`/`booting`/`cancelling`, an orphaned cube dir on a host, a half-re-pointed domain). The order is non-negotiable:
    - **Phase 1 — PREFLIGHT (read-only, no side effects):** input/Zod validation, existence (FK rows present), permission/auth, plan-limit + capacity (`serverHasDiskRoom`, overcommit), state-machine guards (cube/transfer status is what you expect), and best-effort reachability (`isServerReachable`). Evaluate ALL of these BEFORE the first mutation. If any fails, return/throw immediately — nothing has been touched, so there is nothing to clean up.
    - **Phase 2 — COMMIT (side effects):** only after every preflight check passed, claim the row / SSH / write to the host / call the provider.
    - **Checks that GENUINELY depend on intermediate results** (e.g. "did the rsync produce a valid ext4?", "did `InstanceStart` return a PID?") cannot be hoisted — they stay inline, but they MUST be inside the guarded `try` whose `catch` rolls the row back to a clean terminal state (the guarded-connect invariant) so a failure never strands partial state. A check is only "inline-legitimate" if it literally cannot run earlier; if it CAN run earlier, it MUST.
    - **When you ADD a new check to an existing process, default to placing it in the Phase-1 preflight block.** Only leave it inline if it depends on Phase-2 state, and say so in a comment. Never drop a new check into the middle of a side-effecting sequence "where it happens to be convenient."
    - **When you MOVE a check earlier, verify it has no dependency on mid-process state** before hoisting — a check that reads a value computed mid-process will break or read stale data if naively moved up. Hoist only checks whose inputs are all available at entry. When unsure, leave it and flag it rather than risk a regression (rearranging checks carelessly is itself what broke production).

59. **Tests are not optional — every change ships with tests, and "done" means the suite is GREEN.** This is a live production system with real customers and real billing; a regression is negative marketing, not just a bug. The discipline:
    - **The four suites, and the one command.** `pnpm test` (Tier-1 pure unit — `tsx --test` over `lib/**/*.test.ts`, no DB/host, runs anywhere), `pnpm test:integration` (DB-backed — spins a throwaway `postgres:18` on port 55434, applies the full migration chain, runs `tests/integration/**/*.test.ts` against REAL rows via `node --env-file=.env.test --import tsx --test`), `pnpm test:migrations` (the Rule-6 migration-chain smoke), and `pnpm test:host <ssh-target>` (Tier-2 Firecracker lifecycle on a real `/dev/kvm` host — manual, pre-release). **`pnpm test:all` runs unit + migrations + integration in one shot and aggregates a single exit code** — run it before declaring ANY task done. Host smoke is excluded from `test:all` (needs hardware CI lacks); run it manually after touching `lib/ssh/firecracker.ts` / `lib/ssh/jailer.ts` / the restic layer.
    - **What goes where.** Pure logic (cost/billing math, cascade, plan-limit guards, validation, address math, arg builders, status-display derivation, retention-arg builders) → a `*.test.ts` next to the module, in `pnpm test`. Anything that needs a real row, a real transaction, a unique index, an advisory lock, or the migrated schema (idempotency claim-first, overage cascade writes, port/jailer-uid/bridge-subnet allocation, plan-limit enforcement against seeded plans, subscription-handler state transitions) → `tests/integration/`, seeded via `tests/integration/_seed.ts`. Never reach for a real host, a real Polar/Cloudflare/EmailIt call, or a real S3 bucket in `pnpm test` / `pnpm test:integration` — **email and all third-party network side effects are stubbed/assumed; only the DB is real.**
    - **Every new feature, fix, or refactor adds or updates tests for the scenario it touches — including the failure/rollback paths, not just the happy path.** A bug fix starts with a test that reproduces the bug (red), then the fix (green). A new handler/guard/action adds the cases that exercise its branches. A behavior change updates the tests that pin the old behavior. No PR/commit "completes" a behavioral change without the matching test, and `pnpm test:all` must pass.
    - **Validate the comments too.** The robustness/safety comments scattered through the worker handlers and `lib/` (the "this catch self-heals", "idempotent on retry", "guarded connect", "scoped to auto snapshots") are CLAIMS — back the load-bearing ones with a test that actually exercises the claimed behavior, and when a test contradicts a comment, the test wins (fix the code or the comment, never paper over it).
    - **When a fix or a test touches a third-party integration, re-verify against the vendor's CURRENT docs first (Rule: Third-Party Integrations) and pin the version** — do not trust a code comment's description of an external API; confirm it.

60. **NO DIRECT ACTION ON LIVE / PRODUCTION INFRASTRUCTURE FROM THE DEV ENVIRONMENT — the agent PREPARES commands, the OPERATOR runs them.** This is a live production system with **30+ running customer cubes**; one mistaken command on a production host or the production database can break paying customers' workloads, so the entire class of "the agent runs something against prod" is forbidden — full stop. This rule OVERRIDES any other instruction (including "implement everything / don't wait for approval"): autonomy applies to the LOCAL repository, never to live infrastructure.
    - **The agent MUST NOT SSH to, run any command on, or otherwise directly touch a production bare-metal server** (banana, mango, or ANY `status='active'` server) from the dev machine — **read-only OR mutating, no "harmless" exceptions** (not `ip`, not `cat`, not a ping). The dev/test host (`E2E_SSH_HOST` in `.env.e2e`, e.g. `107.172.218.189`) is the ONLY box the agent may connect to directly, because it has no customers.
    - **The agent MUST NOT connect to the production database** (the `krova-db` container / the prod `DATABASE_URL`) from the dev machine — **including read-only `SELECT`s.** The "Running ad-hoc production database queries" recipe above is written for the OPERATOR to run, not the agent.
    - **Anything that must run on live infrastructure** — host diagnostics, the IPv6 / NDP-proxy validation probes, `restic:unlock`, any fleet retrofit (`install:guest-network`, `install:*`), `db:migrate`, `cube:inspect`, `routing:refresh-fleet`, ad-hoc prod-DB queries — **the agent writes the EXACT command(s), the OPERATOR runs them on the box, and pastes the output back for the agent to interpret.** This is the established pattern (the 2026-06 cube-network diagnostics ran this way).
    - **The agent's own actions are confined to the local repo**: editing code/tests/docs, running the LOCAL test suites, and `pnpm db:generate` (GENERATING — never applying — migrations). Every production rollout (applying migrations, fleet scripts, server setup/activation, image rebuilds) is operator-run.
    - When in doubt, hand the command to the operator. Validation of live behavior ("does the cube still have internet after this change?") is done by the operator running the agent-prepared command — NEVER by the agent reaching the box.
