### Running ad-hoc production database queries

The production database is the **`postgres:18`** Docker container named **`krova-db`** (currently exposing host port `8888 → 5432`; creds `POSTGRES_USER` = `POSTGRES_DB` = `krovadb`). **There are TWO Postgres containers on the control-plane host** — the other, **`dokploy-postgres`** (`postgres:16`), is Dokploy's OWN internal database; **never run Krova queries against it.** Resolve the right container by name (its id changes on every deploy), then exec `psql` — the `POSTGRES_USER` / `POSTGRES_DB` env vars expand *inside* the container, so credentials stay correct without hardcoding:

```bash
KROVA_DB=$(docker ps --filter "name=krova-db" --format '{{.ID}}')
docker exec -i "$KROVA_DB" sh -c 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB"' <<'SQL'
\pset pager off
SELECT … ;
SQL
```

Default to **read-only** (`SELECT`) for inspection. Schema changes still go through `pnpm db:generate` → `pnpm db:migrate` (Rule 6) — **never** `pnpm db:push` and **never** an unbounded `UPDATE` against production (Rule 40).
