# Cube-lifecycle deep audit — 2026-05-31

Adversarially-verified findings (38) from the 21-area cube-lifecycle audit (Firecracker v1.15.1 / restic 0.18.1 / Polar / Cloudflare). Confidence in [brackets]. Each was independently refuted-or-confirmed by a second agent; verify against code before fixing (the odd-vCPU lesson).

## BLOCKER (4)

### B1. IP whitelist is silently bypassed for every TCP/SSH mapping because the FORWARD-chain match uses hostPort, but DNAT has already rewritten the dest port to cubePort  _[0.95]_
- **Files:** `lib/ssh/network.ts`, `lib/server/cube-network-host.ts`
- **Impact:** Every customer who restricts a TCP mapping or their SSH port to specific source CIDRs gets zero enforcement — the port is reachable from the entire internet exactly as if no whitelist were set. The UI/iptables show the whitelist as applied (the rules exist) but they never match a forwarded packet (0 packet counts), so customers operate under a false sense of security on a live production system.
- **Fix:** Match the whitelist FORWARD rules on the POST-DNAT tuple the packet actually carries in the FORWARD chain: `FORWARD -p tcp -d ${cubeInternalIp} --dport ${cubePort} -s ${cidr} -m comment --comment "tcp-wl-${hostPort}" -j ACCEPT` and the matching DROP. Thread cubePort + cubeInternalIp through applyTcpWhitelist/updateTcpWhitelist (and the update-cube-port swap so the DROP follows a port change). Keep the comment tag keyed on hostPort so clearTcpWhitelist (comment/line-number based) keeps working unchanged. Add a smoke test asserting a non-whitelisted source is actually refused.

### B2. TCP-mapping source-IP whitelist is silently bypassed: FORWARD rules match the host port, but after DNAT the packet carries the cube port  _[0.9]_
- **Files:** `lib/ssh/network.ts`, `lib/worker/handlers/tcp-mapping-add.ts`, `lib/worker/handlers/tcp-mapping-update-whitelist.ts`
- **Impact:** Any customer who restricts a TCP port mapping to specific source CIDRs (a documented, dashboard- and v1-API-exposed security control) gets effectively ZERO enforcement for the overwhelmingly common case where the host port differs from the cube port (always true for SSH: cubePort 22 vs an allocated host port in 30000-50000, and for any DB/web service on a low cube port). The port stays reachable from the entire internet while the UI/API shows the whitelist as active — a silent data-exposure / unauthorized-access risk.
- **Fix:** Key the whitelist rules on the post-DNAT tuple. Thread cubeInternalIp + cubePort into applyTcpWhitelist/updateTcpWhitelist/clearTcpWhitelist (and the update-whitelist worker payload, which currently carries only hostPort) and emit `FORWARD -d ${cubeInternalIp} -p tcp --dport ${cubePort} -s ${cidr} -m comment --comment "tcp-wl-${hostPort}" -j ACCEPT` plus the matching DROP. Keep the comment tag on hostPort for clean removal, but note the comment must stay unique per mapping since multiple cubes can share the same cubePort (e.g. 22) on different internal IPs — include cubeInternalIp in the tag or keep the unique hostPort in it. Alternatively match on conntrack original dest (`-m conntrack --ctorigdstport ${hostPort}`), which works even when multiple cubes share a cube port. Add a unit test asserting the generated rule references the cube port / original-dest port, not a bare host-port dport match.

### B3. Unguarded connectToServer + claim-before-connect: a transient host blip during customer Restart strands the cube in `stopping` and gets it DELETED by stale-check  _[0.88]_
- **Files:** `lib/worker/handlers/cube-cold-restart.ts`, `lib/worker/handlers/cube-stale-check.ts`, `lib/worker/handlers/cube-state-sync.ts`, `lib/ssh/connect-to-server.ts`, `lib/worker/ensure-queues.ts`
- **Impact:** A customer (or operator) clicks Restart on a healthy running cube. If the worker's SSH connect to the host fails for the duration of one job attempt plus its single retry (host briefly down, network partition, sshd momentarily unreachable for ~60s), the cube is stranded in `stopping`, then ~10 min later auto-marked `error` and enqueued for deletion by stale-check. The VM was never killed (the kill is after the failed connect), so a live, healthy workload is torn down. A salvage backup is created (mitigating total data loss), but the cube, its networking, SSH, and TCP/port mappings are destroyed and must be redeployed.
- **Fix:** Mirror cube-error-recovery.ts and honor Rule 58: do the SSH connect (plus a best-effort isServerReachable preflight) BEFORE the atomic `stopping` claim, inside a guarded block whose failure path leaves the cube untouched (still `running`) and returns/throws cleanly so stale-check never sees it. Alternatively, keep the claim first but move connectToServer INSIDE the existing try and, in the catch, recover the cube to `running` (not `error`) when the failure occurred before the kill (the VM is provably still alive at that point), preserving lastBilledAt. Either way the connect must never be able to throw outside a block that restores a clean cube state.

### B4. Cold resize holds the cube in `stopping` with no withCubeHeartbeat — cube.stale-check can salvage-backup + cube.delete the cube mid-resize (data loss, torn rootfs)  _[0.82]_
- **Files:** `lib/worker/handlers/cube-resize.ts`, `lib/worker/handlers/cube-stale-check.ts`, `lib/worker/cube-heartbeat.ts`, `lib/worker/handlers/cube-error-recovery.ts`
- **Impact:** On a large/fragmented rootfs or a fleet-loaded host, a cold resize (any vCPU change, or disk grow needing e2fsck+resize2fs) can exceed the 10-minute stale threshold. When it does, cube.stale-check concurrently (a) reads the live rootfs.ext4 for a salvage backup WHILE the resize is truncating/resize2fs-ing it (torn ext4 captured), and (b) enqueues cube.delete which rm -rf's the cube directory out from under the in-flight startCube/final transaction. The customer's cube is destroyed mid-resize — irreversible data loss for a routine grow.
- **Fix:** Wrap the slow cold-path span (powerOffCube through startCube + optional re-sleep, i.e. lines ~361-442) in `await withCubeHeartbeat(cubeId, async () => { ... })` from lib/worker/cube-heartbeat.ts, mirroring cube-error-recovery.ts:169. The 2-min pulse keeps cubes.updatedAt fresh so cube.stale-check never sees the in-flight resize as stuck. The live path is short and does not need it.

## HIGH (10)

### H1. Hourly/sleep/backup passes double-count overage and count refused (unfunded) amounts as charges in the ledger — overstates customer-visible "Total charged" and per-line debits  _[0.95]_
- **Files:** `lib/worker/handlers/billing-hourly.ts`, `lib/billing/overage.ts`, `lib/cost.ts`, `lib/billing.ts`, `lib/billing/overage-cascade.ts`
- **Impact:** For any space with overage enabled and prepaid drained, the billing page Total charged and the Orbit billing dashboard overstate spend by the overage slice each hour plus any refused amount; Recent Activity shows an hourly_charge for the full hour AND a separate overage_charge for the overage slice of that same hour, so a $1.00 hour funded $0.50 prepaid + $0.50 overage displays as $1.50 charged with two debit lines. Real credit_balance / thisPeriodOverageUsd are correct (no money lost), so it is a ledger/reporting defect, not a charging defect.
- **Fix:** Mirror the prorated path in all three hourly passes: write the per-item charge event for ONLY the prepaid-funded portion (scale each item's amount by cascade.fromPrepaid / actualTotalCost with a divide-by-zero guard, or insert a single space-level prepaid charge row for cascade.fromPrepaid), and let the cascade's overage_charge cover the overage portion. The sum of charge-event rows + overage_charge must equal fromPrepaid + fromOverage and never include refused. Add a unit test asserting sum(charge events) + overage = fromPrepaid + fromOverage for the overage-funded case.

### H2. cube.sleep / cube.wake / cube.power-off do the SSH connect OUTSIDE the guarded try — a transient host-down strands the cube in `stopping`/`booting`, and cube.stale-check then DELETES it (with salvage backup)  _[0.92]_
- **Files:** `lib/worker/handlers/cube-sleep.ts`, `lib/worker/handlers/cube-wake.ts`, `lib/worker/handlers/cube-power-off.ts`, `lib/worker/handlers/cube-stale-check.ts`, `lib/ssh/connect-to-server.ts`, `lib/ssh/connection.ts`, `lib/worker/handlers/cube-state-sync.ts`, `lib/worker/handlers/snapshot-restore.ts`
- **Impact:** A customer who clicks Sleep, Power Off, or Wake (or whose cube is auto-slept by the zero-balance pass at billing-hourly.ts or plan-downgrade reconcile at lib/plan/reconcile.ts) during a transient host network blip (the documented mango-style EHOSTUNREACH) gets the cube stranded in `stopping`/`booting`. If the blip resolves within ~10 min: the cube is stuck and requires manual operator recovery (no auto-path covers those statuses). If the host stays down past 10 min: cube.stale-check marks it `error`, takes a salvage backup (only if the host is reachable at that moment), and cube.delete destroys the on-host rootfs once the host returns — data survives only via the salvage backup, with redeploy losing the internal IP and requiring re-adding port mappings/domains. Wake is the most surprising case: the customer only wanted to resume an intact sleeping rootfs. Exposure is broad because every sleep path routes through these handlers.
- **Fix:** Mirror snapshot-restore.ts:117-153: move connectToServer into its own guarded try/catch placed AFTER the atomic claim in all three handlers. On connect failure — for cube-sleep / cube-power-off revert the cube to `running` (VM never touched; preserve lastBilledAt per Rules 51/52) and `return` (do not throw — the no-longer-`running` row makes a pg-boss retry no-op for sleep, and for power-off returning avoids burning retries on a doomed connect); for cube-wake revert to `sleeping` with lastBilledAt:null and `return`. vmPaused/vmKilled are both false at connect time, so reverting to the pre-op state is always correct. Optionally add a best-effort isServerReachable(serverId) preflight before the claim to skip the doomed transition. Apply the same guarded-connect pattern to cube-delete.ts:93 as a follow-up (it has the identical out-of-try connect).

### H3. cube-power-off claims the cube to `stopping` then calls connectToServer OUTSIDE the guarded try — a host-down blip strands the cube in `stopping`, where stale-check marks it `error` and deletes it  _[0.9]_
- **Files:** `lib/worker/handlers/cube-power-off.ts`, `lib/worker/handlers/cube-sleep.ts`, `lib/worker/handlers/cube-stale-check.ts`, `lib/worker/handlers/cube-state-sync.ts`, `lib/ssh/connection.ts`, `lib/worker/handlers/backup-create.ts`
- **Impact:** A customer powers off a RUNNING cube while its host is briefly unreachable (the exact transient class retries exist to ride out, e.g. the 2026-05-28 mango outage). Instead of reaching the intended `sleeping` state, the cube is committed to `stopping`, is invisible to state-sync, and after ~10-13 min stale-check flips it to `error` and emails admins a false 'stuck cube' alert. If the host recovers around that window, the salvage backup succeeds and the cube is then deleted — the customer's running cube is destroyed by a routine power-off, recoverable only by manually redeploying from the auto-salvage backup (downtime, must re-add domains/TCP mappings). If the host stays down, the cube is left in `error` intact (data-safety net prevents silent unrecoverable loss). Either way the power-off does not do what the customer asked and produces operator noise plus, in the worst case, a destroy-and-redeploy cycle.
- **Fix:** Move the connectToServer call (cube-power-off.ts L74, and the identical cube-sleep.ts L50) INSIDE the existing recovery try (before the powerOffCube/sleepCube step), assigning `client` to an outer `let client: Client | undefined` and guarding the finally with `client?.end()`. The existing catch already maps vmKilled/vmPaused === false -> recoveryStatus 'running', so a connect failure would self-heal the row back to `running` and let pg-boss retry without ever stranding it in `stopping`. Additionally add an entry-level `if (!(await isServerReachable(serverId))) { throw ... }` BEFORE the atomic claim (Rule 58 preflight) so a known-down host short-circuits before any side effect and pg-boss retries with nothing stranded. Apply both changes to cube-sleep.ts as well.

### H4. Both REST delete routes (dashboard DELETE + v1 API) skip the transferState and snapshot-creating gates the server action enforces — a customer API call can rm -rf a cube mid-transfer or mid-snapshot  _[0.9]_
- **Files:** `lib/cube-actions/cube.ts`, `app/api/spaces/[spaceId]/cubes/[cubeId]/route.ts`, `app/api/v1/spaces/[spaceId]/cubes/[cubeId]/route.ts`, `app/actions/cubes.ts`, `lib/worker/handlers/cube-delete.ts`, `lib/worker/handlers/cube-transfer.ts`, `lib/ssh/firecracker.ts`
- **Impact:** A customer can corrupt a cube's rootfs and torpedo an in-flight cross-server transfer through the documented public v1 API (or the dashboard REST DELETE). Deleting while cube.transfer is mid cp --reflink / rsync of /var/lib/krova/cubes/<id>/rootfs.ext4 produces a torn transfer and a half-provisioned destination cube; deleting while snapshot.create is mid restic backup rm -rf's the cube dir out from under the running backup. Both are data-loss/corruption paths reachable by a customer with a valid API key.
- **Fix:** Move both guards into the shared deleteCubeAction (lib/cube-actions/cube.ts) so every entry point inherits them: (a) add eq(schema.cubes.transferState,'idle') to the atomic-claim where(...) and, when the row exists but transferState!='idle', return {ok:false,status:409,error:'This Cube is being transferred between servers...'} (mirror app/actions/cubes.ts:632-637); (b) before the claim, SELECT cubeSnapshots WHERE cubeId=? AND status='creating' LIMIT 1 and return 409 if present (mirror app/actions/cubes.ts:547-562). As defense-in-depth, add eq(cubes.transferState,'idle') to the cube-delete worker handler's step-1c claim predicate (cube-delete.ts:80-84) so a stale/racing enqueue cannot proceed even if a future caller forgets the gate.

### H5. Transfer step 5 pauses the source VM while transferState is still 'idle' — cube.state-sync races in and flips a running cube to 'sleeping' mid-transfer  _[0.88]_
- **Files:** `lib/worker/handlers/cube-transfer.ts`, `lib/worker/handlers/cube-state-sync.ts`, `app/api/orbit/cubes/[cubeId]/transfer/route.ts`
- **Impact:** On every transfer of a RUNNING cube whose rootfs copy takes long enough to overlap a state-sync tick, the customer sees a spurious 'Cube unexpectedly powered off / paused — marked as sleeping' lifecycle log, a cube.sleeping outbound webhook, an EmailIt contact-sync, and a Pusher cube event — a false 'powered off' alarm and dashboard flicker for a cube that is actually mid-migration and will end running. Steady-state self-heals (the atomic flip at line 995 resets status='running', lastBilledAt, and reconciles both servers' CPU/RAM counters). The lasting-harm tail: if the worker is SIGKILLed during step 5's long cp AND a state-sync tick flipped status→sleeping in the gap, the pg-boss retry re-reads the row (line 92) with status now 'sleeping', so wasRunning=false, takes the already-sleeping branch, and the atomic flip leaves the cube SLEEPING on the destination — silent loss of the customer's running state and compute billing.
- **Fix:** Move the `transferState='snapshotting'` write (cube-transfer.ts:317-320) to BEFORE the source pause in step 5 — set it right after step 4's claim, ahead of the connectToServer(sourceServerId) block at line 247 — so the source is paused only while inside a state-sync-excluded transfer state. The retry-skip guard transferStateAtOrPast('restoring') at line 312 is unaffected (snapshotting < restoring in TRANSFER_STATE_ORDER, so a crash-retry still re-runs the rsync). Verify the crash-retry path: with transferState already 'snapshotting' on retry, state-sync would have skipped the cube, so wasRunning stays correct.

### H6. No reaper for domain_mappings stranded in `stopping` (and `pending`) — a host-down domain.remove permanently bricks the domain for re-add  _[0.86]_
- **Files:** `lib/cube-actions/domains.ts`, `lib/worker/handlers/domain-remove.ts`, `lib/worker/ensure-queues.ts`, `lib/worker/boss.ts`
- **Impact:** A customer who clicks Remove Domain while the cube's host is briefly unreachable (the common mango-style outage) ends with the domain mapping permanently stuck in `stopping`. They can never remove it again (409 at domains.ts:240) and never re-add the same domain anywhere (409 at domains.ts:113). The Caddy route and Cloudflare Custom Hostname may also still be live, so the domain keeps serving from a cube the customer believes is detached. Only an operator DB edit unblocks it. This is the single customer-facing intermediate-state table with no stale-row reaper.
- **Fix:** Add a `domain.stale-check` cron (mirroring snapshot.stale-check) that finds rows in `stopping` with updatedAt older than ~15 min and re-enqueues DOMAIN_REMOVE — or, if the cube is deleted/gone, hard-deletes the row so the customer can re-add. Also sweep `pending` rows older than ~30 min with no in-flight DOMAIN_ADD. Register the queue in QUEUE_OPTIONS (Rule 56) with policy:'exclusive' (recurring-queue rule). Alternatively, give domain-remove.ts a final-attempt branch (mirroring domain-add.ts's retryCount>=retryLimit branch at lines 169-197) that hard-deletes the row when retries are exhausted. Either fix should also guard the connect-inside-try with self-cleanup (Rule 58 guarded-connect invariant) so a host-down teardown returns the row to a re-addable terminal state instead of stranding it.

### H7. Clone-from-snapshot leaves a live Firecracker VM orphaned on the host when a post-boot step fails (catch never kills the VM)  _[0.85]_
- **Files:** `lib/worker/handlers/cube-from-snapshot.ts`, `lib/ssh/firecracker.ts`, `lib/server/allocate.ts`, `db/schema/tcp-mappings.ts`
- **Impact:** A clone that fails AFTER startCube returns (transient DB blip flipping the cube to running at lines 414-427, a transient SSH/iptables error in addTcpPortForward at 369-375, a transient DB error or unique-violation on the tcp_port_mappings insert at 376-384, or a missing portRecord at 365-367) leaves a Firecracker microVM running on the bare-metal host while failClone flips the cube to 'error' and runs reconcileServerResources — which excludes 'error' cubes from the CPU/RAM/disk counters. The allocator then forgets this cube's reservation while the VM keeps consuming real host CPU/RAM/disk, silently overcommitting the host (violating the Rule 53 1:1 no-oversell disk contract) and risking starvation/destabilization of neighbor cubes. server.reconcile only emails an admin (never auto-destroys) and cube.error-recovery skips it (lastStartedAt is null), so the zombie VM persists until a human runs pnpm cube:inspect --destroy.
- **Fix:** Mirror cube-import-rootfs.ts: import `deleteCube as destroyCubeVm` from `@/lib/ssh` and, in the clone catch (cube-from-snapshot.ts:453), call `await destroyCubeVm(client, cubeId, internalIp ?? undefined, launchMode).catch(() => {})` BEFORE failClone, guarded so it only runs when the VM may be live (e.g. after a `vmStarted` boolean set right after line 351). destroyCubeVm kills FC + deletes the TAP + removes the cube dir, subsuming the existing removeTcpPortForward cleanup. This makes the failure path leave no orphaned VM, matching the import handler.

### H8. Guest agent's PTY pump calls conn.sendall() on a NON-BLOCKING vsock socket — heavy stdout (cat large file, build logs) tears down the customer's terminal mid-session  _[0.85]_
- **Files:** `setup/images/krova-agent`
- **Impact:** Any command producing a fast multi-MB burst (cat of a large file, yes, dmesg, verbose build output) under even mild SSH/Soketi backpressure intermittently kills the customer's interactive shell mid-run, losing their in-progress work. Load-dependent and hard to attribute, surfaces as an unstable/flaky terminal.
- **Fix:** Do not put `conn` into the EAGAIN-fatal write path. Either keep `conn` blocking for writes (the read loop already select()s on conn, so a blocking write between selects is safe) by writing the STDOUT frame via a small blocking-write helper, OR keep non-blocking but add `conn` to the select write fd-set with a pending-output bytearray, treating BlockingIOError as 'retry next tick' instead of `break`. The minimal change is to make _send_frame loop on EAGAIN (select-for-write) rather than letting it surface as the fatal `except OSError: break`.

### H9. Live RAM/disk grow failure flips a still-running VM to `error` and clears lastBilledAt — healthy cube shown as error, billing silently stops, error-recovery can't relaunch  _[0.8]_
- **Files:** `lib/worker/handlers/cube-resize.ts`, `lib/ssh/firecracker.ts`, `lib/worker/handlers/cube-state-sync.ts`, `lib/worker/handlers/cube-error-recovery-scan.ts`, `lib/worker/handlers/cube-error-recovery.ts`
- **Impact:** A transient live RAM-grow timeout (guest under memory pressure / brief agent unresponsiveness) or a live disk resize2fs hiccup marks a fully-healthy, still-serving cube as 'error' and zeroes lastBilledAt. The customer sees their working cube as 'error', the platform stops billing a running cube (revenue loss), and automatic error-recovery repeatedly fails with TAP EBUSY (it never kills the live process) until an operator intervenes.
- **Fix:** In the catch block, when effectiveLive is true (VM never stopped), do NOT flip to error. Restore the pre-resize status (running) and KEEP lastBilledAt intact for a was-running cube — mirroring snapshot-restore's guarded-failure path that preserves lastBilledAt because the VM was never stopped (Rule 51). Persist the grown diskLimitGb if diskHostFileGrown, but leave status=running and billing running. Only the cold path (VM actually powered off) should land in error. The handler docstring lines 49-54 enshrines the buggy 'live RAM PATCH / live disk PATCH → error' behavior and should be corrected alongside.

### H10. Host helper krova-vsock-pty calls sock.sendall() on a NON-BLOCKING vsock socket — a large stdin paste tears the session down  _[0.72]_
- **Files:** `setup/server/krova-vsock-pty`
- **Impact:** Pasting a moderately large block of text/script into the browser terminal can intermittently kill the session (`session ended (ssh_stream_closed)`), losing in-progress work — same failure as the agent-side bug, on the host→guest path. Lower frequency because interactive stdin is low-volume and the guest must be momentarily slow to drain.

Separately, harden the stdout write — if stdout_fd is ever made non-blocking, handle os.write's partial-write return value to avoid the same EAGAIN/partial-write class on that path.

### M4. SSH-port change strands the SSH mapping row in 'pending' forever if the host is unreachable for >~90s, permanently locking the customer out of changing their SSH port (connect is outside the guarded try; no reaper)  _[0.85]_
