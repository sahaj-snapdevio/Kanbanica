# Krova v1 API Reference

Base URL: `https://<your-domain>/api/v1`

A machine-readable OpenAPI 3.1 spec is available at `GET /api/v1/openapi.json`.

Cubes are hardware-isolated Firecracker microVMs — each with its own kernel, a per-cube sandbox, and no public IP of its own (you reach a Cube through the port mappings you create). Rates are quoted per hour but billed by the minute.

## Authentication

All v1 endpoints (except `/images`, `/pricing`, and `/regions`) require an `X-API-KEY` header containing a valid API key.

API keys are created in the **Space Settings → API Keys** section of the dashboard. Each key is:
- **Scoped to a single Space** — a key for Space A cannot access Space B.
- **Bound to the membership of the user who created it.** Permissions on the key follow the role/permissions of that membership at the time of every request — revoking a permission from the underlying membership takes effect immediately.

Keys are 32 random bytes encoded as base64url with the prefix `kro_`. Example:

```
X-API-KEY: kro_aBcD12345678abcdef...
```

The full key is shown to you exactly once at creation time and stored as a SHA-256 hash on the server. If you lose it, you must rotate.

## Scopes / Permissions

Permissions are set at the **membership** level (not the key level). When creating an API key, the key inherits the permissions of the creating user's membership. The seven permission values are:

| Permission         | Grants                                                                                |
|--------------------|---------------------------------------------------------------------------------------|
| `cube.view`        | List and get cubes, domains, TCP mappings, snapshots, imports                         |
| `cube.create`      | Create cubes; initiate / complete / cancel cube imports                               |
| `cube.manage`      | Sleep, wake, delete cubes; change SSH port; add/remove domains, TCP mappings; create/delete snapshots; restore snapshots; download backups |
| `billing.view`     | View billing history and credit balance (no v1 endpoints today; reserved for future)  |
| `billing.manage`   | Manage subscriptions and credit purchases (no v1 endpoints today; reserved for future)|
| `members.invite`   | Send space invites (dashboard only; no v1 endpoints today)                            |
| `members.manage`   | Manage space members (dashboard only; no v1 endpoints today)                          |
| `webhook.manage`   | Create, update, delete, and inspect outbound webhook endpoints + delivery history     |

**Owner memberships bypass all permission checks** — they can call every endpoint.

Non-owner API keys must have the relevant permission granted by a space owner through the dashboard.

## Error Envelope

All error responses use this shape:

```json
{ "error": "Human-readable error message" }
```

A small number of errors include extra fields alongside `error` (for example, the cube-create insufficient-credits error includes `required` and `available`). Treat any fields beyond `error` as informational; relying on them is fine for human-readable display but the `error` string is the only stable contract.

HTTP status codes returned by v1:

| Status | Meaning                                                                  |
|--------|--------------------------------------------------------------------------|
| `400`  | Validation error (bad field value, missing required field)               |
| `401`  | Missing or invalid API key                                               |
| `403`  | Forbidden — banned account, missing permission, or plan-limit refusal    |
| `404`  | Resource not found                                                       |
| `409`  | Conflict (duplicate, operation already in progress, wrong state)         |
| `422`  | Semantic error (cube in wrong state for this operation; insufficient credits) |
| `429`  | Rate limited (see `Retry-After` header)                                  |
| `500`  | Internal server error (safe to retry)                                    |
| `503`  | Capacity error (no host ports available, no storage backend); retry with backoff |

## Rate Limits

Mutating endpoints (`POST` / `DELETE` / `PUT`) are rate-limited to **10 requests per 60 seconds per client IP**. Read endpoints (`GET`) are not currently rate-limited.

When you exceed the limit, the API returns `429 Too Many Requests` with a `Retry-After` header containing the number of seconds until the window resets:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 23
Content-Type: application/json

{ "error": "Too many requests. Please try again later." }
```

The limiter is an in-memory fixed-window counter keyed on the first IP in `X-Forwarded-For` (falling back to `X-Real-IP`). For background polling, use intervals of at least 5 seconds.

## Idempotency

Mutating `POST` endpoints support an optional `Idempotency-Key` header. If you send the same key on a retry, the original response is returned unchanged without re-executing the operation.

```
Idempotency-Key: <any unique string, max 255 chars>
```

- **Scope**: per Space — the same key sent to a different space is treated as a new request.
- **TTL**: 24 hours after first use.
- **Cached responses**: any non-5xx response (including 4xx errors) is cached and replayed.
- **Replayed responses** include the header `Idempotency-Replayed: true`.
- **Key length** is silently capped at 255 characters; longer keys are treated as if no key was sent.

Supported on:

- `POST /spaces/{id}/cubes`
- `POST /spaces/{id}/cubes/{cubeId}/domains`
- `POST /spaces/{id}/cubes/{cubeId}/tcp-mappings`
- `POST /spaces/{id}/cubes/{cubeId}/snapshots`

Other `POST` / `PUT` / `DELETE` endpoints ignore the header.

## Pagination

List endpoints accept `?page=<n>&limit=<n>` query parameters. Defaults: `page=1`, `limit=10`. Maximum `limit` is `100`.

Responses include a `pagination` object alongside the data:

```json
{
  "cubes": [ ... ],
  "pagination": {
    "page": 1,
    "limit": 10,
    "total": 47,
    "totalPages": 5
  }
}
```

Currently, pagination is implemented on `GET /spaces/{id}/cubes`. Other list endpoints return the full list (use the resource itself to bound size — domains/TCP mappings/snapshots are capped by your plan).

## Billing Semantics

### Credit model

Cubes are billed in credits (1 credit = $1 USD). Credits are consumed hourly while a cube is `running`. The hourly cost is:

```
cost = (vcpus × vcpuRate + ramGb × ramRate + diskGb × diskRate) × tierMultiplier
```

Where `ramGb = ramMb / 1024`. Every allocated GB of RAM and disk is billed — Krova sells 1:1 with host resources and does not oversell. Current rates and tier multipliers are available at `GET /api/v1/pricing`.

### Sleep vs. stopped

`sleeping` means the Firecracker process is paused (memory frozen, disk preserved). **vCPU + RAM charges stop immediately on sleep.** The disk component continues — billed at `diskRate × diskLimitGb × tierMultiplier` per hour, the same per-GB rate and full-disk basis as the running-disk component (running and sleeping cubes both occupy every allocated GB on the host).

When a cube goes to sleep (via `POST /cubes/{id}/sleep`, zero-balance auto-sleep, or any other customer-initiated stop), a prorated charge covers the partial hour of running compute (vCPU + RAM + disk) since the last full-hour billing tick. After that prorated charge, hourly sleep-storage charges start on the next tick for the disk component only.

### Unexpected shutdowns

If a cube transitions to `error` or `sleeping` due to an event not initiated by the customer (kernel panic, OOM, host reboot, `shutdown -h` from inside the guest), **no prorated charge is applied** for the partial hour leading up to the event. Full-hour charges that already ran stand because they covered actual running time.

### Postpaid overage (optional)

Spaces on plans that allow overage can opt in to keep cubes running after the prepaid balance is exhausted, up to a customer-set cap per billing period. Configure from the dashboard. There are no v1 endpoints for overage settings today.

## Cube Request & Response Shapes

### Create cube — request body

```json
{
  "name": "my-cube",
  "image": "ubuntu-24.04",
  "sshPublicKey": "ssh-ed25519 AAAA...",
  "resources": {
    "vcpu": 2,
    "ramGb": 4,
    "diskGb": 40
  },
  "region": "us-east",
  "userData": "#cloud-config\npackages:\n  - nginx\n"
}
```

Field reference:

| Field              | Type    | Required | Notes                                                                                                  |
|--------------------|---------|----------|--------------------------------------------------------------------------------------------------------|
| `name`             | string  | yes      | 1–64 printable characters                                                                              |
| `image`            | string  | yes      | Image id from `/v1/images`                                                                             |
| `sshPublicKey`     | string  | yes      | See SSH key rules below                                                                                |
| `resources.vcpu`   | integer | yes      | Must be a supported value (`/v1/pricing` lists tiers; the platform exposes the allowed range)          |
| `resources.ramGb`  | number  | yes      | RAM in **GB**; converted to MB server-side (`ramMb = ramGb × 1024`)                                    |
| `resources.diskGb` | integer | yes      | Disk in **GB**                                                                                         |
| `region`           | string  | no       | Region slug from `/v1/regions`. If omitted, the platform picks a region with available capacity        |
| `userData`         | string  | no       | cloud-init script, max 16 384 bytes                                                                    |

Only the keys above are accepted. `hostname`, `vcpus`, `ramMb`, `diskGb` at the top level, `ram_gb`, `disk_gb`, `imageId`, `user_data` etc. are not recognized — the request body is camelCase only and resources are nested.

`sshPublicKey` must start with `ssh-ed25519`, `ssh-rsa`, `ecdsa-sha2-*`, `ssh-dss`, or `sk-*@openssh.com`. Krova does not persist a per-user SSH key store; the key you pass is written directly into the cube's `/root/.ssh/authorized_keys` at boot.

#### `userData` / cloud-init

If supplied, the string is written into the guest rootfs as a [cloud-init NoCloud](https://cloudinit.readthedocs.io/en/latest/reference/datasources/nocloud.html) seed before the VM boots. Both supported images (Ubuntu 24.04 and Ubuntu 24.04 + Docker) ship `cloud-init` pre-installed. The script runs once on first boot.

Maximum size: **16 384 bytes**. Common use cases: installing packages, writing files, creating users, running a startup script.

## Outbound Webhooks

Krova POSTs signed HTTP callbacks to your endpoints whenever entities in your space change. Each endpoint is per-Space and can subscribe to any combination of events. Both the API and the customer dashboard at `/{spaceId}/webhooks` manage endpoints; the `webhook.manage` member permission is required (space owners always have it).

### Creating an endpoint

```http
POST /api/v1/spaces/{spaceId}/webhooks
Content-Type: application/json
X-API-KEY: kro_...

{
  "url": "https://example.com/hooks/krova",
  "description": "Production receiver",
  "events": ["cube.running", "cube.deleted", "subscription.canceled"]
}
```

Response (`201 Created`):

```json
{
  "webhook": {
    "id": "<cuid2>",
    "url": "https://example.com/hooks/krova",
    "description": "Production receiver",
    "events": ["cube.running", "cube.deleted", "subscription.canceled"],
    "enabled": true,
    "createdAt": "2026-05-27T12:00:00.000Z",
    "updatedAt": "2026-05-27T12:00:00.000Z",
    "secret": "a3f2...deadbeef"
  }
}
```

**The `secret` field is returned only once at creation — store it securely.** Use the dashboard's "Rotate signing secret" action to mint a new one (the previous secret stops verifying future deliveries immediately).

URLs are rejected at create time AND at delivery time if they resolve to non-routable address space (loopback, link-local including the 169.254.169.254 cloud metadata endpoint, RFC 1918 private space, carrier-grade NAT, IPv6 ULA / link-local). `http://` is accepted for local development; production endpoints should always be `https://`.

### Verifying signatures (Stripe-style, v1)

Every delivery includes an `X-Krova-Signature` header in the form:

```text
X-Krova-Signature: t=<unix-seconds>,v1=<sha256-hex>
```

The HMAC-SHA256 input is `{t}.{rawBody}` (the timestamp, a literal `.`, then the raw request body bytes) using the endpoint secret (the hex string Krova returned on create / rotate).

Reject any delivery whose `t` is more than **300 seconds** away from your server's current time — this is the replay-protection window. A captured POST replayed outside the tolerance fails verification by construction.

Example (Node.js):

```js
import { createHmac, timingSafeEqual } from "crypto";

function verify(secret, rawBody, signatureHeader) {
  const parts = Object.fromEntries(
    signatureHeader.split(",").map((kv) => kv.split("="))
  );
  const ts = Number(parts.t);
  if (!ts || Math.abs(Date.now() / 1000 - ts) > 300) {
    return false; // outside 5-minute replay window
  }
  const expected = createHmac("sha256", secret)
    .update(`${ts}.${rawBody}`)
    .digest("hex");
  return timingSafeEqual(Buffer.from(parts.v1), Buffer.from(expected));
}
```

Always use a constant-time comparison (`timingSafeEqual`) — a plain `===` is vulnerable to timing attacks.

### Delivery, retries, and flap protection

- A delivery is considered successful when your endpoint returns a `2xx` status within **10 seconds**.
- Krova retries failed deliveries **up to 4 times** with a **60-second delay** between attempts (5 total attempts before giving up).
- **Flap protection:** after **50 consecutive failed deliveries**, the endpoint is auto-disabled. The space owner gets an email; you re-enable from the dashboard or via the v1 API after fixing the receiver. Re-enable resets the counter.
- Delivery history is retained for **30 days** and pruned by a daily background job.

Every outbound request includes these headers in addition to `X-Krova-Signature`:

| Header | Value |
| ------ | ----- |
| `X-Krova-Event`    | The event name (e.g. `cube.running`) |
| `X-Krova-Delivery` | Unique delivery ID — useful for de-duplication if your endpoint receives the same event twice during retries |
