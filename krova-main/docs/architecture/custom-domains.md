### Custom Domains (Cloudflare for SaaS)

Customer custom domains route through **Cloudflare for SaaS (Custom Hostnames)**. The customer points ONE CNAME at the fixed `dns.krova.cloud`; Krova registers the domain as a Cloudflare Custom Hostname whose `custom_origin_server` is the Cube's server's derived origin hostname (`serverOriginHostname(server.hostname)`, e.g. `banana.krova.cloud`). Cloudflare manages the visitor-facing TLS certificate and DDoS protection; the origin leg uses a wildcard Cloudflare Origin CA cert installed on Caddy (`installOriginCaCert`). If the customer's domain DNS is itself on Cloudflare, that CNAME must be **DNS only (grey cloud)**, not Proxied.

Each server has two derived DNS records, both A-records pointing at `servers.publicIp`, created together by `setUpServerCloudflareOrigin()` during the (now **mandatory**) Cloudflare step of the `install` setup phase: the **proxied** `<hostname>.krova.cloud` origin and the **DNS-only** `connect.<hostname>.krova.cloud` SSH/landing record. The proxy flags are non-negotiable — Cloudflare for SaaS requires the custom origin to be orange-cloud, and SSH (port 2822 / TCP mappings) plus the landing-page ACME HTTP-01 challenge require the connect record to be grey-cloud. If the Cloudflare env is not configured, the `install` phase now fails rather than skipping.

The `domain.add` / `domain.remove` worker jobs register / delete the Custom Hostname (plus the Caddy host route). `cube.transfer` re-points the Custom Hostname's origin to the destination server; `cube.delete` deletes the Cube's Custom Hostnames. A `cloudflare.hostname-poll` cron (every 1 min) refreshes `domain_mappings.cloudflareStatus` and emits a Pusher `domain.update` event. Caddy is still the per-server reverse proxy — it host-routes the customer domain to the Cube's internal IP — but it no longer issues per-customer-domain ACME certificates (on-demand TLS was removed). The branded per-server landing page is served on **both** derived hostnames (`serverOriginHostname()` + `serverConnectDomain()`): the grey-cloud connect host uses direct ACME, while the proxied origin host is served the wildcard Origin CA cert (excluded from ACME via `automatic_https.skip`).

**Per-domain edge-cache purge (Clear cache).** Because a customer's own DNS record is grey-cloud / DNS-only (caches nothing), ALL edge caching for a custom domain lives on Krova's Cloudflare zone — so only Krova can purge it. Each active custom domain exposes a **Clear cache** action (customer cube-detail → Networking, the Orbit `/orbit/domains` table, and the v1 API `POST /api/v1/spaces/[spaceId]/cubes/[cubeId]/domains/[mappingId]/purge-cache`) that enqueues the `domain.purge-cache` job. The handler does a Cloudflare **purge-by-hostname** (`POST /zones/{zone}/purge_cache` body `{ hosts: [domain] }` — verified all-plans since Apr 2025; NEVER `purge_everything`, which would drop every customer's cache on the zone). The shared `purgeDomainCacheAction` / `adminPurgeDomainCacheAction` in [lib/cube-actions/domains.ts](lib/cube-actions/domains.ts) atomically claim the per-domain cooldown (conditional UPDATE on `lastCachePurgeAt`, returns 429 + a `Retry-After` header + `retryAfterSeconds` within the window) before enqueue; the worker auto-retries a CF 429 (`retryLimit:5`/`retryDelay:60`, `policy:"exclusive"` + `singletonKey:mappingId`) and fires a `domain.cache-purged` Pusher event (`success`/`failed`) so the UI can toast the outcome. **Cloudflare's purge-by-hostname rate limit is ZONE-WIDE and plan-tiered** (Free 5/min + 25-token burst bucket; Pro 5/s; Business 10/s; Enterprise 50/s) — the per-domain cooldown bounds per-domain spam but NOT the fleet aggregate, so the 60s retry spacing (a full refill window) is what makes a rate-limited purge eventually land on the Free plan instead of false-failing. The CF token needs the `Zone → Cache Purge → Purge` permission. **Wildcard / catch-all custom hostnames cannot be edge-purged** — Cloudflare purge-by-hostname rejects `*`, so `buildPurgeByHostnameBody` + the action both refuse a `*` domain with a clear error (`validateDomain` already rejects wildcards at add-time; this is defense-in-depth). **This feature is coupled to Cloudflare-for-SaaS and is slated to be retired with it** if custom domains move to the self-hosted Caddy ingress (see [docs/superpowers/specs/2026-06-02-custom-domain-self-hosted-ingress-design.md](docs/superpowers/specs/2026-06-02-custom-domain-self-hosted-ingress-design.md) §8) — under that design there is no Krova-controlled edge cache to purge. The CF helper is `purgeCacheByHostname` in [lib/cloudflare/cache.ts](lib/cloudflare/cache.ts); pure cooldown math in [lib/domains/cache-purge.ts](lib/domains/cache-purge.ts).

**Origin-fallback Worker — REMOVED (2026-06-02).** A Cloudflare Worker on a catch-all `*/*` route once branded the "origin unavailable" page for customer custom domains when a whole host/Caddy was down. It was removed: a per-request edge Worker's cost scales with TOTAL request volume, and on the Workers Free plan the 100k-requests/day cap produced an Error-1027 fleet-wide blackout (the in-code fail-safe never runs once the quota is exhausted — Cloudflare stops executing the Worker entirely). **Do NOT reintroduce a per-request edge Worker for this.** Accepted trade-off: Cloudflare's default grey 52x page shows for the rare whole-host-down case; Caddy's own branded "Starting up" 503 (`handleErrorsBlock` in [lib/ssh/caddy.ts](lib/ssh/caddy.ts)) still covers the common cube sleeping/booting/crashed cases. If host-down branding ever justifies a paid plan, the non-scaling options are a Cloudflare **Snippet** (Pro+, flat cost, no per-request billing) or a Load-Balancing fallback pool — NOT a Workers-Paid Worker (still billed per request). The `workers/origin-fallback/` folder, the `pnpm cloudflare:worker:*` scripts, and the `X-Krova-Edge` Caddy marker were deleted; the marker removal reaches the fleet on the next `pnpm routing:refresh-fleet`.
