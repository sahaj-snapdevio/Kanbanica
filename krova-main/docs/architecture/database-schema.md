# Database Schema Notes (db/schema/)

> On-demand detail extracted from CLAUDE.md. CLAUDE.md keeps only a short summary + a pointer to this file; the full reference lives here.

### Database (db/schema/)

- **CUID2** text primary keys on user-facing tables
- **pgEnum** for status/type fields
- **`numeric(12,4)`** for monetary values (never float)
- Better Auth tables use singular names (`user`, `session`, `account`, `verification`); all others plural
- Schema changes go through `pnpm db:generate` (creates SQL file under `db/migrations/`) ‚Üí commit ‚Üí `pnpm db:migrate` (after deploy)
- **`cubes.has_virtio_mem`** ‚Äî boolean, default false; set true when the cube has a `virtio-mem` device declared (any fresh boot from the live-resize Firecracker config). Drives whether RAM grow can stay live or must fall back to a cold restart.
- **`cubes.launch_mode`** (`cube_launch_mode` enum `bare|jailed`, default `bare`) + **`cubes.jailer_uid`** (int, nullable; migrations 0066/0067) ‚Äî how the cube's Firecracker is launched on the host. `bare` = legacy `nohup firecracker` as root (paths under `/var/lib/krova/cubes/<id>`); `jailed` = run under the Firecracker **jailer** (per-cube unprivileged uid = `jailer_uid`, chroot at `/var/lib/krova/jail/firecracker/<id>/root`, new PID namespace ‚Äî **the isolation boundary**). `--cgroup-version 2` is set so the jailer never falls back to v1 on our cgroup-v2-only hosts. **Per-cube `cpu.weight` fairness (audit C2 / L1) is wired but GATED behind `CPU_CGROUP_ENABLED` (default `false`):** with the flag OFF, NO `--cgroup` is passed ‚Äî byte-identical to legacy, no confinement (the VMM + Krova's allocator are the resource caps); with it ON, `buildJailerArgs` emits `--parent-cgroup krova --cgroup cpu.weight=<cubeCpuWeight(vcpus) = vcpus√ó100>` so each cube lands in a `/sys/fs/cgroup/krova/<cubeId>` leaf weighted by its vCPUs (work-conserving ‚Äî overselling preserved, NO `cpu.max`, so idle cubes' cycles still redistribute). Host prep = the boot-persistent `krova-cgroup-prep` oneshot (`install` phase / `pnpm install:cpu-cgroup`); `launchJailed` **preflights** the parent (`cpuCgroupReadyCommand`) and omits `--cgroup` if it isn't ready, so a missing/half-prepped parent can NEVER brick a boot (the cube launches without the weight). `teardownJail` `rmdir`s the leaf to avoid a `/sys/fs/cgroup` leak. **Operator invariant: never pre-create `/sys/fs/cgroup/firecracker` with domain controllers in its `subtree_control`, or every jailed launch on that host fails** (the cgroup-v2 no-internal-process rule) ‚Äî the L1 path deliberately uses a DEDICATED `krova` parent (NOT `firecracker`) to sidestep this. Confirmed live on a dev-host canary 2026-06-03 (leaf at `krova/<id>`, `cpu.weight` applies, cube boots + networks). Design + plan + rollout: [docs/superpowers/specs/2026-06-03-oversold-cpu-fairness-numa-design.md](docs/superpowers/specs/2026-06-03-oversold-cpu-fairness-numa-design.md) + [docs/superpowers/plans/2026-06-03-cpu-weight-fairness-l1.md](docs/superpowers/plans/2026-06-03-cpu-weight-fairness-l1.md). **NUMA-aware placement (L2) is also wired, GATED behind `NUMA_PLACEMENT_ENABLED` (default `false`):** on a multi-socket host `launchJailed` ADDITIONALLY emits `--cgroup cpuset.cpus=<assigned node's cores ‚àí HOUSEKEEPING_CORES_PER_HOST> cpuset.mems=<node>`, binding each cube to ONE NUMA node (local L3 + memory, no cross-socket UPI hop) while preserving oversell (floats within the node, NO 1:1 pin). The node is chosen **least-loaded** at provision by `assignNumaNode` (`lib/server/numa-nodes.ts`, advisory-lock seed 4) + persisted on `cubes.numa_node`; per-node topology is detected at bootstrap into `servers.numa_topology`/`numa_node_count` (retrofit existing hosts: `pnpm install:numa-detect`; retrofit existing CUBES ‚Äî a node is assigned only at create, so pre-L2 cubes stay `numa_node IS NULL`/unpinned forever ‚Äî via `pnpm install:numa-backfill`, which assigns each a least-loaded node so it pins on its next cold-restart). The `krova` parent delegates `cpuset` UNCONDITIONALLY in the prep (inert when off ‚Äî a leaf with no cpuset inherits the full machine), so a host is L2-ready after `install:cpu-cgroup` regardless of the flag. Same fail-safe as L1: missing topology / un-delegated cpuset / single-socket host ‚Üí launch WITHOUT cpuset (cube still boots). The single launch decision is the unit-tested pure `shouldBindCpuset` (cpuset delegated + value-aware subset of the parent's LIVE `cpuset.{cpus,mems}.effective` per review H1 + **the OVERSELL guard: a cube with MORE vCPUs than its assigned node has usable cores launches UNPINNED across the whole host ‚Äî it is never throttled below its sold vCPUs while the other socket idles; a cube that fits the node still shares it work-conservingly via `cpu.weight` with NO `cpu.max`, so oversell is preserved WITHIN the node**; with the 16-vCPU sales cap this guard never fires on a normal socket ‚Äî defense-in-depth for small dual-socket hosts / future plan changes). `HOUSEKEEPING_CORES_PER_HOST` (2) cores are reserved for the host. Threaded into provision + the everyday relaunch paths (wake/cold-restart/auto-relaunch) via `cubeNumaLaunchOpts`; the rarer relaunch paths (snapshot-restore / import / from-snapshot / backup-redeploy / error-recovery / reboot-recovery / resize) are fail-safe follow-ups ‚Äî they boot unpinned + re-pin on the next everyday relaunch. A **cross-server transfer** additionally RE-ASSIGNS `cubes.numa_node` against the DESTINATION's topology (via `assignNumaNode` in an isolated tx after the residency flip ‚Äî best-effort, clears on failure) so the cube doesn't carry the source's stale node id; its destination boot is unpinned and it pins on the destination's next cold-restart. cgroup-v2 cpuset binding validated on the dev host (a process in a `krova/<id>` leaf with `cpuset.cpus=1` ‚Üí kernel `Cpus_allowed_list=1`); real multi-node binding is operator-canary'd on a dual-socket host (the dev host is single-socket). Rollout mirrors L1: deploy flag-off ‚Üí `pnpm install:numa-detect` + re-run `install:cpu-cgroup` ‚Üí dual-socket canary ‚Üí flip `NUMA_PLACEMENT_ENABLED`. Design + plan: [docs/superpowers/plans/2026-06-03-cpu-numa-placement-l2.md](docs/superpowers/plans/2026-06-03-cpu-numa-placement-l2.md). `jailer_uid` is allocated per-server (like ports, `lib/server/jailer-uids.ts`, base 100000, `UNIQUE(server_id, jailer_uid)`); the gid is the uid (unique per-cube group). A transfer takes BOTH the source and destination servers' uid-allocation advisory locks (sorted, deadlock-free) so a concurrent source-side relaunch can't steal the picked uid. The bare‚Üîjailed transition is decided at every (re)launch by `ensureLaunchMode` (`lib/cubes/launch-mode.ts`) under the `JAILER_ENABLED` policy. **The jailer is the security boundary that backs the "VM-grade isolation" claim ‚Äî a VMM/guest escape lands as an isolated per-cube uid in a chroot, not root on the host.** Gated behind `JAILER_ENABLED` in `config/platform.ts` (**enabled fleet-wide 2026-05-30** after canary validation on banana + mango); plan + rollout in [docs/superpowers/plans/2026-05-29-firecracker-jailer-hardening.md](docs/superpowers/plans/2026-05-29-firecracker-jailer-hardening.md), canary smoke in [docs/security/jailer-canary-smoke.md](docs/security/jailer-canary-smoke.md), host posture in [docs/security/host-hardening.md](docs/security/host-hardening.md). The pure path/arg resolver is `lib/ssh/jailer.ts` (`cubePaths(id, mode)` ‚Äî `bare` returns the exact legacy paths); the SSH-side jail ops (`launchJailed`/`teardownJail` ‚Äî rootfs+kernel **hardlinked** into the chroot, no bind-mount) live in `lib/ssh/firecracker.ts`. **Rule: every host-path resolution (socket/vsock/pid/fc.log) + every kill/status/lifecycle call MUST route through `cubePaths(id, mode)` / pass the cube's `launch_mode` ‚Äî never hardcode `/var/lib/krova/cubes/<id>/‚Ä¶` again.** A separate **`ENTROPY_DEVICE_ENABLED`** flag (**`true` since 2026-06-03** ‚Äî audit W5) attaches a Firecracker virtio-rng `/entropy` device pre-boot so the guest seeds its CRNG instantly (sshd host-key + TLS + WireGuard key/handshake gen). The `PUT /entropy` is **fail-safe** (wrapped exactly like `/cpu-config` ‚Äî a host on an older Firecracker logs a warning and boots without it, so the flag can NEVER brick a boot); it applies on a cube's NEXT cold boot only (a resume-from-paused doesn't re-run boot config). Needs the guest kernel's `CONFIG_HW_RANDOM_VIRTIO` (in `build-all-images.sh`; **confirmed bound as `virtio_rng.0` on the current 6.1.174 kernel** via a real cube boot) ‚Äî a cube on an older kernel ignores the device harmlessly until `build:images` + Update Images. Existing running cubes get it on their next cold boot (no retrofit; the entropy benefit is boot-time). See [docs/audits/2026-06-02-microvm-stability-audit.md](docs/audits/2026-06-02-microvm-stability-audit.md) W5.
- **`cubes.transfer_state`** ‚Äî enum `cube_transfer_state` (`idle|snapshotting|restoring|finalizing|completed|failed|cancelling`), default `idle`; tracks admin-initiated cross-server transfers. Paired with `cubes.transfer_destination_server_id` (target server) and `cubes.transfer_started_at` (UTC start timestamp) so the UI / reconciler can spot stuck transfers. `cancelling` is the in-flight cancel state (set by the cancel API before enqueuing `cube.transfer-cancel`); a cancelled transfer ends in `failed` (there is no distinct `cancelled` value). Cubes in `cancelling` are excluded from `cube.state-sync` / `cube.reachability` and have no reaper, so `cube.transfer-cancel` MUST retry to completion (it has an explicit `QUEUE_OPTIONS` entry with `retryLimit: 3`, per Rule 56). **The value list here is copied verbatim from the `cubeTransferState` pgEnum in [db/schema/cubes.ts](db/schema/cubes.ts).** When documenting any enum's values in this file, copy them from the pgEnum, never from memory ‚Äî the 2026-05-29 audit found this bullet listing three values (`preparing|transferring|complete`) that never existed in the schema.
- **`cubes.user_data`** ‚Äî text, nullable; the optional cloud-init `user_data` script passed at cube create via the API. Threaded `AllocateServerInput` ‚Üí cube row ‚Üí `CubeProvisionPayload` ‚Üí `CubeBootInput` ‚Üí `createCube()`, where it is written into the guest rootfs as a NoCloud seed. See "Guest hostname & cloud-init" below.
- **`servers.bridge_subnet`** ‚Äî integer, nullable; the per-server subnet `S` (globally unique across the fleet, partial `UNIQUE WHERE bridge_subnet IS NOT NULL`, range `[1, 511]`). Drives BOTH cube address families so every cube is globally unique: IPv4 `198.18.0.0/15` base+offset (the S-th `/24` inside the range ‚Äî `cubeIpv4Address(S, octet)` = `base + S√ó256 + octet`) and IPv6 `fd00:c0be:<S-hex>::<octet>/64`. New servers get a real `S ‚â• 1` at create via `allocateBridgeSubnet(tx)` in [lib/server/bridge-subnets.ts](lib/server/bridge-subnets.ts) (a single global advisory lock, disjoint seed `3` ‚Äî seeds 0/1/2 are taken by `acquireSpaceLock` / per-user / jailer-uid). The IPv4 range was rebased off `10.0.0.0/8` onto `198.18.0.0/15` (2026-05-31) to stop colliding with customer Docker-Swarm / k8s in-guest CIDRs; the old `S=0`-reproduces-legacy-`10.0.0.x` special-case is gone. See "Cube IPv6 + globally-unique networking" below.
- **`cubes.internal_ipv6`** ‚Äî text, nullable; the cube's ULA IPv6 address `fd00:c0be:<S-hex>::<octet>` (globally unique, partial `UNIQUE WHERE internal_ipv6 IS NOT NULL AND status <> 'deleted'`). Derived at write-time from the SAME freshly-picked host octet as `internal_ip` plus the server's `bridge_subnet` (pure math in `lib/server/cube-network.ts`). **Operator-only ‚Äî NOT exposed to customers** (dropped from `buildCubeSummary` / outbound webhooks / the v1 API alongside `internal_ip`); shown only on the Orbit cube-detail surfaces. The `internal_ip` is now an `198.18.0.0/15` address (`cubeIpv4Address(S, octet)` = `198.18.0.0 + S√ó256 + octet`, globally unique via `bridge_subnet`) ‚Äî rebased off the prior `10.x` schemes (the original per-host `10.0.0.<octet>` and the interim `10.<S>>8>.<S&0xff>.<octet>`) so it can't collide with a customer's in-guest Docker/k8s `10/8` CIDRs.
- **`servers.last_boot_id`** ‚Äî text, nullable; the host's `/proc/sys/kernel/random/boot_id` as last recorded by the worker. A change means the bare-metal host rebooted ‚Äî the trigger for `server.reboot-recovery`. Written by `server.reboot-recovery` (and seeded once by `cube.state-sync`).
- **`servers.overhead_disk_gb`** ‚Äî integer, default `0` (migration `0064`); the measured non-cube disk footprint on the host (OS, swap file, kernel/rootfs images, restic cache, /tmp staging, logs), refreshed hourly by the `server.measure-disk` cron as `df_used(/) ‚àí du(/var/lib/krova/cubes)` (both in actual-block GB, NOT apparent size). Paired with **`servers.disk_measured_at`** (timestamptz, nullable; the last measurement time). **Disk is sold 1:1 with no overselling (Rule 53), so cube placement reserves the full `diskLimitGb` per cube and caps total reservations at the EFFECTIVE capacity = `totalDiskGb ‚àí overheadDiskGb`** ‚Äî worst case (every cube fills its reservation) the host still can't overflow real disk. The single source of truth for this math is `lib/server/disk-capacity.ts` (`effectiveDiskCapacityGb` / `availableDiskGb` / `serverHasDiskRoom`); EVERY disk-capacity decision routes through it ‚Äî `lib/server/allocate.ts` (cube create/clone/import), the two Orbit transfer routes, the transfer-targets list, and the `cube.transfer` worker re-check ‚Äî never compare against raw `totalDiskGb` (Rule 14). Default `0` means a freshly-deployed or never-measured server behaves exactly like the historical `totalDiskGb` ceiling until the first cron tick lands. The **`server.measure-disk`** cron (hourly at :40, `policy: "exclusive"`, [server-measure-disk.ts](lib/worker/handlers/server-measure-disk.ts)) is a pure capacity observer ‚Äî it never transitions cube state or touches the `allocated*` counters; it stays quiet per-tick and audits only the meaningful signal (`server.disk_overcommitted`) when a host's existing reservations now exceed its freshly-measured effective capacity. Unreachable hosts are skipped (prior measurement retained) and retried next hour.
- **`domain_mappings.cloudflareHostnameId`** ‚Äî text, nullable; the Cloudflare Custom Hostname ID for a customer custom domain. Paired with **`domain_mappings.cloudflareStatus`** (text, nullable; mirror of the Cloudflare hostname status, refreshed by the `cloudflare.hostname-poll` cron) and **`domain_mappings.cloudflareCheckedAt`** (timestamptz, nullable; migration `0070` ‚Äî when the poll last read this hostname's live status). The poll re-checks NON-active rows every 1-min tick but already-`active` rows only every ~30 min (via `cloudflareCheckedAt`), so a regressed/expired cert flips its badge back instead of showing a stale "active" forever (Cloudflare CAN move a hostname out of `active` ‚Äî cert expiry, customer DNS change, re-validation ‚Äî so `active` is NOT terminal). The older `verificationStatus` / `verifyAttempts` / `verificationCheckedAt` / `verificationError` / `tlsStatus` columns are vestigial ‚Äî left in place but no longer driven after the Cloudflare-for-SaaS migration. Also **`domain_mappings.lastCachePurgeAt`** (timestamptz, nullable; migration `0071`) ‚Äî when an edge-cache purge was last REQUESTED for this hostname; stamped atomically at enqueue (NOT worker completion) and drives the per-domain purge cooldown (`DOMAIN_CACHE_PURGE_COOLDOWN_SECONDS` in `config/platform.ts`, default 60s) so a customer can't exhaust Cloudflare's zone-wide purge rate limit.
- **`space_domain_claims`** (migration `0072`) ‚Äî space-wide **domain locking**. A space proves ownership of a registrable domain (e.g. `acme.com`) via a TXT record (`_krova-verify.<domain>` = `krova-domain-verification=<token>`); once `verified` (status enum `pending|verified|failed`), that domain AND all its subdomains are locked to `space_id` ‚Äî no other space may map any hostname under it. Constraints: `UNIQUE(space_id, domain)` + a **partial `UNIQUE(domain) WHERE status='verified'`** (a domain is verified by at most one space ‚Äî the DB-level lock). Enforced in `addDomainAction` via `findCrossSpaceLock` ([lib/domains/claim-service.ts](lib/domains/claim-service.ts) ‚Äî an indexed `domain IN (candidateParentDomains(hostname))` lookup). Claiming is **optional + additive**: an unclaimed domain still maps first-come, so existing live domains are undisturbed. Pure coverage/normalization logic in [lib/domains/claim-coverage.ts](lib/domains/claim-coverage.ts) (`claimCovers` with the dot-boundary so `acme.com` does NOT cover `notacme.com`); fail-closed TXT check in [lib/domains/verify-txt.ts](lib/domains/verify-txt.ts). The `domain-claim.recheck` daily cron re-resolves verified claims and auto-releases (`verified‚Üífailed` + owner email) after `DOMAIN_CLAIM_MAX_FAILED_CHECKS` consecutive misses so a removed TXT can't hold a domain hostage. Surfaced as a **"Verified Domains" card in space settings** (gated by `cube.manage`); read-only list in Orbit ‚Üí Domains. **Architecture-independent** (ownership, not routing) ‚Äî survives the CF-for-SaaS ‚Üí self-hosted-ingress pivot. See [docs/superpowers/specs/2026-06-02-space-domain-claims-design.md](docs/superpowers/specs/2026-06-02-space-domain-claims-design.md).
- **`servers.hostname`** ‚Äî text; the single DNS label for a server. Every server-facing hostname is **derived** from it (never stored, never operator-edited, so they cannot drift): `lib/server/server-hostnames.ts` exports `serverOriginHostname()` ‚Üí proxied `<hostname>.krova.cloud` (the Cloudflare for SaaS `custom_origin_server`), `serverConnectDomain()` ‚Üí DNS-only `connect.<hostname>.krova.cloud` (customer SSH/TCP endpoint), and `serverLandingHosts()` ‚Üí the `{ originHostname, connectDomain }` pair. Caddy serves the branded Krova landing page on **both** derived hostnames ‚Äî a single host-matched `server-domain-landing` route covers them, so visiting either the proxied origin or the connect domain directly shows the branded "Krova server node" page rather than a bare 404. The connect host gets a real Caddy-managed ACME cert; the proxied origin host is served the wildcard Origin CA cert and is excluded from ACME via `srv0`'s `automatic_https.skip` (an HTTP-01 challenge there would be intercepted by Cloudflare's proxy). The base zone is `PLATFORM_BASE_DOMAIN` in `config/platform.ts`. `hostname` is immutable after creation and validated as a DNS label at create time. The former `servers.serverDomain` and `servers.cfOriginHostname` columns were removed (2026-05-17 hostname-derivation migration `0027`).
- **`email_events`** ‚Äî append-only EmailIt webhook telemetry (`email.delivered|bounced|complained|failed|‚Ä¶`). Idempotent on EmailIt's `evt_xxx` id; full payload kept as JSONB; pruned after 90 days by the `email.events-prune-cron` daily job. See `db/schema/email-events.ts`.
- **`user.marketing_opt_in`** ‚Äî boolean, default true; users are opted into marketing email at signup (covered by the ToS/privacy policy) and opt out from profile settings. Synced to EmailIt as the inverse of the contact's `unsubscribed` flag. Paired with **`user.emailit_contact_id`** (text, the `con_xxx` id once synced) and **`user.emailit_synced_at`** (timestamptz of the last successful contact sync).
- **`spaces.plan_id`** ‚Äî FK to `plans.id`; the space's current plan, read by `lib/plan/` for feature-limit enforcement (merged with the per-space `override_*` columns via `effectiveLimits`). Paired with **`spaces.low_balance_threshold`** (`numeric(12,4)`, default `5`) ‚Äî the per-space balance level at/under which the hourly worker sends the low-balance email; default + hard floor come from `platform_settings`.
- **`spaces` per-space override columns** ‚Äî eleven nullable columns (`override_max_concurrent_cubes`, `override_max_vcpus`, `override_max_ram_mb`, `override_max_disk_gb`, `override_max_seats`, `override_max_backups`, `override_max_domains`, `override_included_credit_usd`, `override_allow_topup`, `override_allow_overage`, `override_overage_cap_max_usd`). Null = use the plan's value; set = override wins. Operator-only via Orbit ‚Üí Space ‚Üí Plan overrides (`setSpaceOverride`); the runtime allowlist in `app/actions/orbit-space-overrides.ts` is the gate.
- **`spaces` overage columns** ‚Äî **`spaces.overage_enabled`** (boolean, default `false`; plans with `allowOverage=false` ‚Äî e.g. Trial ‚Äî are blocked from setting `true` by the server action), **`spaces.overage_cap_usd`** (`numeric(12,4)`, default `0`; customer-set hard cap per billing period, bounded by `platform_settings.overageCapMinUsd` / `overageCapMaxUsd` with a per-space `override_overage_cap_max_usd` that can raise the ceiling), **`spaces.this_period_overage_usd`** (`numeric(12,4)`, default `0`; running counter for the current billing period, reset to 0 by `subscription-handler.ts` only when `currentPeriodEnd` strictly advances ‚Äî a same-period `active ‚Üî past_due` flip does NOT reset).
- **`spaces` subscription columns** ‚Äî `payment_provider`, `provider_subscription_id`, `subscription_status`, `current_period_end`, `cancel_at_period_end` (all nullable / default false on Trial / canceled). Written by `lib/billing/subscription-handler.ts` from Polar webhook events ‚Äî the provider is authoritative for subscription state. `cancel_at_period_end` (migration `0054`) mirrors Polar's pending-cancel flag ‚Äî Polar keeps a canceling subscription in `status="active"` until the period actually ends, so the UI reads this column to render the "Ending on X" banner + the Resume button. The customer flips this column via `cancelPlan()` / `resumePlan()` server actions in `app/actions/subscriptions.ts` ‚Äî both call Polar's `subscriptions.update({cancelAtPeriodEnd: true/false})` and the round-trip webhook persists the column. The terminal `canceled` / `expired` path that drops the space to the default plan also clears the flag back to false. Paired with **`spaces.last_plan_credit_grant_at`** (timestamptz, nullable; the last plan-credit grant ‚Äî drives the `platform_settings.planCreditGrantCooldownDays` anti-abuse cooldown) and **`spaces.subscription_event_at`** (timestamptz, nullable; the `occurredAt` of the last applied subscription event ‚Äî a staleness guard so an out-of-order Polar redelivery cannot regress plan/period state). **`spaces.polar_customer_id`** (text, nullable; migration `0041`) is the canonical Polar customer id (e.g. `cus_‚Ä¶` / UUID) captured first-write-wins from the subscription / renewal webhook event (`sub.customer.id` / `order.customer.id`); preserved across cancel (a cancel does not delete the Polar customer record). Polar's customer `external_id` is PER-EMAIL and SHARED across sibling spaces of the same user (`external_id` is immutable + pinned to the FIRST space that subscribed), so `external_id` is NOT addressable for any non-first space ‚Äî every Polar API call that needs a per-space customer handle (`subscriptions.list`, `customers.update`, meter events) must resolve `polar_customer_id` instead. Backfill legacy rows with `tsx scripts/backfill-polar-customer-id.ts --apply`.
- **`subscription_credit_grants`** ‚Äî one row per plan-credit grant (subscription `activation` or `renewal`). `UNIQUE(provider_subscription_id, period_end)` makes each billing period grant-at-most-once ‚Äî idempotent against webhook redelivery. Stores `space_id`, `plan`, `period_start`/`period_end`, `amount`, and `reason`. See `db/schema/subscription-grants.ts`.
- **`subscription_intents`** ‚Äî durable record of a customer-initiated subscription checkout, inserted `pending` BEFORE the Polar checkout is created so the reconcile cron can heal a lost activation webhook. `subscription_intent_status` pgEnum is `pending|completed|failed|orphaned`; stores `space_id`, `plan`, `payment_provider`, `provider_checkout_id` (uniquely indexed). See `db/schema/subscription-intents.ts`.
- **`cubes.last_started_at`** ‚Äî timestamptz, nullable; when the Cube was last started. Plan over-limit reconcile (`lib/plan/reconcile.ts`) sleeps the most-recently-started Cubes first (`ORDER BY last_started_at DESC`) when a downgrade/cancel drops the space below its concurrent-Cube cap.
- **`cubes.error_recovery_attempts`** ‚Äî integer, default 0 (migration `0062`); count of automatic error-recovery attempts since the cube last ran. The `cube.error-recovery` cron tries to revive a cube parked in `error` up to `MAX_ERROR_RECOVERY_ATTEMPTS` (`config/platform.ts`, default 3) times, then gives up + notifies admins. Reset to 0 whenever the cube reaches `running` again (auto-recovery success OR a manual wake / `cube:inspect --restart`), so a later unrelated error episode gets a fresh budget. See "Automatic error recovery" below.
- **`credit_purchases`** ‚Äî durable record of a customer-initiated prepaid credit purchase (Polar). Inserted `pending` BEFORE the provider checkout is created; `credit_purchase_status` pgEnum is `pending|paid|partially_refunded|refunded|failed|orphaned`. Stores `amount` (base credit) + `surcharge_amount` separately, `provider_checkout_id` / `provider_order_id` (both uniquely indexed ‚Äî one row per checkout/order), and `refunded_amount`. The Polar webhook + `billing.topup-reconcile` cron are the writers; `applyPaidTopup()` does the idempotent flip. See `db/schema/credit-purchases.ts`.
- **`billing_events.type`** ‚Äî `billing_event_type` enum includes `plan_credit` (a plan's included credit granted on subscription activation/renewal ‚Äî kept distinct from the Orbit-admin `credit_grant` so plan credit is separately reportable), `credit_refund` (written by the Polar refund webhook handler when it claws back the base fraction of a refunded top-up from the space balance), `overage_charge` (one row per hourly-billing tick that consumed postpaid overage; written by `lib/billing/overage.ts` `applyOverageCascadeTx()`), and `sleep_storage_charge` (one row per cube per hourly-billing tick that the cube was `status='sleeping'` ‚Äî written by the sleep-storage pass in `billing-hourly.ts`; covers the on-host disk footprint of a paused/killed Firecracker while compute is unbilled). Paired with **`billing_events.polar_meter_reported_at`** (timestamptz, nullable; only meaningful on `overage_charge` rows ‚Äî set after the row's event has been ingested into Polar's meter, either by the inline post-commit reporter or the `polar.meter-reconcile` cron).
