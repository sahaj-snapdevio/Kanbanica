# Signup Validation, Email, Webhooks, Analytics & Deploy-Skew

> On-demand detail extracted from CLAUDE.md. CLAUDE.md keeps only a short summary + a pointer to this file; the full reference lives here.

### Signup email-domain validation

New customer signups via magic link are gated on the email domain to block disposable / spam-trap email services. `lib/email-validation/` exposes `validateEmailForSignup(email)` which runs two layers in order: (1) a DB lookup against the **`disposable_email_domains`** table (`domain TEXT PRIMARY KEY` + `added_at`, an indexed PK probe — sub-millisecond on a warm Postgres); (2) a DNS MX lookup via `node:dns/promises` with a 3 s timeout, **fail-open** on timeout / SERVFAIL / network blip but rejecting on a definitive `ENOTFOUND` / `ENODATA` (catches typos like `gmial.com` or `gmail.con` without locking customers out when the worker's DNS is flaky). The disposable lookup ALSO fails open on DB error — a transient DB blip is not worth blocking a legitimate signup over; the MX check is still a downstream filter. Wired into `lib/auth.ts` `sendMagicLink` BEFORE the banned-user check, and ONLY runs for emails that do NOT already have a `user` row — existing accounts always bypass the gate (so the admin-triggered Orbit magic-link path keeps working for legacy users, and a customer onboarded before their provider was added to the upstream list can still sign in). On reject the callback throws a Better Auth `APIError("BAD_REQUEST", ...)`; `app/(auth)/_components/auth-form.tsx` already surfaces it via `form.setError("root", …)` so the existing inline error UI just works. Google OAuth signups skip the gate (Google has already verified the address). The block is audit-logged via `auth.magic_link_blocked_email_domain` with the `reason` (`disposable` / `unknown_domain` / `invalid_format`).

**Blocklist refresh.** The `disposable_email_domains` table starts empty after migration. It is populated weekly by the **`disposable-emails.refresh` pg-boss cron** (Sundays 04:00 UTC, `retryLimit: 0`, `policy: "exclusive"`, 5-min budget — handler at [disposable-emails-refresh.ts](lib/worker/handlers/disposable-emails-refresh.ts)) and on-demand by an operator via **`pnpm refresh:disposable-emails`** ([scripts/refresh-disposable-domains.ts](scripts/refresh-disposable-domains.ts)). Both paths share `lib/email-validation/refresh.ts` `refreshDisposableEmailDomains()`, which fetches the CC0-licensed upstream list from `disposable-email-domains/disposable-email-domains` and replaces the table contents in a single TRUNCATE + bulk INSERT transaction (all-or-nothing — a crash mid-refresh leaves the prior contents intact). The helper refuses to overwrite if the upstream returns fewer than 1000 entries, guarding against an empty-body 200 that would otherwise blank the blocklist. After deploying this feature for the first time, run `pnpm refresh:disposable-emails` once to prime the table — otherwise the gate is effectively off until the first Sunday cron tick. Failures (upstream HTTP error, suspicious size, DB error) are surfaced in the cron's `disposable_emails.refresh_failed` audit log; the prior blocklist contents remain in place either way.

### Email (EmailIt)

All email goes through **EmailIt** (API v2, `https://api.emailit.com/v2`, `Authorization: Bearer EMAILIT_API_KEY`) — not SMTP. `lib/email/index.ts` `sendEmail()` / `enqueueEmail()` keep their signatures; under the hood `sendEmail()` calls `lib/emailit/emails.ts` `sendEmailViaApi()` (`POST /emails`). React Email templates are still rendered to HTML upstream and passed as strings — that pipeline is unchanged. `EMAILIT_FROM` must be on a domain verified in the EmailIt workspace.

**Webhooks.** `POST /api/webhooks/emailit` ingests EmailIt delivery telemetry (`email.delivered|bounced|complained|failed|rejected|suppressed`) into the `email_events` table. EmailIt signs every delivery with HMAC-SHA256; the route verifies it per `emailit.com/docs/webhooks/request-signature/` — headers `X-Emailit-Signature` (hex digest) + `X-Emailit-Timestamp` (Unix seconds), signed content `{timestamp}.{rawBody}`, key = the full webhook secret string incl. the `whsec_` prefix, with a 300s timestamp tolerance (replay guard) and timing-safe compare. The secret is created in the EmailIt dashboard (Webhooks → Webhook secret) and stored as `EMAILIT_WEBHOOK_SECRET`. Inserts are idempotent on the `evt_xxx` id; the route does no heavy work so it persists inline (no pg-boss job).

**Marketing contact sync.** Krova users are synced into an EmailIt **audience** (`EMAILIT_AUDIENCE_ID`) as contacts so campaigns/automations can be built in the EmailIt dashboard. `lib/emailit/sync-contact.ts` `syncUserToEmailit()` upserts one contact (create via `POST /contacts` with `audiences` on first sync — audience membership cannot be changed on update; subsequently `POST /contacts/{id}`) and recomputes `custom_fields` from DB state: `signup_date`, `email_verified`, `account_role`, `lifecycle_stage`, `space_count`, `owned_space_count`, `is_team_member`, `cube_count`, `running_cube_count`, `credit_balance`, `last_active_at`. The contact's `unsubscribed` flag mirrors the inverse of `user.marketing_opt_in` (banned users are always unsubscribed). **The sync is event-driven, not scheduled.** `lib/emailit/enqueue-sync.ts` exposes `enqueueEmailitSync(userId)` and `enqueueEmailitSyncForSpaceOwner(spaceId)` — fire-and-forget helpers that fan out `emailit.sync-contact` jobs with a `singletonKey` so back-to-back events for the same user collapse into one upsert. Triggers wired into every code path that mutates a tracked field: every cube terminal-status transition via `triggerCubeLifecycleEvent` in `lib/pusher.ts` (running / sleeping / deleted / error — the single chokepoint covering all 14+ cube handlers); top-up paid + refunded in `lib/billing/apply-paid-topup.ts` and the Polar refund webhook; plan credit grants (activation + renewal) in `lib/billing/subscription-handler.ts`; the three Orbit billing actions (void overage, manual refund, retroactive grant); space create + delete (both customer and admin paths) + member removal + invite acceptance; admin role flip in `PATCH /api/orbit/users/[userId]`; every Better Auth `user.create` / `user.update` / `session.create` hook in `lib/auth.ts`; profile opt-in toggle. **There is no recurring full-audience sweep** — `pnpm sync:emailit` is the manual one-shot bulk backfill used when seeding a fresh audience or after adding a new custom field; do not reintroduce a daily cron sweeping thousands of users. When `EMAILIT_AUDIENCE_ID` is unset, contact sync is inert (transactional email + webhooks still work) — every `enqueueEmailitSync*` helper no-ops at the top.

### Outbound webhooks

Customers register HTTPS endpoints that receive signed POSTs whenever entities in their space change. The dashboard lives at `/{spaceId}/webhooks`; the Orbit admin lists every space's endpoints at `/orbit/webhooks`. The `webhook.manage` permission gates create/update/delete/list/deliveries; space owners bypass.

**Event catalogue is the single source of truth in [lib/webhook-events.ts](lib/webhook-events.ts).** Each entry is `{value, label, description, category}` where category is one of `cube | snapshot | backup | domain | tcp_mapping | member | subscription`. `WEBHOOK_EVENT_VALUES` is the tuple Zod needs; `groupedWebhookEvents()` is what the UI renders. Adding an event = adding a row here and firing it from the relevant handler — nothing else. Rule 45 forbids duplicate event lists; every consumer imports from this file.

**Billing events are intentionally NOT delivered as webhooks** — no `hourly_charge`, `top_up_paid`, `plan_credit_granted`, `low_balance`, `overage_*`. Customers needing billing telemetry query the billing endpoints directly. This keeps the webhook stream focused on resource lifecycle.

**Dispatcher: `dispatchWebhookEvent(spaceId, event, data)` in [lib/webhook-dispatch.ts](lib/webhook-dispatch.ts).** Fire-and-forget; never throws. Inserts a `outbound_webhook_deliveries` row per subscribed endpoint inside an envelope `{id, event, createdAt, spaceId, data}` and enqueues `outbound-webhook.deliver`. The per-event `data` payload is built by helpers in [lib/webhook-payloads.ts](lib/webhook-payloads.ts) (`buildCubeSummary`, `buildSnapshotPayload`, `buildBackupPayload`, `buildDomainPayload`, `buildTcpMappingPayload`, `buildMemberPayload`, `buildInvitePayload`, `buildSubscriptionPayload`) — call sites pass loaded rows in, the helpers shape the wire format. **Never inline a payload object at a call site** — drift between handlers is the whole reason these helpers exist (Rule 14).

**Worker handler: [lib/worker/handlers/outbound-webhook-deliver.ts](lib/worker/handlers/outbound-webhook-deliver.ts).** HMAC-SHA256 signature over `{timestamp}.{rawBody}` using the endpoint's decrypted secret; header is `X-Krova-Signature: t=<unix>,v1=<sha256>` (Stripe-style v1). Customers must verify within a 300s replay window (documented in [docs/api/v1.md](docs/api/v1.md)). 10 s fetch timeout; success = `2xx`. Retry config lives at queue level in [ensure-queues.ts](lib/worker/ensure-queues.ts): `retryLimit: 4, retryDelay: 60` (5 total attempts before giving up) — never set retry options per-enqueue. **SSRF guard at delivery time** ([lib/webhook-ssrf.ts](lib/webhook-ssrf.ts)) re-resolves the hostname on every attempt and auto-disables the endpoint (`disabled_reason='ssrf_blocked'`) if it resolves to RFC1918 / loopback / link-local (incl. 169.254.169.254) / IPv6 ULA / link-local — the create-time check would have caught it, but DNS can change post-create. **Flap protection:** `outbound_webhook_endpoints.consecutive_failures` increments on every failed attempt and resets to 0 on success; at 50 consecutive failures the endpoint auto-disables (`disabled_reason='consecutive_failures'`), a one-shot email goes to the space owner (template at [lib/email/templates/webhook-auto-disabled.ts](lib/email/templates/webhook-auto-disabled.ts)), and the dashboard surfaces an "Auto-disabled" badge. Re-enable clears the counter back to 0.

**Delivery retention.** `outbound_webhook_deliveries` rows older than 30 days are deleted by the existing `job-logs.prune` daily cron (03:00 UTC) — see the trailing block in [job-logs-prune.ts](lib/worker/handlers/job-logs-prune.ts). The deliveries index `owhd_created_at_idx` makes the prune `DELETE` cheap.

**Customer actions: [app/actions/outbound-webhooks.ts](app/actions/outbound-webhooks.ts)** — `createWebhook`, `updateWebhook`, `setWebhookEnabled`, `rotateWebhookSecret`, `deleteWebhook`, `listWebhookDeliveries`, `testFireWebhook`, `redeliverWebhookDelivery`. All gated by `webhook.manage` via `requireActionMembershipAndPermission`. **`v1` API surface** at `app/api/v1/spaces/[spaceId]/webhooks/...` mirrors create/list/get/delete/deliveries, also gated by `webhook.manage`. The two POSTs share a single source-of-truth event-name validator (`WEBHOOK_EVENT_VALUES` from [lib/webhook-events.ts](lib/webhook-events.ts)) — never re-list event names in route files.

**Rotation.** "Rotate signing secret" replaces `encrypted_secret` with a fresh `randomBytes(32).toString("hex")` and is shown ONCE in the UI; no UI surface ever decrypts and re-shows the existing secret. The previous secret stops verifying future deliveries immediately. Customers delete + re-create only if they want a fresh endpoint URL.

### Analytics (Google Tag Manager)

Client-side analytics goes through a **hybrid** setup: the canonical GTM loader snippet (the official `(function(w,d,s,l,i){...})` IIFE plus a `<noscript>` `ns.html` iframe) is injected directly into the SSR HTML by `app/layout.tsx` via `next/script` with `strategy="afterInteractive"` — this is what makes the snippet visible to Google's "Install Google Tag Manager → Test your website" detector, which does a static-HTML check and does NOT execute deferred client JS. On top of that, **`analytics`** (the [getanalytics.io](https://getanalytics.io/) core) + the **`@analytics/google-tag-manager`** plugin + **`use-analytics`** are layered for the developer-facing API. `lib/analytics.ts` is a `"use client"` module that lazily constructs a single `Analytics({ app: "krova", plugins: [googleTagManager({ containerId })] })` instance via `getOrCreateAnalytics(containerId)` and exports module-level `track()` / `identify()` / `page()` helpers that no-op when the instance is not initialized. Both the `next/script` snippet and the plugin reference the same container id, so GTM internally dedupes — there is one GTM container per page despite two initialization paths. Every `analytics.track(event, payload)` becomes a `window.dataLayer.push({ event, ...payload })`, which is where the vendor tags configured inside the GTM workspace (GA4, Ads, Facebook, LinkedIn, etc.) pick events up. `components/analytics-provider.tsx` wraps the tree with `use-analytics`'s `AnalyticsProvider`, mounts an internal `PageTracker` that calls `analytics.page({ path: pathname })` on every `usePathname()` change (route changes in the App Router don't reload the GTM container script, so manual `page()` calls are how the dataLayer sees the navigation), and is wired into `app/layout.tsx` inside `TooltipProvider`. The container id is read from `env.NEXT_PUBLIC_GTM_CONTAINER_ID` (regex-validated as `GTM-XXXXXXX` in `lib/env.ts`) and passed down as a prop — no client code reads `process.env` directly. **When `NEXT_PUBLIC_GTM_CONTAINER_ID` is unset, the inline `<Script>` + `<noscript>` aren't rendered, the provider renders children verbatim, and every track/identify/page call is a silent no-op** — analytics is effectively off in dev and in any environment without the var. **A GA4 measurement id (`G-XXXX`) is NOT a GTM container id** — the customer-facing GA4 property is set up as a tag inside the GTM workspace, not in this env var. Custom events fire two ways: (1) inside React components via `const { track } = useAnalytics(); track("event", { foo: 1 })` from `use-analytics`, or (2) anywhere on the client via `import { track } from "@/lib/analytics"; track("event", { foo: 1 })`. Server-side events are NOT in scope yet — GTM's loader is browser-only, so server actions / API routes that need to record an event do it by returning a flag that a client component then fires.

### Deployment version-skew recovery

A browser tab left open across a deploy still holds Server Action ids + JS/CSS chunk references from the PREVIOUS build. When those stale references hit the new server they fail — `Failed to find Server Action <hash>. This request might be from an older or newer deployment` for actions, `ChunkLoadError` for assets. Two layers make this invisible to customers:

1. **Stable Server Action ids — `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY`.** Set as a **build-time** env var in Dokploy (base64-encoded, 16/24/32-byte AES key — generate with `openssl rand -base64 32`), the SAME value across every deploy and never rotated casually. By default Next.js derives a fresh random key per build, which re-hashes every action id on every deploy; a fixed key makes the ids deterministic so an action's id survives a deploy **even when its body changes**, so a stale tab keeps working. The key is embedded in the build output and used automatically at runtime — it is consumed by Next.js internally, NOT read by app code, so it deliberately does NOT live in `lib/env.ts` (Rule 5 governs app code reading `process.env`). On Railpack/Dokploy, confirm it actually reached the build (older Dokploy wrapped Railpack build env in quotes, [#1982](https://github.com/Dokploy/dokploy/issues/1982), now fixed).
2. **Client reload net — `lib/deployment-skew.ts`.** Single source of truth: `isDeploymentSkewError(error)` (the match-fragment list) + `recoverFromDeploymentSkew()` (a loop-guarded `window.location.reload()` — stamps `sessionStorage` BEFORE reloading and refuses a second reload inside a 10s cooldown so a genuinely-recurring error can never trap the user in a reload cycle). Wired into BOTH (a) ALL FOUR route error boundaries ([app/global-error.tsx](app/global-error.tsx), [app/(dashboard)/error.tsx](app/(dashboard)/error.tsx), [app/(dashboard)/[spaceId]/error.tsx](app/(dashboard)/[spaceId]/error.tsx), [app/(orbit)/error.tsx](app/(orbit)/error.tsx)) via a `useEffect` — server-action skew that React routes to a boundary reloads instead of showing the error screen — and (b) a window-level `error`/`unhandledrejection` listener ([components/deployment-skew-reload.tsx](components/deployment-skew-reload.tsx), mounted once in [app/layout.tsx](app/layout.tsx)) for stale-chunk errors during prefetch/dynamic import that never reach a boundary. **Every route group with an `error.tsx` MUST wire this** — the per-space `[spaceId]` boundary (the most-hit customer surface) was missed in the original implementation and surfaced the error screen instead of reloading until the 2026-05-29 audit caught it; when adding a new `error.tsx`, add the same `useEffect`.

**Do NOT add Next.js `deploymentId` on this stack.** It was deliberately rejected: the client compares its BUILD-baked deployment id against the server's RUNTIME `x-nextjs-deployment-id` header, and `next start` re-evaluates `next.config.mjs` at runtime. Railpack's runtime image has no `.git`, so any config that derives the id from `git rev-parse` (or a `Date.now()` fallback) resolves to a DIFFERENT value at runtime than at build → every customer's tab mismatches on every request → **fleet-wide infinite reload loop**. `deploymentId` is only safe with a per-deploy value present identically at build AND runtime (which Dokploy/Railpack doesn't hand you cleanly), and the reactive reload net above covers the same cases without that footgun. If you ever reconsider, the id MUST be build==runtime deterministic — verify before shipping.
