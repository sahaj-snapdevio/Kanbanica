# Image Hosting, Rootfs & Guest Networking

> On-demand detail extracted from CLAUDE.md. CLAUDE.md keeps only a short summary + a pointer to this file; the full reference lives here.

### Image Hosting

Built kernel + rootfs images live on the Dokploy host's local filesystem under `KROVA_BUILD_OUTDIR`. The `platform_images` table records each artifact's local path, sha256, and size. `pnpm build:images` produces them. The `server.pull-images` setup phase reads from the worker's filesystem (via the `/opt/krova-build:/opt/krova-build` bind mount) and SFTPs each file to `/var/lib/krova/images/` on the bare-metal server. No object storage, no R2.

**Kernel is built from source, not downloaded from Firecracker CI as a binary.** The Firecracker CI prebuilt vmlinux at `s3.amazonaws.com/spec.ccfc.min/firecracker-ci/v1.15/x86_64/vmlinux-6.1.x` lacks the kernel features Docker / OCI runtimes need: overlay filesystem, br*netfilter, veth, bridge, iptables NAT, IPVS, vxlan, user namespaces, the cgroup_pids/blkio/bpf controllers. It ALSO disables `CONFIG_NF_TABLES` (nftables) — required for any modern userspace tool that calls `nft` directly, including the default `iptables-nft` binary that ships with Ubuntu 24.04. Our `build_kernel` function in `setup/images/build-all-images.sh` pulls **Linux 6.1.172 source** (latest 6.1 LTS) + the **Firecracker CI baseline `.config`** (downloaded from S3 at `firecracker-ci/v1.15/x86_64/vmlinux-6.1.155.config` — the only 6.1 config Firecracker publishes for the v1.15 line; verified by listing the S3 prefix), layers Docker-required `=y` options + nftables `=y` options on top, runs `make olddefconfig` + `make vmlinux`, and verifies the resulting ELF before saving. The script tracks the kernel source version (`KVER`) and the config baseline (`CONFIG_KVER`) as separate variables — `make olddefconfig` safely forward-upgrades the older baseline config to the newer source tree, defaulting any new `CONFIG*\*`symbols. Bump`CONFIG_KVER`only when Firecracker CI ships a newer baseline. **DO NOT use the "starting point" baseline at`firecracker/resources/guest_configs/microvm-kernel-ci-x86_64-6.1.config`in the source repo** — it is incomplete (missing`CONFIG_GENERIC_MSI_IRQ`, `CONFIG_PCI`, `CONFIG_LIBNVDIMM`, etc.) and produces a kernel where modern virtio_mmio devices fail to probe with EINVAL because they cannot allocate MSI-X interrupt vectors. The S3 config is the post-`olddefconfig` config that ships alongside the working CI binary, so it is guaranteed to produce a working kernel for the same Firecracker release. The single kernel serves all rootfs distros — Linux ABI is stable, so per-distro kernels would be wasted complexity.

**Supported guest distros: single source of truth in `config/platform.ts` `CUBE_IMAGES`.** The platform currently ships two Ubuntu rootfs flavors — Ubuntu 24.04 (default for ~80% of customer workloads, what Docker/Dokploy/k3s docs assume) and **Ubuntu 24.04 + Docker** (same Ubuntu base with Docker Engine + Compose plugin preinstalled from Docker's official apt repo per [docs.docker.com/engine/install/ubuntu](https://docs.docker.com/engine/install/ubuntu/), with `docker.service` and `containerd.service` enabled at boot — for customers who want a ready-to-run container host with no install step on first boot). The kernel already carries every `CONFIG_*` Docker / OCI runtimes need (overlay, bridge, veth, vxlan, cgroup_*, NF_TABLES, etc.) — the preinstall variant just spares the customer the apt install on first boot. The Docker apt repo stays configured post-install so the customer can `apt update && apt upgrade docker-ce` for newer releases themselves. The bash builder only carries a `build_deb` (debootstrap) path with a `PREINSTALL_DOCKER` toggle for the Docker variant; the `build_rpm` (`dnf --installroot`) path was removed when AlmaLinux 9 was dropped, because RHEL 9 doesn't ship `systemd-networkd` in BaseOS/AppStream (it lives in the CRB repo) and guest networking depends on systemd-networkd uniformly across distros. To re-add a RHEL-family distro you'd need to re-introduce a `build_rpm` function that enables CRB before `dnf install systemd-networkd`. To add or remove a rootfs flavor, edit ONLY `CUBE_IMAGES` in `config/platform.ts`. Everything else derives:

- `IMAGE_OPTIONS` (UI dropdown) is a `.map()` of `CUBE_IMAGES` in the same file.
- `scripts/build-images.ts` imports `CUBE_IMAGES` and derives both the compress/register pipeline AND a generated bash snippet (parallel arrays of distro params) written to a temp file.
- `setup/images/build-all-images.sh` sources that temp file via `$KROVA_DISTROS_FILE`, then dispatches generically (no `do_<id>` functions, no per-distro case branches). Run via `pnpm build:images` (which writes the snippet) OR directly (which falls back to a hardcoded copy of the same list — keep that fallback in sync if you ever need direct invocation).
- `platform_images` table rows are auto-pruned by `scripts/build-images.ts` after each successful build: any row whose `name` is not in `CUBE_IMAGES` (or "kernel") gets deleted, and its on-disk artifact in `$KROVA_BUILD_OUTDIR` is also removed — provided no active (non-deleted) cube still references it via `cubes.imageId`. If active cubes do reference an orphan, the prune is skipped and a loud warning lists the cube IDs so the operator can either re-add the distro to `CUBE_IMAGES` or migrate the cubes off it. This means removing a distro is a one-step action: delete from `CUBE_IMAGES`, run `pnpm build:images`, the DB and disk both self-clean.

The only live host-side build/setup artifacts are `setup/images/build-all-images.sh` (the image builder, invoked by `pnpm build:images`), `setup/images/krova-agent` (the in-guest agent), and `setup/server/krova-vsock-pty` + `setup/server/krova-vsock-exec` (the host vsock helpers, deployed by the `install` setup phase). The former legacy reference scripts `setup/server/setup-server.sh` and `setup/images/upload-to-r2.sh` were **removed (2026-05-31)** — server setup is driven entirely by the worker's `server.*` phase handlers (`lib/worker/handlers/server-*.ts`), never a hand-run shell script.

The narrow distro surface keeps the maintenance + testing burden achievable for a small team — solo-dev rule of thumb: deeply test on Ubuntu 24.04 (the dominant distro), trust that the Docker-preinstall variant tracks the same Ubuntu base, run a manual smoke boot of both (including the Docker preinstall variant: `docker run --rm hello-world` post-boot) only after kernel rebuilds. Removing a distro is destructive for any cubes already provisioned against it — the rootfs file on the host stays usable so existing cubes keep booting, but the dropdown stops offering it for new cubes. All required options are compiled in (`=y`) because the build runs only `make vmlinux` (never `make modules_install`), so the rootfs has no `/lib/modules/<kver>` tree — any feature built as a module (`=m`) would have no `.ko` to load. `CONFIG_MODULES=y` is kept (some userspace tools probe for module support) but no required feature ships as `=m`. First-time build is ~10–25 min on a beefy box, ~120 MB source download, output ~80–150 MB. Cached locally: if `$OUTDIR/kernel/vmlinux` exists, the function returns immediately.

**Refreshing images on already-active servers.** After `pnpm build:images` produces new artifacts (kernel rebuild, security-patched rootfs, etc.), the operator clicks the **Update Images** button on the server detail page (visible only when `setupPhase === "ready"`). This enqueues a `server.update-images` job that reuses the same image-sync core as the `pull_images` setup phase but does NOT touch phase state or reboot the box — safe to run on active servers with customer Cubes scheduled. Implemented by extracting the validate / SFTP / sha256 / decompress logic into `lib/server/sync-images.ts:syncPlatformImages()` shared by both handlers. Existing Cubes keep their currently-loaded kernel until they next boot from cold; new Cubes pick up the refreshed images immediately.

**Live RAM resize (virtio-mem) and live disk grow.** Every cube boots with `mem_size_mib=1024` (the platform-min `VIRTIO_MEM_BOOT_FLOOR_MIB`) plus a `virtio-mem` device with `total_size_mib=31744` (platform-max minus the floor). After `InstanceStart`, `plugInitialMemory` PATCHes `/hotplug/memory` to bring the cube up to its configured `ramMb`. This means total guest RAM = `mem_size_mib + plugged_size_mib`. The boot floor is invariant for the life of the cube — never raise it on a live cube. Live RAM grow goes through the same PATCH endpoint; live disk grow truncates the host rootfs file, PATCHes `/drives/rootfs` so Firecracker re-reads the size, then runs `resize2fs /dev/vda` inside the guest via `guestExec`. CPU changes are NOT live — Firecracker has no CPU hotplug; resize falls back to a cold restart cycle. The `cubes.has_virtio_mem` flag tracks whether a cube has the device declared (true for any fresh boot, including transfer destinations and snapshot restores).

**AVX-512 CPUID mask (custom CPU template).** Firecracker passes the host CPUID straight through to the guest, advertising AVX-512 in CPUID leaf 0x7, but the microVM's XSAVE state does NOT back the ZMM/opmask registers — so any AVX-512 instruction the guest runs faults with **SIGILL** (Geekbench 6, NumPy-with-AVX-512, etc. — the 2026-06-01 benchmark cube). On every boot, right after `PUT /machine-config`, `createCube`/`startCube` do `PUT /cpu-config` with `avx512MaskCpuidModifiers()` from [lib/ssh/cpuid-template.ts](lib/ssh/cpuid-template.ts), which clears the leaf-0x7 AVX-512 feature bits so guest software never detects AVX-512 and falls back to AVX2 (AVX2/AES/SSE4/BMI stay untouched). **It is a CUSTOM template, NOT the static `T2`/`T2A` enum, on purpose:** static templates enforce an exact host family/model/stepping allowlist at `InstanceStart` and ERROR — refusing to boot — on any non-allowlisted CPU (Cooper Lake, Sapphire/Emerald Rapids, off-stepping Skylake, AMD non-Milan), and the presence of AVX-512 does NOT prove allowlist membership; a custom template skips that gate entirely (verified in Firecracker v1.15.1 `custom_cpu_template.rs` — the `Custom` arm returns the template unchecked), so the SAME vendor-agnostic mask is safe on every Intel/AMD host (clearing an absent bit is a no-op). The `PUT /cpu-config` is **fail-safe** — wrapped so a failure (e.g. a host on a Firecracker predating `/cpu-config`) logs a warning and the cube still boots (worst case = no mask = pre-fix behavior), so the change can never brick a boot. Verified-against-v1.15.1 details (apply overwrites `entry.flags` → leaf 0x7 carries `flags=1` SIGNIFICANT_INDEX; apply errors if the leaf/subleaf is absent → we touch only the universal leaf 0x7 subleaf 0; the 32-char bitmap format) live in the module's header comment, and the pure bitmap/modifier builders are unit-tested in [lib/ssh/cpuid-template.test.ts](lib/ssh/cpuid-template.test.ts). Validated end-to-end on a real Firecracker v1.15.1 host (Docker, NumPy, gcc, AES-NI, stress-ng all-methods, live disk grow — clean; T2 was empirically shown to brick boot on a non-allowlisted host). **Existing running cubes only pick up the mask on their next cold boot** (cold-restart / wake-from-cold / snapshot-restore / transfer / auto-relaunch) — a resume-from-paused does not re-run the boot config.

**Refreshing a server's routing — the Refresh Routing button.** The **Refresh Routing** button on the server detail page (next to Update Images, visible only when `setupPhase === "ready"`) enqueues a `server.refresh-caddy` job that re-asserts the server's entire external routing, idempotently, in three steps: (1) both Cloudflare DNS records via `setUpServerCloudflareOrigin()` — the proxied `<hostname>` origin and the DNS-only `connect.<hostname>` record (create-or-update through `ensureDnsRecord`); (2) the wildcard Origin CA cert, re-installed on Caddy from the `CLOUDFLARE_ORIGIN_CERT` / `CLOUDFLARE_ORIGIN_KEY` env — so a rotated cert is picked up here, no CLI needed; (3) the Caddy `srv0` routes array — the branded landing route (one host-matched route covering **both** `serverOriginHostname()` and `serverConnectDomain()`) plus one route per active customer custom domain from `domain_mappings` — swapped via a single atomic `PATCH /config/apps/http/servers/srv0/routes`, plus the ACME automation policy and the origin-hostname `automatic_https.skip`. An existing server that pre-dates hostname-derivation (missing its grey `connect.` record, stale landing route) is fully healed by this one button. `reconcileCaddyRoutes()` in `lib/ssh/caddy.ts` does the atomic route rebuild; `customDomainRoute()` (extracted from `addCustomDomainRoute()`) is the shared per-domain route builder; `getActiveCustomDomainsForServer()` in `lib/server/server-domains.ts` is the DB query.

**Upgrading Caddy on an active server — the Update Caddy button.** The **Update Caddy** button on the server detail page (next to Update Images / Refresh Routing, visible only when `setupPhase === "ready"`) enqueues a `server.update-caddy` job that upgrades the Caddy package to the platform-pinned `CADDY_VERSION` (`config/platform.ts`) — `apt-mark unhold` → `--only-upgrade caddy=<version>` → `apt-mark hold` on Debian/Ubuntu, `dnf upgrade caddy` on RHEL — restarts the service, and verifies the installed version. Caddy's `--resume` systemd override reloads `autosave.json` on restart so all `srv0` routes survive; the job does not change phase state, reboot the box, or touch routes/DNS. New servers get the pinned `CADDY_VERSION` directly from the `install` phase (`apt-get install caddy=<version>`).

**Guest networking (systemd-networkd, all supported distros).** Every cube's guest network is configured by writing `/etc/systemd/network/10-eth0.network` into the loop-mounted rootfs via `writeCubeGuestNetworkConfig()` in [lib/ssh/cube-guest-network.ts](lib/ssh/cube-guest-network.ts). The same helper is called by `createCube()`, `cube.transfer` (after rsync to a new server), `backup.redeploy` (with the new internal IP) and `cube.import-rootfs` (same). This is renderer-agnostic — systemd is universal, and the rootfs builder enables `systemd-networkd.service` on the Ubuntu rootfs. It replaced a legacy netplan-YAML implementation that was renderer-specific (a non-Ubuntu renderer would ignore the file and the cube would boot with no IP — SSH, TCP port mappings, and any host-to-guest networking would fail; the vsock-based browser terminal still worked because vsock is independent of eth0). The helper also wipes any stale `/etc/netplan/99-krova.yaml` from existing Ubuntu cubes so transfers and redeploys do not carry old IPs forward. `backup.redeploy` and `cube.import-rootfs` ALWAYS mount the rootfs to rewrite the network config (previously these flows only mounted for the optional SSH-key inject, so redeployed / imported cubes booted with the source cube's old IP even on Ubuntu). DNS is kept in static `/etc/resolv.conf` (set by the rootfs builder) so `systemd-resolved` is deliberately NOT enabled.

**Cube IPv6 + globally-unique networking.** Every cube is dual-stack and globally unique fleet-wide, both families derived from the server's `servers.bridge_subnet` (`S`, range `[1, 511]`): IPv4 on `198.18.0.0/15` (the S-th `/24` — `cubeIpv4Address(S, octet)` = `198.18.0.0 + S×256 + octet`, gateway `.1`) and IPv6 `fd00:c0be:<S-hex>::<octet>/64` (gateway `::1`, **NAT66 over the ULA `fd00:c0be::/32`**). IPv6 is **outbound + DNS only** — there is no inbound IPv6 (no AAAA, no `ip6tables` DNAT, no `servers.public_ipv6`); inbound stays IPv4 via the existing port-mapping DNAT. The pure address math is the single source of truth in [lib/server/cube-network.ts](lib/server/cube-network.ts) (`cubeIpv4Address`/`cubeIpv4Subnet`/`cubeIpv4Gateway`/`cubeIpv6Address`/`cubeIpv6Subnet`/`cubeIpv6Gateway`, `subnetOf(ip)` = `(ipToInt(ip) − ipToInt(CUBE_IPV4_BASE)) >> 8` — **throws if `ip` is outside `198.18.0.0/15`**, `octetOf(ip)` — base-10 last-octet parse, unchanged), unit-tested via `pnpm test` (`tsx --test` over `lib/**/*.test.ts`, zero new deps). **Never hardcode an IPv4 prefix, the gateway, or the v6 prefix again — derive from `bridge_subnet` via `cube-network.ts`.**

**Why `198.18.0.0/15` (the IPv4 range — rebased off `10.0.0.0/8` on 2026-05-31).** The original `10.<…>.<octet>/24` scheme put cubes inside `10.0.0.0/8`, which collides with the IPv4 ranges customer software uses INSIDE the cube — Docker Swarm's default overlay/ingress pool is `10.0.0.0/8`, and k3s (`10.42/10.43`) / kubeadm (`10.96/12`) / Cilium default into `10.x` too. A live incident (2026-05-30/31) proved it: a Dokploy cube whose `dokploy-network` overlay was allocated the SAME `/24` as its own `eth0` routed the container's replies into the overlay instead of back to the gateway, so Caddy/external access timed out while a plain non-overlay service on the same port worked. **There is no conflict-free IPv4 range** — every sizable block is some tool's default (`172.16/12` Docker `docker0`/Calico; `192.168/16` LANs/Docker; `100.64/10` Tailscale, hard-routed; `240/4` Class-E, rejected by non-Linux stacks; TEST-NET blocks too small). `198.18.0.0/15` (RFC 2544 benchmarking, IANA `Forwardable=True, Reserved-by-Protocol=False` — fully usable on Linux, NAT'd so it never appears on the public internet) is the **lowest realistic risk for Krova's workloads**, NOT zero-risk (IPv4 can't offer that). **Residual caveat — Clash / Mihomo / sing-box / Surge Fake-IP:** `198.18.0.0/15` is that proxy family's default Fake-IP range, but it only triggers in **TUN mode** (a personal-device pattern, uncommon on a production cloud server). **Customer escape-hatch:** a one-line `fake-ip-range` / `inet4_range` change on their side. **Operator hedge:** the range is a config constant (`CUBE_IPV4_BASE`), so if one customer ever collides the operator can relocate just that server's cubes without a code change. **Capacity ceiling: 511 servers × 253 cubes ≈ 129k cubes** (vs the old `0xffff` S cap) — far beyond foreseeable bare-metal scale; if Krova ever nears 511 servers, switch to a smaller per-server block (`/25` → 1024 servers) or a second range. **IPv6 is unchanged** — `fd00:c0be:<S-hex>::/64`, suffix = the host octet (hex); S is simply capped at 511 by the IPv4 range now.

The host side is the single source [lib/server/cube-network-host.ts](lib/server/cube-network-host.ts) `applyHostNetworking(client, S, opts)` — dual-stack `br0` addresses, IPv4+IPv6 forwarding sysctls written to a DEDICATED `/etc/sysctl.d/98-krova-forwarding.conf` (NEVER `99-krova.conf`, whose vm tuning the network phase used to clobber on reboot), WAN-iface `accept_ra=2` detection (pure builder `buildForwardingSysctlScript`, unit-tested — derives the WAN from the v6 **and** the always-present v4 default route, so a forwarding host with no v6 default route *yet* STILL gets `accept_ra=2` and can LEARN one from RAs; the prior v6-only derivation emitted no `accept_ra` line on such a host and `forwarding=1` then made the kernel ignore every RA → a permanent v6 blackhole for every cube on it — the root cause behind the "DNS timeouts + intermittent drops inside cubes" incident), NAT66 + IPv4 MASQUERADE, **raised conntrack UDP timeouts** (`nf_conntrack_udp_timeout` 180 / `_stream` 600 in a DEDICATED `98-krova-conntrack.conf` + `modules-load.d` so an idle WireGuard-mesh / UDP-overlay flow keeps its NAT mapping instead of stalling until a restart — audit W2), a **mangle-table `TCPMSS --clamp-mss-to-pmtu` FORWARD clamp** (in the **mangle** table, NOT filter — the filter FORWARD egress `-i br0 ! -o br0 -j ACCEPT` is terminating and would short-circuit a clamp appended there; audit W3), an **egress-only `br0` FORWARD** (not a blanket `-o br0 ACCEPT`, which would open unsolicited inbound v6 to cube ports), and a **stateful default-deny `INPUT` firewall on BOTH families** (allow-list-first — lo / ESTABLISHED,RELATED / ICMP+ICMPv6 / SSH 2822 / 80 / 443 tcp+udp — then `-P INPUT DROP`). It resolves the `*-legacy` iptables/ip6tables backend to match the cube-DNAT path, persists both families (Debian `rules.v4`/`rules.v6` or RHEL `/etc/sysconfig/{iptables,ip6tables}` + enabling BOTH services), and on the retrofit path (`opts.retrofit`) arms a `sleep 60 && -P INPUT ACCEPT` cancel-on-success rollback net so a mistaken allow-list can't lock the worker out. Shared by the `network` setup phase (new servers) and the re-IP migration (existing servers); the conntrack-timeout + mangle-MSS-clamp tuning is retrofitted onto existing active servers by `pnpm install:network-tuning` (active-host-safe + idempotent — verified end-to-end with a live cube boot: ping/DNS/HTTPS/3 MB download all pass, clamp fires, no connectivity break).

The guest side stays in `writeCubeGuestNetworkConfig(client, mountDir, internalIp)` — **signature UNCHANGED**: it derives `S = subnetOf(internalIp)` + `octet = octetOf(internalIp)` from `internalIp` itself, so no caller threading or `createCube` opts change was needed. It writes a dual-stack `10-eth0.network` (repeated `Address=`/`Gateway=` for both families, v4 gateway gated on the same `S` so it can't drift from `br0`, plus **`IPv6AcceptRA=no`** — the cube's v6 addr+gw are STATIC so it never needs RAs; left unset, systemd-networkd runs an RA-client on eth0 that never gets an RA, times out, periodically reconfigures the link and re-arms DAD → the static v6 flaps present↔absent, the observed instability) and **unconditionally** rewrites a **v4-first** `/etc/resolv.conf` from `CUBE_DNS_SERVERS` in `config/platform.ts` (`1.1.1.1`, `2606:4700:4700::1111`, `2001:4860:4860::8888` — glibc `MAXNS=3`) **plus an `options ${CUBE_RESOLV_OPTIONS}` line (`timeout:1 attempts:2 single-request-reopen`)**. v4-first + `timeout:1` means DNS NEVER depends on v6 egress — a blackholed/flapping v6 path costs ≤1s on a fallback instead of ~5s on the primary (validated on real glibc: 5019ms→7ms). The transport family does not constrain the record type, so v4-first loses no IPv6 capability for customer apps. `snapshot.restore` now also rewrites guest networking (a pre-re-IP snapshot would otherwise boot a stale old IP). `setup/images/build-all-images.sh` bakes a **byte-identical** resolv.conf (enforced by `lib/ssh/cube-guest-network.test.ts`, replacing the old comment-only "keep in sync"). **Running cubes are retrofitted live by `pnpm install:guest-network`** — it writes the new resolv.conf (glibc re-reads → DNS fix is live immediately) + the `IPv6AcceptRA=no` unit to disk (flap fix applies on the next cold restart), touching NO live link (no `networkctl reload`/`systemctl restart`/`ip link`), so a running cube's sessions are never disturbed. Config constants: `CUBE_IPV4_BASE` (`198.18.0.0`) / `CUBE_IPV6_PREFIX` (`fd00:c0be`) / `CUBE_BRIDGE_SUBNET_MIN` (`1`) / `CUBE_BRIDGE_SUBNET_MAX` (`511` — the `198.18.0.0/15` ceiling, was `0xffff`) / `CUBE_DNS_SERVERS` / `CUBE_RESOLV_OPTIONS`.

With `JAILER_ENABLED=true` fleet-wide, every cube is jailed, but re-IP needs no mechanism change: the jailer uses `--new-pid-ns` + `--cgroup-version 2` (NO `--netns`), so TAP/br0/NAT/v6-sysctls/ip6tables all live in the host net namespace and `applyHostNetworking` is mode-agnostic; the loop-mount writes the canonical `/var/lib/krova/cubes/<id>/rootfs.ext4` inode that is hardlinked into the chroot. A **paused** cube was SKIPPED by the cutover entirely (a resume never reloads eth0 — committing a new IP would brick it; it converts only on a real cold restart, still invariant-guarded in `cube-wake.ts`). The octet is preserved across the per-server re-IP but NOT across a cross-server transfer (`cube.transfer` derives both addresses from a fresh destination octet and rolls both back on failure). The fleet-wide cutover (`10.x → 198.18.x`) was **completed 2026-05-31**; it computed new IPs explicitly from the old ones (never round-tripping a `10.x` address through the `198.18`-based `subnetOf`) and **detected in-guest Docker Swarm / k8s, surfacing — never auto-running — the one-time `docker swarm init --force-new-cluster --advertise-addr <new IP>` recovery** (a re-IP breaks the cluster's pinned advertise-addr; once on `198.18` the cube's own overlays use Docker's default `10/8` pool collision-free). The one-shot migration tooling (`migrate-cube-network` / `check-duplicate-cube-ips` / `lib/cubes/reip.ts`) was removed post-cutover — the runtime is now `198.18`-only with the fail-loud guards above. Design + rollout: the IPv4 rebase [docs/superpowers/specs/2026-05-31-cube-ipv4-rebase-198-18-design.md](docs/superpowers/specs/2026-05-31-cube-ipv4-rebase-198-18-design.md) + [docs/superpowers/plans/2026-05-31-cube-ipv4-rebase-198-18.md](docs/superpowers/plans/2026-05-31-cube-ipv4-rebase-198-18.md), and the prior dual-stack work [docs/superpowers/specs/2026-05-30-cube-ipv6-design.md](docs/superpowers/specs/2026-05-30-cube-ipv6-design.md), [docs/superpowers/plans/2026-05-30-cube-ipv6-rollout-runbook.md](docs/superpowers/plans/2026-05-30-cube-ipv6-rollout-runbook.md).

**Cube guest distro coverage.** Two Ubuntu rootfs flavors are supported — Ubuntu 24.04 and Ubuntu 24.04 + Docker (Ubuntu base + Docker Engine + Compose plugin from Docker's official apt repo, with `docker.service` + `containerd.service` enabled at boot). The build pipeline is debootstrap-only; the prior `build_rpm` path was removed with AlmaLinux. If a RHEL-family distro is ever re-added, a new `build_rpm` function would need to (1) enable the AlmaLinux/Rocky **CRB repo** before `dnf install systemd-networkd` (the package is NOT in BaseOS/AppStream and is NOT bundled with base `systemd` as on Debian/Ubuntu) and add it to the `required_pkgs` hard-fail list so a partial transaction fails loudly, (2) leave `firewalld` installed but NOT enabled (enabled-by-default silently drops customer TCP-mapping traffic at the guest edge for any port other than 22; the host iptables DNAT already covers ingress filtering at the bridge layer), (3) set SELinux to `disabled` in `/etc/selinux/config` because the platform kernel does not have `CONFIG_SECURITY_SELINUX` compiled in (only AppArmor) — the RHEL default of `enforcing` triggers confused service errors on every boot, and (4) mask `NetworkManager` as defense in depth in case `install_weak_deps=True` ever pulls it in. The ONLY guest-side network manager is systemd-networkd. Adding any new guest-side config that touches networking / firewalls / mandatory-access-control MUST be considered against this matrix.

**In-guest OS security updates (unattended-upgrades).** Every cube rootfs ships `unattended-upgrades` ENABLED with the Ubuntu package's default **security-only** origin set, a **daily** `apt-daily-upgrade.timer`, **`Automatic-Reboot "false"`** (the kernel is external — a guest reboot is meaningless and Firecracker treats it as shutdown→auto-relaunch), and `needrestart` configured (`/etc/needrestart/conf.d/99-krova.conf`: `restart="l"`, `kernelhints=0`, `ucodehints=0`, `override_rc` pinning `krova-agent`+`ssh`) so a security upgrade **never bounces sshd or krova-agent** mid-session. This is the only path that patches a long-lived cube's userspace CVEs — the per-cube rootfs is immutable, so `build:images` only reaches NEW cubes. Existing running cubes provisioned before this policy are retrofitted in place (over the vsock `exec` channel, no service restart) with `pnpm install:unattended-upgrades`. The config is written by `setup/images/build-all-images.sh` (new cubes) and `scripts/install-unattended-upgrades-fleet.ts` (existing cubes) — keep the two byte-identical. The needrestart `kernelhints=0` setting is also what silences the cosmetic "Failed to retrieve available kernel versions" line on `apt upgrade` (the kernel is host-supplied, so the in-guest kernel check has nothing to enumerate). Krova/customer boundary: Krova owns the host + Firecracker + the guest kernel (swapped on cold-restart); the customer owns guest userspace — see [docs/security/shared-responsibility.md](docs/security/shared-responsibility.md).

**Guest hostname & cloud-init.** Every cube's guest hostname is set from its cube name at provision time — `createCube()` in `lib/ssh/firecracker.ts` slugifies the name to a valid RFC-1123 label (`slugifyHostname()` in `lib/utils.ts`, fallback `cube-<id prefix>`) and writes `/etc/hostname` + `/etc/hosts` into the loop-mounted rootfs, overriding the `krova` default baked into the image. This applies to every cube and has no cloud-init dependency. cloud-init itself is installed in every rootfs but kept DISABLED by default (`/etc/cloud/cloud-init.disabled`, touched by `build-all-images.sh`). It is enabled per-cube ONLY when the customer passes `user_data`: `createCube()` removes the disable flag, writes a NoCloud seed (`/var/lib/cloud/seed/nocloud/{meta-data,user-data}`), pins `datasource_list: [ NoCloud, None ]`, and rewrites the `- default` users entry in `/etc/cloud/cloud.cfg` to `- root` so no default distro user is created (cloud-init merges config lists by append, so a `users: []` override in cloud.cfg.d would NOT work — the base file must be edited). Krova's `99-krova-cloud-init.cfg` policy keeps `disable_root: false`, `ssh_deletekeys: false`, `preserve_hostname: false` — cloud-init never touches the customer's root SSH key or the guest host keys, and a customer `hostname:` directive overrides the slugified default. Cubes without `user_data` keep cloud-init fully inert (identical boot to before). **Phase 2 requires an image rebuild** (`pnpm build:images`) + Update Images per server before `user_data` works — older rootfs images have no cloud-init installed; the `createCube()` cloud.cfg edit is guarded on the file existing so a `user_data` cube provisioned on a pre-cloud-init image still boots cleanly (the script just does not run).
