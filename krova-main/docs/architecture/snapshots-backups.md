### Snapshots & Backups

**Per-plan snapshot configuration (2026-05-25 overhaul).** Auto cadence, retention buckets, and the manual-snapshot cap all live on the `plans` table now (the legacy `SNAPSHOT_AUTO_*` constants in `config/platform.ts` and the `getSnapshotConfig()` reader are gone). Each plan declares:

- `auto_snapshot_cadence_hours` ŌĆö NULL = auto disabled (Trial). Min 2h. Defaults: **Trial NULL, Starter 12, Pro 6, Business 4.**
- `auto_snapshot_keep_last / _daily / _weekly` ŌĆö retention buckets passed to `restic forget`. Defaults: **Starter 4/7/1, Pro 8/7/2, Business 12/14/4** (so daily auto-prune keeps roughly two weeks of recent + a few weekly anchors).
- `max_manual_snapshots_per_cube` ŌĆö hard cap on customer-created snapshots per cube. 0 = customer cannot create manual snapshots (Trial). Defaults: **Trial 0, Starter 1, Pro 2, Business 4.** Pinning an auto snapshot consumes a manual slot.

Two crons now drive auto snapshots, registered in [boss.ts](lib/worker/boss.ts):

1. **`snapshot.scheduler`** (hourly, `0 * * * *`, `policy: "exclusive"`). [snapshot-scheduler.ts](lib/worker/handlers/snapshot-scheduler.ts). For every running/sleeping cube whose plan declares a cadence, the pure decision `shouldScheduleAutoSnapshot(cube, plan, now)` returns true iff the cadence has elapsed (running) or the cube hasn't snapshotted since its last sleep transition (sleeping ŌĆö `cubes.snapshotted_since_sleep` is reset by `cube-sleep.ts` / `cube-power-off.ts` on transition and re-set by `snapshot-create.ts` on success). Inserts a `cube_snapshots` row with `kind='auto'` and enqueues `snapshot.create`.
2. **`snapshot.auto-prune`** (daily 03:30 UTC, `policy: "exclusive"`). [snapshot-auto-prune.ts](lib/worker/handlers/snapshot-auto-prune.ts). Per cube: `buildResticForgetArgs(buckets, autoTagIds)` ([lib/snapshots/forget-args.ts](lib/snapshots/forget-args.ts)) produces `--tag <autoId> ŌĆ” --keep-last N --keep-daily N --keep-weekly N` ŌĆö one `--tag` per `kind='auto'` snapshot's id (its restic tag), scoping the retention policy to EXACTLY the auto snapshots so pinned/manual snapshots are never forget candidates (restic has NO `--keep-id` flag; see Rule 47). After forget, the handler reconciles DB rows against `restic snapshots --json --no-lock`: any `kind='auto'` row whose `storagePath` is no longer in the repo is hard-deleted.

**Snapshot kind enum** (`snapshot_kind`: `auto | manual`). Migration 0055 added the column with default `'manual'` and backfilled rows that had `is_automatic=true` to `'auto'`. The legacy `is_automatic` boolean is still maintained in sync (snapshot-create writes both; pin/scheduler write both) ŌĆö kept for status-display filters that haven't migrated to `kind` yet. Customer-side `deleteSnapshot` action refuses `kind='auto'` snapshots with a message pointing at the Pin button.

**Snapshot status model ŌĆö a snapshot's status describes ONLY the snapshot (2026-05-31 decoupling).** `snapshot_status` is `pending | creating | complete | restoring | failed`, but the live model is narrow: a real snapshot follows `pending ŌåÆ creating ŌåÆ complete`, and once `complete` it STAYS `complete` until deleted ŌĆö no operation performed *with* a snapshot ever changes its status. **Restore is read-only on the snapshot** (it reads the per-cube restic repo and writes the cube's live rootfs), so it claims the **cube** (`status='stopping'`, set atomically by BOTH the dashboard `restoreSnapshot` action and the v1 API restore path) as the restore lock and NEVER marks the snapshot `restoring`/`failed`. The worker (`snapshot-restore.ts`) guards on the cube being `stopping` and normalizes the snapshot back to `complete` on every terminal outcome ŌĆö success OR failure ŌĆö because the restic data is always intact (restore never writes the repo); the guarded-connect failure path also returns the cube to its pre-restore state (preserving `lastBilledAt` for a was-running cube, since the VM was never stopped ŌĆö Rule 51), and the main catch RETURNS rather than throws after recovering (a pg-boss retry would skip on the now-cleared `stopping`, so returning closes a stale-restore-job race). A failed restore therefore leaves the snapshot fully re-restorable ŌĆö this fixed the bug where a transient host blip stamped the snapshot `failed` and the `status === 'complete'` restore gate then permanently blocked re-restore. **Create-failures** route through `snapshotCreateFailureAction(kind)` (`lib/snapshots/failure-policy.ts`, the single source of truth, unit-tested): `auto ŌåÆ delete the row` (the scheduler retries next cadence ŌĆö no list noise), `manual` ŌåÆ a non-blocking, dismissible `failed` note (storagePath null, excluded from the manual cap via `countManualSnapshotsForCube`, surfaced with **Retry**/**Delete** in the snapshots UI); `deleteSnapshot` dismisses a `failed`+null-storagePath note via a direct row delete (no `SNAPSHOT_DELETE` job ŌĆö nothing in restic). The create handler also never downgrades an already-`complete` row when a trivial post-success step (audit/email/usage adjust) throws. Net: `restoring` is no longer written by anything, and `failed` is written ONLY by a manual create-failure. **Rule: never write `restoring`/`failed` to a `cube_snapshots` row for a restore ŌĆö the cube's `stopping` status is the restore lock; a snapshot is a snapshot.** One-shot heal for rows wrongly left `failed`/`restoring` by the old behavior: `tsx scripts/heal-snapshot-status.ts` (dry-run by default; `--apply` commits) ŌĆö `failed`/`restoring` WITH restic data ŌåÆ `complete`, auto without data ŌåÆ delete, manual without data ŌåÆ left as the note (a `pnpm snapshots:heal-status` alias is pending the resolution of unrelated uncommitted `package.json` dep-bumps ŌĆö see the spec).

**`cube_snapshots` operations the customer can do** (`cube.manage` permission required):

- **Create a manual snapshot** ŌĆö gated by `assertCanCreateManualSnapshot(limits, manualCount)` against the plan's `maxManualSnapshotsPerCube`. Trial returns "does not include manual snapshots"; capped plans return "delete one first".
- **Restore** (overwrites the live rootfs from the snapshot ŌĆö existing flow, unchanged).
- **Delete** ŌĆö only `kind='manual'`; auto snapshots refuse with the "pin to convert" message.
- **Download as `.cube`** ŌĆö [snapshot-export.ts](lib/worker/handlers/snapshot-export.ts). `restic dump <snapshotId> rootfs.ext4` on the source cube's host ŌåÆ `buildCubeArchive` (zstd + manifest + tar) ŌåÆ `s3HostUpload` to `<env>/exports/<spaceId>/<exportId>.cube` ŌåÆ presigned 24h GET URL emailed via [snapshot-export-ready](lib/email/templates/snapshot-export-ready.ts) ŌåÆ row inserted into [snapshot_exports](db/schema/snapshot-exports.ts). The `snapshot.export-reap` hourly cron deletes the S3 object + flips the row to `expired` once `expiresAt` passes; failed/expired rows older than 7 days hard-delete. Rate limit: at most one in-flight export per snapshot (rejected at the action with "already in progress, check your email").
- **Clone to new cube** ŌĆö [cube-from-snapshot.ts](lib/worker/handlers/cube-from-snapshot.ts). Customer picks name/region/size/SSH key in [snapshot-clone-sheet.tsx](components/snapshot-clone-sheet.tsx). Server action allocates a destination server inside a per-space-locked tx (same plan-tier gate as `createCube`), then the handler `restic dump`s the source snapshot's rootfs straight onto the destination cube's `/var/lib/krova/cubes/<id>/rootfs.ext4` (no S3 round-trip through a `.cube` archive ŌĆö saves ~5 GB of redundant transfer), optionally grows the disk (`truncate -s` + `e2fsck` + `resize2fs`; shrink rejected ŌĆö ext4 can't shrink), loop-mounts to overwrite the customer's `authorized_keys` + rewrite the guest network config for the new IP, then `startCube` + iptables forward + tcp_port_mappings row exactly like `cube-import-rootfs.ts`.
- **Pin auto ŌåÆ manual** ŌĆö `pinAutoSnapshot` action flips `kind` from `auto` to `manual` in the DB. The restic snapshot's tag is its own `cube_snapshots.id` (unchanged); auto-prune scopes its policy to the `kind='auto'` ids via `--tag`, so a pinned snapshot (now `kind='manual'`) is simply absent from the candidate set and never forgotten. Cap-checked: pinning consumes a manual slot.
- **Promote to backup** ŌĆö [snapshot-promote-to-backup.ts](lib/worker/handlers/snapshot-promote-to-backup.ts). Server action inserts a `cube_backups` row via `createPreDeletionBackup({ skipEnqueue: true })` (reusing the cubeConfig capture logic) then enqueues the promote handler, which `restic dump`s the source snapshot to `<env>/backups/<spaceId>/<backupId>.cube` and flips the backup row to `complete`. Source cube + source snapshot are untouched. Cap-checked against `maxBackups`.

**Pre-deletion backup default.** The "Preserve backup" checkbox in the delete-cube dialog is now **hidden on Trial** (`planAllowsBackups = effective.maxBackups === null || effective.maxBackups > 0`) and **defaults CHECKED on paid plans** so accidental deletes preserve data ŌĆö customers must opt OUT, not opt IN. Threaded as `planAllowsBackups` prop through [cube-detail-shell.tsx](components/cube-detail-shell.tsx) ŌåÆ [cube-detail-header.tsx](components/cube-detail-header.tsx).

**Backup storage billing.** Per-GB-month rate from `platform_settings.backup_storage_rate_per_gb_per_month` (default $0.01, operator-tunable in Orbit ŌåÆ Platform settings). [billing-hourly.ts](lib/worker/handlers/billing-hourly.ts) charges hourly: `(sizeBytes / 1024^3 || diskSizeGb) ├Ś rate / 730` per `cube_backups` row with `status='complete'`. Prefers the compressed `sizeBytes` (what the `.cube` actually consumes on S3) over the original `diskSizeGb`. Zero balance after backup billing triggers the same auto-sleep + zero-balance email path as the cube-billing pass.

**Sleep storage billing ŌĆö ALWAYS ON, full disk size, single source of truth.** A sleeping cube releases host CPU + RAM back to the allocator pool (Firecracker is paused or killed) but its rootfs file remains on the bare-metal host's disk. Sleep storage is billed at **`DISK_RATE ├Ś diskLimitGb ├Ś tier multiplier` per hour** (from [config/platform.ts](config/platform.ts), via the shared `calculateSleepHourlyCost()` helper in [lib/cost-shared.ts](lib/cost-shared.ts)) ŌĆö same per-GB rate AND full-disk basis as the running-disk component (running and sleeping cubes both occupy every allocated GB on the host; there is no free-disk allowance ŌĆö Krova sells RAM and disk 1:1 with the host, no overselling). **There is no operator toggle and no per-installation override** ŌĆö setting `DISK_RATE = 0` in `config/platform.ts` is the only way to disable sleep-storage billing, and it disables running-disk billing in lockstep (one knob, one truth). The legacy `platform_settings.sleep_storage_rate_per_gb_per_month` column is deprecated ŌĆö no code reads it as of this deploy, kept only per Rule 40 pending a follow-up `DROP COLUMN` migration. Both [billing-hourly.ts](lib/worker/handlers/billing-hourly.ts) and [getSpaceBurnRate](lib/billing.ts) read DISK_RATE directly from `config/platform.ts`, so the customer-facing burn rate matches the worker's billed amount exactly. [billing-hourly.ts](lib/worker/handlers/billing-hourly.ts)'s sleep-storage pass is independent of `lastBilledAt` (that flag is the running-compute clock and stays `null` while sleeping) ŌĆö every sleeping cube joins the rotation each tick. **Routed through `applyOverageCascadeTx`** so customers with overage enabled have their sleep-storage cost covered by overage when prepaid is empty; refused funding triggers the same auto-sleep + zero-balance email path as the compute pass. Customer-visible: lifecycle log on sleep/power-off says "compute charges stopped; sleep-storage billing continues for disk"; the customer billing page surfaces a fourth pillar (Hourly / Prorated / Sleep storage / Backup storage); `getSpaceBurnRate()` uses the same `calculateSleepHourlyCost()` helper so the runway projection matches the worker's billed amount exactly.

**Backup storage billing also routes through the overage cascade.** Same change as sleep storage (Fix #1 from the 2026-05-27 billing audit). Previously backup storage debited balance with a `Math.max(0, balance - cost)` clamp that silently lost the refused delta to a ledger gap, and could never use the customer's overage budget. Now: prepaid first, overage second, refused triggers auto-sleep of running cubes in the space. The `platform_settings.backup_storage_rate_per_gb_per_month` knob still controls the rate.

**Removed: "Save as Backup from running cube".** The customer-facing sheet and the `createBackupFromCube` server action are gone. Backups now come from exactly two places: the pre-deletion checkbox AND the new "Save as Backup" button on snapshot rows (which goes through promote-to-backup).

Snapshots and backups share one S3 bucket but use **different on-S3 layouts and different transfer pipelines**, because they have different workload shapes:

|                                  | Snapshots                                                       | Backups                                                          |
| -------------------------------- | --------------------------------------------------------------- | ---------------------------------------------------------------- |
| **Captures**                     | Repeated point-in-time of the SAME cube (mostly unchanged data) | One-off captures of distinct cubes (different content each time) |
| **Storage layer**                | **Restic** content-addressed chunked dedup, per-cube repo       | Full `.ext4.zst` compressed blob                                 |
| **On-S3 path**                   | `<env>/snapshot-repos/<cubeId>/` (config + chunks + indexes)    | `<env>/backups/<spaceId>/<backupId>.ext4.zst`                    |
| **First capture cost**           | ~5 GB compressed (full chunked upload)                          | ~5 GB compressed (full blob upload)                              |
| **Repeated capture cost**        | ~50ŌĆō200 MB (only changed chunks)                                | ~5 GB every time                                                 |
| **Tool on bare-metal host**      | `restic backup` / `restic restore` / `restic forget --prune`    | `rclone copy` (multipart)                                        |
| **`cube_snapshots.storagePath`** | The restic snapshot id (64-hex)                                 | n/a                                                              |
| **`cube_backups.storagePath`**   | n/a                                                             | The S3 object key                                                |

**Storage backends** themselves are stored in the `storage_backends` table ŌĆö `endpoint` / `region` / `bucket` + AES-256-GCM-encrypted `access_key_id` / `secret_access_key`. `selectBackend()` in [lib/storage/backends.ts](lib/storage/backends.ts) picks the active backend with the most free space. Both pipelines (restic + rclone) talk to that same bucket. **`storage_backends.usedBytes` is reconciled against ground truth every 30 min by the `storage.health-check` cron** ([storage-health-check.ts](lib/worker/handlers/storage-health-check.ts) ŌåÆ `s3SumObjectSizes` streams + sums the real S3 object sizes under the backend's `<env>/` prefix and writes `usedBytes`). The per-op `adjustBackendUsage` deltas (`+data_added_packed` on snapshot create, `ŌłÆsizeBytes` on delete, `┬▒` on backup create/delete) are only a between-tick APPROXIMATION ŌĆö restic dedup means a delete frees only the snapshot's unique chunks (not its full `data_added_packed`), and the dominant removal paths (`snapshot.auto-prune`, `restic.prune`'s chunk reclaim, `cube.delete`'s repo-prefix wipe) never decrement at all, so the ledger alone drifts UPWARD unbounded. The periodic reconcile is the source of truth; never treat `usedBytes` as exact between ticks, and never remove the reconcile thinking the deltas suffice (2026-05-30 audit).

**Backup transfers** are host-side via `rclone copy` ([lib/storage/s3-transfer.ts](lib/storage/s3-transfer.ts)) ŌĆö credentials enter as inline `RCLONE_CONFIG_BOX_*` env vars on the bash command line, so no on-disk config and no leak into `ps` output. Tunings (`--multi-thread-streams 4 --s3-upload-concurrency 4 --s3-chunk-size 64M`) come from the iDrive E2 EU benchmark ŌĆö ~215 MB/s sustained.

**Snapshot transfers** are host-side via `restic` ([lib/storage/restic/](lib/storage/restic/)). Per-cube restic repository at `s3:https://<endpoint>/<bucket>/<env>/snapshot-repos/<cubeId>`. Each cube has its own AES-256-GCM-encrypted password stored in `cubes.snapshot_repo_password_enc` (column added in migration 0050; lazily generated on the first `restic init` via `getOrCreateRepoPasswordForCube`). Credentials are passed as inline `RESTIC_REPOSITORY` / `RESTIC_PASSWORD` / `AWS_*` env vars on the bash command. `RESTIC_CACHE_DIR=/var/lib/krova/restic-cache` keeps chunk-index metadata warm between operations. All commands pass `-o s3.bucket-lookup=path` because path-style addressing is required for non-AWS S3 endpoints (per restic docs). The pinned version is `RESTIC_VERSION` in `config/platform.ts` ŌĆö install via the bare-metal `restic` step in `server-install.ts` for new servers, or `pnpm install:restic` to retrofit existing servers.

**Storage-cost asymmetry.** Snapshots win storage cost by ~10ŌĆō20├Ś over the old full-blob model because restic dedupes ~95% of chunks between consecutive snapshots of the same cube. The `cube_snapshots.sizeBytes` column stores the **dedup'd `data_added_packed` value** from the restic `--json` summary message ŌĆö the bytes ACTUALLY written to the repo AFTER compression (restic 0.18.x repo format v2 compresses by default, so `data_added` would overstate the real S3 footprint; the parser in [commands.ts](lib/storage/restic/commands.ts) falls back to `data_added` only when `data_added_packed` is absent). `sizeBytes` is **display-only** (the customer-facing snapshots table renders it under the **"Added"** column ŌĆö an incremental dedup metric, NOT the snapshot's restore size ŌĆö with a header tooltip explaining that; never relabel it back to a flat "Size") plus `storage_backends` free-space accounting via `adjustBackendUsage`. **Snapshots are NOT billed by size** ŌĆö only `cube_backups` are billed per-GB-month. The values look large/erratic per snapshot because restic chunks the raw `rootfs.ext4` image: ext4 journal/metadata churn + block reallocation make each snapshot's `data_added_packed` swing with the workload; this is expected, not a bug. A steady-state (non-first, non-resize) snapshot whose size approaches the full used-disk would be the only red flag (dedup defeated ŌĆö e.g. in-guest disk encryption or a full-image rewrite).

**Lightweight operations** (object deletion, bucket listing, capacity probes) go through the AWS SDK directly from the worker process via [lib/storage/s3-direct.ts](lib/storage/s3-direct.ts). The worker never holds file bytes on either path.

**Restic operational crons** (registered in [lib/worker/boss.ts](lib/worker/boss.ts)):

- `restic.prune` ŌĆö Sundays 04:00 UTC. Iterates every non-deleted cube with a snapshot repo, runs `restic prune` on each. Reclaims chunks no longer referenced by any snapshot (e.g. after many `forget`s have left orphaned chunks). Acquires an exclusive repo lock per cube; concurrent backups wait ~30s for the lock and proceed.
- `restic.check` ŌĆö Sundays 06:00 UTC. Runs `restic check --read-data-subset=2%` per cube ŌĆö over 50 weeks the entire repo is verified. Any failure emails the platform-admin recipients (`getErrorNotifyEmails()`) with the cube id + stderr so an operator can investigate before the next snapshot/restore exposes the corruption.

**Stale-lock auto-recovery ŌĆö every lock-taking restic op routes through `runResticWithLockRecovery` ([commands.ts](lib/storage/restic/commands.ts)).** A restic process that dies mid-op (host OOM/SIGTERM, host down mid-snapshot, or a cube transferred off a host killed mid-op) leaves an EXCLUSIVE lock in the cube's per-cube S3 repo. Because the repo lives in S3, the lock outlives the dead host, so every later `forget`/`backup`/`prune`/`check`/`restore`/`dump` for that cube fails with **exit 11 "repository is already locked"** ŌĆö silently retried forever by pg-boss (pg-boss does NOT print handler rejections to stdout; the captured error lives in the `pgboss.job.output` column). This stranded a snapshot delete for 28h in the 2026-05-30 incident (lock from a dead PID on a host the cube no longer lived on). The shared wrapper prepends `--retry-lock 1m` (rides out brief genuine contention) and, on exit 11, **only when the held lock is provably stale by AGE** (`>= RESTIC_STALE_LOCK_MIN_AGE_MS`, 45 min ŌĆö deliberately ABOVE the longest 30-min op timeout so a still-running legitimate op would already have been killed), runs a plain `restic unlock` (which removes ONLY locks restic itself judges stale ŌĆö NEVER `--remove-all`, which would clobber a live op and risk repo corruption) and retries the command exactly once. A younger/unparseable lock is NEVER force-removed ŌĆö the op fails and pg-boss retries later, so a concurrent live backup is never disturbed. restic checks lock AGE before hostname (verified against restic 0.18.1 `Lock.Stale()`), so a >30-min lock clears even cross-host (the incident's case: unlock ran from the cube's new host, the lock was created on the old one). Forced unlocks are audit-logged (`restic.stale_lock_recovered`). The operator escape hatch for a lock automation refuses to touch is `pnpm restic:unlock <cubeId>` (see commands table). **Rule: never build a `restic <verb>` shell string inline ŌĆö route every lock-taking op through a `commands.ts` wrapper so it inherits this recovery (the auto-prune cron's inline `restic forget` was the last bypass, now folded into `resticForgetWithRetention`).**

Pre-deletion backups also persist the Cube's full configuration in `cube_backups.cubeConfig` (JSONB) for identical redeployment.

**Two ways a `cube_backups` row gets created**, both run through the same `backup.create` worker handler:

1. **Pre-deletion backup** ŌĆö `deleteCube({ preserveBackup: true })` from the cube detail delete dialog, or `cube-stale-check` for stuck-cube salvage. The handler's `finally` block enqueues `cube.delete` after the backup completes (even if the backup fails ŌĆö the salvage attempt never blocks the delete).
2. **Save-as-backup** ŌĆö `createBackupFromCube()` server action from the Snapshots tab's "Save as Backup" button. The source cube must be `running` or `sleeping` and `transferState === "idle"`. The handler runs the same compress + upload, but the `finally` block sees `deleteCubeAfter !== true` and does NOT enqueue `cube.delete`. The cube keeps running.

Both flows reuse [lib/cubes/create-pre-deletion-backup.ts](lib/cubes/create-pre-deletion-backup.ts) for the config snapshot (CPU/RAM/disk/image/region + active domain mappings + non-SSH TCP mappings with their whitelisted CIDRs). The `BackupCreatePayload.deleteCubeAfter` field distinguishes the two flows. **Deletion is opt-in**: the field defaults to `false` (both at the helper layer and at the handler's gate, which uses `=== true`). Pre-deletion callers MUST pass `deleteCubeAfter: true` explicitly ŌĆö that way a forgotten flag at any future call site cannot accidentally destroy a customer's running cube.

On **redeploy** ([backup-redeploy.ts](lib/worker/handlers/backup-redeploy.ts)), if a domain or TCP mapping from the captured config is still in use by the source cube, the redeploy handler **skips it and logs the skip on the new cube's lifecycle log** (`(partial: N mapping(s) failed to re-create)`). The new cube boots without those mappings ŌĆö the customer re-adds them manually if they want to migrate routing.

**Capability gating.** When no `storage_backends` row is active, the customer dashboard hides the Snapshots tab, the Backups nav, and the "Preserve backup before deleting" checkbox in the delete-cube dialog ([lib/storage/capabilities.ts](lib/storage/capabilities.ts)). The Orbit admin panel always shows storage management so operators can provision the first backend.

**Snapshot/backup data-safety invariants (2026-05-30 bulletproof audit ŌĆö do NOT regress).**

- **Restore never corrupts the live rootfs.** [snapshot-restore.ts](lib/worker/handlers/snapshot-restore.ts) restores restic into a TEMP dir (`<cubeDir>/.restore-tmp`), `e2fsck`s it, then ATOMICALLY renames the current rootfs to `.bak` and the validated restore into place. The live `rootfs.ext4` is therefore always a COMPLETE ext4 (the intact original until the instant of the atomic rename, then the validated restore) ŌĆö never a half-written hybrid, even if the worker is SIGKILL/OOM'd mid-restore. On re-entry (row stays `restoring`) the handler recovers: puts `.bak` back if it died mid-cutover, clears stale temp, and re-runs. The failure `catch` kills any lingering Firecracker BEFORE moving `.bak` back. Never reintroduce an in-place `restic restore` over the live rootfs.
- **A pre-deletion backup that produced NO usable backup must NEVER delete the cube.** [backup-create.ts](lib/worker/handlers/backup-create.ts) enqueues `cube.delete` ONLY on a `success` outcome. `interrupted-no-upload` AND `failed-during-upload` (e.g. a transient S3 blip) both leave the rootfs intact, flip the cube to `error` (so `cube.stale-check` can't re-loop the salvage), and notify admins. The customer chose to preserve their data ŌĆö a failed backup must not destroy it.
- **`transferState='idle'` gate on every op that touches the live rootfs.** Snapshot create (scheduler WHERE + `snapshot-create` step 2b + the `createSnapshot`/`restoreSnapshot` actions, the latter also in the atomic claim), and `deleteCube` (atomic claim + a clear error) all refuse when the cube is mid cross-server transfer ŌĆö `cube.transfer` is copying `rootfs.ext4` and a concurrent restic-backup / overwrite / `rm -rf` would capture or corrupt a torn ext4. Export/clone/promote are SAFE mid-transfer (they read the immutable S3 repo, not the live rootfs) and are intentionally NOT gated. `deleteCube` also refuses while a snapshot is `creating` (a `restic backup` is reading the live rootfs).
- **Restic-dump output is ext4-validated before it's packaged.** [host-build.ts](lib/storage/cube-archive/host-build.ts) runs `file -b` (ext4 magic) before compressing, so a truncated/garbage `restic dump` (e.g. host disk full mid-write) fails the build immediately instead of producing a `complete` backup that's unrestorable.
- **The `usedBytes` reconciler refuses a partial S3 listing.** `s3SumObjectSizes` / `s3ListObjects` throw on `IsTruncated && !NextContinuationToken` (an iDrive E2 / S3-compatible pagination quirk) so a short page can't silently shrink `usedBytes` and defeat the capacity safety net. See `nextContinuationToken` in [s3-direct.ts](lib/storage/s3-direct.ts).
- **Stale-lock auto-recovery never removes a LIVE lock.** Beyond the Ōēź45-min age gate, `runResticWithLockRecovery` parses the lock holder from restic's exit-11 stderr and, if it's the SAME host with a still-alive PID (`kill -0`), refuses to unlock (a refresh-stalled-but-alive op). Cross-host locks rely on the age gate; never `--remove-all` in automation.

**"Transferring" cube badge.** A cube mid cross-server transfer keeps `status='running'/'sleeping'`, so the status badge would misleadingly show "Running". [cube-status-badge.tsx](components/cube-status-badge.tsx) takes an optional `transferState` and renders a derived **"Transferring"** badge when `isActiveTransferState(transferState)` ([status-display.ts](lib/status-display.ts) ŌĆö active = `snapshotting|restoring|finalizing|cancelling`; `idle|completed|failed` show the real status). Every cube-list/detail surface (Orbit cubes/space/server tables + cube detail, customer cube list + detail header) passes `transferState`; the data queries select `cubes.transferState`. "Transferring" is a DERIVED display state, not a `cubes.status` enum value.

**`.cube` archive format (export + import).** A backup or a customer upload can travel as a single portable file with extension `.cube`. The format is a plain tar (no outer compression ŌĆö the rootfs blob inside is already zstd) bundling exactly three bare-name entries:

```text
<backupId>.cube
Ōö£ŌöĆŌöĆ manifest.json        ŌåÉ UTF-8 JSON (Ōēż64 KB)
Ōö£ŌöĆŌöĆ rootfs.ext4.zst      ŌåÉ compressed rootfs (bulk)
ŌööŌöĆŌöĆ checksums.txt        ŌåÉ sha256sum format
```

The wire-format tag is `format: "krova-cube-v1"` and the schema is the single source of truth in [lib/storage/cube-archive/manifest.ts](lib/storage/cube-archive/manifest.ts) (Zod). The tar is built host-side by [host-build.ts](lib/storage/cube-archive/host-build.ts) (the worker writes the manifest via base64 to dodge heredoc quoting traps per Rule 39) and consumed by [host-extract.ts](lib/storage/cube-archive/host-extract.ts) which verifies the archive has exactly the three expected entries (no path traversal, no nested dirs), checks `sha256sum -c`, parses the manifest, decompresses the rootfs, and runs `file rootfs.ext4` as the final ext4 sanity gate.

**Export.** `backup.create` produces a `.cube` archive on the host and rclone-uploads it to `<env>/backups/<spaceId>/<backupId>.cube` in one shot ŌĆö no separate `.ext4.zst` step. The customer downloads it from the Backups list via a 15-minute presigned S3 URL ([app/actions/cube-import.ts](app/actions/cube-import.ts) `getBackupDownloadUrl` for the dashboard, [app/api/v1/.../backups/[backupId]/download/route.ts](app/api/v1/spaces/[spaceId]/backups/[backupId]/download/route.ts) for the v1 API). The URL is bound to whichever storage backend holds the archive; the browser fetches the object directly from S3.

**Import.** The customer picks a `.cube` file in the Import Cube sheet ([components/cube-import-sheet.tsx](components/cube-import-sheet.tsx)). The browser reads the first ~64 KB of the file locally, parses the tar header, and extracts `manifest.json` ŌĆö no upload yet. The form pre-populates from the manifest's vcpus/ramMb/diskLimitGb/imageId, clamped to the space's plan limits. Customer chooses **SSH key handling**: `replace` (default ŌĆö worker mounts the rootfs and overwrites `/root/.ssh/authorized_keys`) or `keep` (worker leaves authorized_keys alone ŌĆö customer must have the matching private key). Disk override can only grow (resize2fs upward), never shrink (would corrupt ext4).

The submit flow:

1. `initiateCubeImport()` ŌåÆ server picks a backend, runs plan-tier pre-flight inside a per-space-locked transaction, calls `CreateMultipartUpload` on S3, signs N per-part PUT URLs (8 MB chunks), inserts `cube_imports` row in `uploading`, returns part URLs.
2. Browser uploads parts in parallel (4-way concurrency) directly to S3, collects ETags.
3. `completeCubeImport()` ŌåÆ atomically flips `uploading ŌåÆ finalizing`, calls `CompleteMultipartUpload`, verifies the on-S3 ContentLength matches the customer's declared size (┬▒5%), runs plan-tier check again, calls `allocateServerAndCreateCube` in the SAME per-space-locked transaction, marks row `provisioning` with the new cube id, enqueues `cube.import-rootfs`.
4. [cube-import-rootfs.ts](lib/worker/handlers/cube-import-rootfs.ts) SSHes to the allocated server, rclones the `.cube` from S3, runs `extractCubeArchive` (sha256 verify + decompress + ext4 sanity), optionally grows the rootfs (`truncate -s` + `e2fsck` + `resize2fs`) if the customer chose a larger disk, runs `e2fsck` to replay any dirty journal, and ŌĆö if `ssh_key_mode='replace'` ŌĆö loop-mounts the rootfs to overwrite `/root/.ssh/authorized_keys`. Boots via `startCube`, sets up the SSH port forward + `tcp_port_mappings` row, waits for the guest agent, marks the cube `running` + the import `complete`, deletes the `.cube` from S3 (the new rootfs supersedes it).

**`cube_imports` table** (migration 0051) tracks every active upload: `s3_key`, `s3_upload_id`, `expected_size_bytes`, `chunk_size_bytes`, `ssh_key_mode` (`replace` | `keep`, enforced via CHECK constraint), `ssh_public_key` (nullable when mode=keep), `region_id`, optional size overrides, `user_data`. Status enum: `uploading | finalizing | provisioning | complete | failed | expired`.

**`cube-imports.reaper` cron** (every 6 hours at :10) aborts abandoned multipart uploads:

- `uploading` rows older than 24h ŌåÆ `AbortMultipartUpload` + status `expired`
- `finalizing` rows older than 1h ŌåÆ `AbortMultipartUpload` + status `failed`
- `provisioning` rows older than 4h ŌåÆ status `failed` (the cube row's `error` state is set by the handler's own cleanup)
- `failed`/`expired` rows older than 7 days ŌåÆ hard-delete the row (audit trail preserves history)

**S3 bucket CORS requirement (operator setup).** The import flow is browser-direct multipart upload to S3 via presigned PUT URLs. For this to work, every active `storage_backends` bucket MUST have CORS configured to allow PUT from the platform's origin AND expose the `ETag` response header (without `ETag` in `ExposeHeaders`, the browser can't read the per-part ETag, breaking multipart completion). Minimal CORS rule:

```json
[
  {
    "AllowedOrigins": ["https://krova.cloud", "https://*.krova.cloud"],
    "AllowedMethods": ["PUT"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3600
  }
]
```

For iDrive E2 + Backblaze B2 + MinIO, set this via the provider's bucket CORS configuration UI / CLI. AWS S3 uses the same JSON shape under `PutBucketCors`. The download flow doesn't need CORS (presigned GET URLs are fetched as navigations).

**Storage layout** under each S3 backend's bucket:

```text
<bucket>/<env>/backups/<spaceId>/<backupId>.cube              ŌåÉ finished backups
<bucket>/<env>/imports/<spaceId>/<importId>.cube              ŌåÉ uploads in flight
<bucket>/<env>/snapshot-repos/<cubeId>/                       ŌåÉ restic repos (unchanged)
```

`pnpm storage:audit` scans all three prefixes and reports orphans per category; `--delete` removes orphans older than `--min-age-hours`.

**On `backup.redeploy`** ([backup-redeploy.ts](lib/worker/handlers/backup-redeploy.ts)), the worker downloads the `.cube`, calls `extractCubeArchive` (same code path as `cube.import-rootfs`), then injects the customer's new SSH key ŌĆö so the redeploy + import pipelines share the host-extract phase.
