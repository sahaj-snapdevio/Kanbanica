# Plans, Tiers, Subscriptions & Credit Top-Up

> On-demand detail extracted from CLAUDE.md. CLAUDE.md keeps only a short summary + a pointer to this file; the full reference lives here.

### Plans & Tiers / Subscriptions / Credit Top-Up

**Plans & Tiers.** Every space's plan is a row in the `plans` table, joined via `spaces.plan_id`. Migration 0037 seeds four public plans (`plan_trial` / `plan_starter` / `plan_pro` / `plan_business`); operators add, duplicate, rename, archive, and tweak limits from Orbit → Plans (`app/actions/orbit-plans.ts`). A plan row sets the per-Cube ceilings (`maxVcpus` / `maxRamMb` / `maxDiskGb`), `maxConcurrentCubes`, `maxSeats`, `maxBackups`, `maxDomains` (`null` = unlimited), `includedCreditUsd`, `priceUsd`, `allowTopup`, `allowOverage`, plus `visibility` (`public` = every space sees it; `custom` = only spaces in `plan_space_visibility`) and `isDefaultForNewSpaces` (exactly one row, enforced by partial unique index). **Per-space overrides** on the `spaces` row (`override_max_*`, `override_allow_*`, `override_included_credit_usd`, `override_overage_cap_max_usd` — all nullable) let an operator give one space more (or less) than the plan's default without duplicating the plan; `effectiveLimits(plan, overrides)` in `lib/plan/limits.ts` is the merge — for each field, `override ?? plan` wins. Limits are enforced by `lib/plan/`: pure allow/deny guards in `limits.ts` (the `*V2` variants take `EffectiveLimits`), DB usage counts in `usage.ts`. Callers acquire a per-space advisory lock (`acquireSpaceLock`), run the usage count, then run the guard inside that transaction — so count + create/wake/invite/etc. is atomic per space. Plan rows are cached in process for 60s (`planCache` in `usage.ts`); Orbit writes invalidate via `invalidatePlanCache(planId)`.

**Subscriptions (Polar).** Spaces subscribe to a paid plan via **Polar recurring subscriptions**, behind the same provider-agnostic `PaymentProvider` interface in `lib/payments/`. The server actions in `app/actions/subscriptions.ts` (owner / `billing.manage` only) drive it: `createSubscriptionCheckout()` starts a hosted checkout (a `subscription_intents` row is inserted `pending` BEFORE the checkout is created); `changePlan()` upgrades immediately while a **downgrade is BLOCKED until the space already fits the lower tier's limits** (`checkSpaceFitsPlanV2()` in `lib/plan/limits.ts`); `cancelPlan()` cancels at the provider (sets Polar `cancelAtPeriodEnd=true` — the subscription stays `active` until `currentPeriodEnd`); `resumePlan()` undoes a pending cancel (sets `cancelAtPeriodEnd=false`) provided the period has not yet expired. Both cancel + resume buttons live directly on the main billing page (`components/space-billing.tsx`) — the customer never has to log into Polar's dashboard to manage their subscription. The pending-cancel state ("Ending on X — Resume subscription") is loud + always visible. The `<PlanSelectionSheet />` (`components/billing/plan-selection-sheet.tsx`) is the customer UI; it lists only the plans returned by `visiblePlansForSpace()` (`lib/plan/visibility.ts`) — public plans + any custom plan assigned to the space via `plan_space_visibility` + the space's own current plan (so a customer on an archived plan can still cancel). **The Polar webhook is the source of truth** — `POST /api/webhooks/polar` normalizes each event and `lib/billing/subscription-handler.ts` applies it under a per-space advisory lock with an `occurredAt` staleness guard (an out-of-order redelivery cannot regress plan/period state). It writes the four `spaces` subscription columns + first-touch captures `spaces.polar_customer_id` from `sub.customer.id` (or `order.customer.id` on a renewal); activation/renewal calls `applyPlanCredit()` (`lib/billing/apply-plan-credit.ts`) which grants the plan's `includedCredit` at most once per billing period (`subscription_credit_grants` UNIQUE) and, for activation only, at most once per `platform_settings.planCreditGrantCooldownDays` (anti-abuse). A `past_due`/`unpaid` status is **non-terminal** (the customer stays on the plan); a `canceled` subscription drops the space back to the default plan (`is_default_for_new_spaces = true`) and runs `reconcileSpaceCubeCount()` (`lib/plan/reconcile.ts`) — Cubes over the new plan's concurrent-Cube cap are auto-slept (most-recently-started first, by `cubes.last_started_at`; data preserved). The `subscription.reconcile` cron (`lib/worker/handlers/subscription-reconcile.ts`, hourly at :15) is the backstop for dropped webhooks — it polls Polar for every space with a subscription or a pending intent (via `polar.subscriptions.list({metadata: {spaceId}, active: true})` — sibling-safe vs. the legacy `customers.getStateExternal({externalId: spaceId})` path, which 404s for any non-first space and silently regressed siblings to Trial every hour) and feeds any divergence through the same handler via a synthetic `subscription.synced` event (`lib/billing/reconcile-subscription.ts`). When the metadata-keyed lookup returns empty for a space that still has a recorded `providerSubscriptionId`, the cron performs a **defense-in-depth** `getSubscription(id)` call before synthesizing a terminal event — if Polar reports the sub is still active/past_due/etc., the synthetic-cancel is refused and a warning is logged (the operator must add `metadata.spaceId` to the Polar subscription manually). When `POLAR_ACCESS_TOKEN` / `POLAR_WEBHOOK_SECRET` are unset, subscription checkout is inert (the webhook route returns 503, the reconcile cron no-ops).

**Postpaid overage (optional).** Each paid-plan space can opt into postpaid overage so Cubes keep running after the prepaid balance is exhausted, billed up to a customer-set cap on the next subscription invoice. The hourly worker (`lib/worker/handlers/billing-hourly.ts`) runs a **three-bucket cascade** per space via `lib/billing/overage.ts` `applyOverageCascadeTx()`: bucket 1 debits prepaid `creditBalance`; bucket 2 debits the remaining overage budget (`overageCapUsd − thisPeriodOverageUsd`) and inserts an `overage_charge` `billing_events` row; bucket 3 is the `refused` amount that auto-sleeps the space's Cubes (same path as zero-balance). Overage is **opt-in per space** — `spaces.overage_enabled` (false by default), `spaces.overage_cap_usd` (customer-set per billing period, bounded by `platform_settings.overageCapMinUsd` / `overageCapMaxUsd` with the per-space `override_overage_cap_max_usd` raising the ceiling for VIPs), `spaces.this_period_overage_usd` (running counter for the period). **A plan with `allowOverage=false` (e.g. Trial) cannot enable overage** — blocked by `updateOverageSettings()` in `app/actions/billing.ts` AND defended-in-depth in the cascade itself (effective `allowOverage` check), so any code path that ever flips such a space to `overage_enabled` still cannot accrue overage. Bucket 2 also requires `subscription_status === "active"` — a `past_due` subscription automatically stops accruing further overage (and a one-shot past-due email fires from `subscription-handler.ts`). The per-period counter is reset to `0` by `subscription-handler.ts` only when `currentPeriodEnd` strictly advances; a same-period `active ↔ past_due` flip does NOT reset. After the worker tx commits, `reportOverageEventNow()` ingests the event into Polar's metered-billing meter; failures are caught and left for the `polar.meter-reconcile` cron (`lib/worker/handlers/polar-meter-reconcile.ts`, `*/10 * * * *`) which re-reports any `overage_charge` row older than 5 minutes with `polar_meter_reported_at IS NULL`. The customer also receives up to five threshold emails per billing period (deduped via `lifecycleLogs` markers): `overage_started` (first overage debit), `overage_50` (50% of cap), `overage_80` (80% of cap), `overage_cap_hit` (cap reached → Cubes auto-sleeping), and `overage_past_due` (subscription went `past_due` while overage was on). **Operator setup (one-time, in Orbit → Platform Settings + Polar dashboard)** — (1) in the Polar dashboard, create a meter named `krova_overage_usd` (the value in `POLAR_OVERAGE_EVENT_NAME` in `config/platform.ts`); (2) paste the meter id into Orbit → Platform Settings → "Polar overage meter id" (saved to `platform_settings.polarOverageMeterId`); (3) provision each paid plan via Orbit → Plans → "Provision in Polar" (this creates the Polar product with a metered price referencing the meter). Overage reporting refuses to silently no-op when the meter id is unset — a no-op would lose meter events permanently.

**Processing-fee gross-up (Polar).** Both manual credit top-ups AND each paid subscription's monthly charge apply the same **processing-fee gross-up** (`platform_settings.paymentFeePercent` + `paymentFeeFlatUsd`): the customer pays `base + fee` to Polar; the space receives the full `base` as credit; the fee nets to Polar's processor cost so the platform breaks even on payment processing and earns its margin from compute usage. Each plan's `plans.priceUsd` is its face price AND the credit amount granted that period. The Polar subscription PRODUCT must be configured at the grossed-up amount so Polar bills the customer the right total — the gross-up math is `ceil((base + flat) / (1 − percent))` per `computeTopupCents` in `lib/billing/topup-checkout.ts` (and its client-safe mirror `paymentBreakdown` in `components/billing/topup-math.ts`, used by both the Add-Credits sheet and the plan-selection sheet). Provisioning a plan via Orbit → Plans → "Provision in Polar" computes the gross-up and creates the Polar product automatically; the resulting product id is written to `plans.polarProductId`. **Credit top-up flow.** The customer picks a base USD amount (`platform_settings.creditTopupMinUsd` / `Max` / `Default`). A `credit_purchases` row is inserted **before** the provider checkout is created (so the webhook always finds it). Polar signs every webhook (Standard Webhooks); `POST /api/webhooks/polar` verifies the signature via the provider abstraction (throw → HTTP 403), normalizes the event, and dispatches: `topup.paid` → `applyPaidTopup()` (idempotent `pending→paid` flip + `applyCreditTopup()` credit + `billing_events` ledger row); `topup.refunded` → base-fraction clawback writing a `credit_refund` event. The `billing.topup-reconcile` cron (hourly at :30, offset from `billing.hourly`) is a backstop for missed webhooks — it polls `pending` purchases from the last 48h against the Polar API and heals any Polar already marked paid. The top-up product id (`platform_settings.polarCreditProductId`) and the Polar secrets (`POLAR_ACCESS_TOKEN` / `POLAR_WEBHOOK_SECRET` in env) must all be set; if any are missing, top-up is inert (the webhook route returns 503, the reconcile cron no-ops). The Orbit admin grant path is unaffected. Each space has a `low_balance_threshold` (default + hard floor from `platform_settings.lowBalanceThresholdDefaultUsd` / `lowBalanceThresholdMinUsd`) controlling when the hourly worker sends the low-balance email.
